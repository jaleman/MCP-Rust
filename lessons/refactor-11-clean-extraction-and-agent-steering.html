Clean Extraction and Agent Steering

Refactor step 6 — driven by a real agent failure KUKA MCP Server in Rust

The incident

A Claude session was asked: "what is the minimum safe distance for a KMP 1500P?" The answer was in the bundle — the distance table extracted fine. But the agent's searches kept returning table-of-contents lines and intro text, so it gave up on search_docs, tried to read the source PDF from the filesystem, then browsed the raw resource list. Everything it did after the first search was wrong — and almost none of it was the agent's fault. This lesson is about the three server defects that invited the misbehavior, because they generalize to any tool you ever build for an AI agent.

Defect 1: the TOC always wins the excerpt contest

Excerpt anchors are ranked by co-occurrence: positions where many query terms appear close together. Look at a table-of-contents through that lens: 

3   Minimum Safe Distances ........................................ 5
3.1 KMP 600P / KMP 1500P .......................................... 7

Every term of the query, packed into two adjacent lines — a perfect co-occurrence score. And the tie-break preferred earlier positions, and the TOC is always at the front of the file. So for almost any query, all three excerpt windows were spent on navigation lines before reaching content. The repeated-lines filter couldn't help: TOC lines appear once.

The fix's real insight: pipeline position

We had already moved boilerplate filtering from query time to index time (Appendix 9). This step moves cleanup one stage further upstream — to extraction, where the bundle files are written:

query time    → filter on every search       (steps 1–4: wasteful, excerpts still dirty)
index time    → filter once per startup      (step 5b: anchors clean, excerpt TEXT still dirty)
extract time → filter once, ever          (step 6: files clean — everything downstream clean)

The Java analogy is ETL design: validate and cleanse at ingestion, not with WHERE clauses in every query. The further upstream a data-quality rule lives, the more consumers it protects — excerpts read raw bytes from the bundle files, so only extract-time cleaning could stop headers appearing inside excerpt text: 

/// chunk.rs — runs before chunking, so page boundaries survive
pub fn clean_extracted_text(text: &str) -> String {
    // count repetition across the whole document (page breaks count as line breaks)
    let boilerplate = repeated_lines(&text.replace('\x0c', "\n"));

    text.split('\x0c')
        .map(|page| page.lines()
            .filter(|line| {
                let norm = normalize_line(line);
                norm.is_empty()                              // keep paragraph structure
                    || (!boilerplate.contains(&norm)         // strip repeated headers/footers
                        && !line.contains("....."))          // strip TOC dot-leaders
            })
            .collect::<Vec<_>>().join("\n"))
        .collect::<Vec<_>>().join("\x0c")
}

Update (step 7): repetition alone turned out to be the wrong test — this version of the cleaner deleted a legitimate lookup table that was printed under three payload sections. Boilerplate detection is now position-aware (page-edge zones), and the index-time filter mentioned below was removed entirely. The full story: Refactor Appendix 12.

(The index-time filter from Appendix 9 stays as defense-in-depth for hand-written bundle files.) One bug the tests caught immediately: str::lines() splits only on \n, so a header directly after a page break merged into the previous line and escaped the repetition count — hence the replace('\x0c', "\n") before counting. We also switched to pdftotext -layout, which preserves columnar structure: the distance table now extracts as readable rows instead of a vertical smear of labels followed by numbers.

Defect 2: the tool output pointed at a file the agent couldn't use

Every search hit used to end with:

Source: kuka-docs/KUKA.AMR - Safe Minimum Distances v1.1.pdf

Put yourself in the agent's position: excerpts unhelpful, and the tool itself hands you… a PDF path. Reaching for the filesystem was practically suggested. Worse, the designed escalation — reading the document's kuka://docs/… resource — was not discoverable from the hit; the agent had to browse the whole resource list, which is precisely the misbehavior we saw. And in production the PDFs won't exist on the agent's machine at all. The fix is an API-design principle older than MCP: return actionable references, not internal paths (REST calls this HATEOAS — each response carries the links for the next step): 

// SearchHit: resource (PDF path) replaced by stem (bundle identifier)
pub struct SearchHit {
    pub title: String,
    pub stem: String,      // → presentation renders kuka://docs/{stem}
    pub score: usize,
    pub excerpts: Vec<String>,
}

// format_hit now emits the ACTIONABLE pointer:
"• {title}\n  Resource: kuka://docs/{stem}\n\n  ...{excerpts}..."

A bin test now pins the contract: assert!(!text.contains(".pdf")) — no source-file path ever appears in tool output again. The PDF name survives in the bundle file's frontmatter for human auditing; agents never see it.

Defect 3: nobody told the agent the intended workflow

Agents read two things when deciding how to use your server: the per-tool description and the server-level instructions from the handshake. Ours described what the tools did but not how to work with them — nothing said "retry with different terms before giving up" or "the resource URI in the hit is the follow-up". Both now encode the workflow: 

#[tool(description = "Search KUKA robot documentation. Always the FIRST tool for any \
    KUKA question. Returns ranked excerpts, each with a kuka://docs/ resource URI. \
    If the excerpts do not answer the question, retry with different search terms, \
    or read the hit's resource URI for the full section — do not look for source \
    files instead.")]

Think of descriptions and instructions as Javadoc that gets executed: an agent literally follows what they say, so vague docs produce vague behavior. A CLAUDE.md in the repo root now carries the same policy for coding sessions working inside this project.

The result

The exact query that derailed the other session, after the fix:

• KUKA.AMR - Safe Minimum Distances v1.1
  Resource: kuka://docs/kukaamr---safe-minimum-distances-v11

  ...This document describes the required clearances and distances...
  ...D1/DR1      Minimum safe distance between the AMR/Load and the obstacles
     D2/DR2      Minimum safe distance between two AMRs/Loads when passing...

No TOC noise, real content, a readable table row, and the URI for reading the complete 3 KB section. The agent's correct path is now the obvious path — which is the actual moral: when an agent misuses your tools, fix the tools before blaming the agent.
