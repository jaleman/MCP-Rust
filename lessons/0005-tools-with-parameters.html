Lesson 5

Tools with Parameters

~20 minutes Hands-on — run inside your dev container KUKA MCP Server in Rust

What you're building

Right now your server has one tool — ping — that takes no arguments. Every useful KUKA tool will need inputs: a search query, a document name, a robot model number. This lesson teaches you how to add those.

By the end you'll have a search_docs tool that accepts a query string and echoes it back as confirmation. It's a stub — the real PDF search comes in Lesson 6 — but the parameter wiring is exactly what the final production tool will use.

Before you start

This lesson builds on the KukaServer skeleton from Lesson 4. Your src/main.rs should contain the ping tool and cargo build should pass. If not, complete Lesson 4 first — the code to write is there.

How MCP carries tool arguments

When Claude calls a tool on your server, it sends a JSON message like this: 

{"name": "search_docs", "arguments": {"query": "AMR navigation"}}

Your server receives that JSON blob. rmcp's job is to:

  1. Look up "search_docs" in the tool router
  2. Deserialize {"query": "AMR navigation"} into a Rust struct
  3. Call your function with that struct
  4. Serialize the CallToolResult back to JSON and send it

Step 2 is handled by serde — Rust's standard serialization library. [serde.rs] Your struct needs to implement serde::Deserialize, which tells serde how to reconstruct the struct from incoming JSON.

Why rmcp also needs a JSON Schema

Before Claude can call search_docs, it needs to know what arguments are valid — their names, types, and what they mean. MCP exposes this as a JSON Schema in the tools/list response. rmcp generates it automatically using schemars. [schemars docs]

What the MCP client sees when it calls tools/list: 

{
  "name": "search_docs",
  "description": "Search KUKA robot documentation for a given query",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "Search term to look up in the KUKA documentation"
      }
    },
    "required": ["query"]
  }
}

Claude reads this schema and knows exactly what to send. Your job is just to annotate the struct — schemars generates the schema at compile time.

Your Rust struct
   → #[derive(schemars::JsonSchema)]
   → rmcp serializes the schema into tools/list
   → Claude reads the schema, fills in arguments
   → #[derive(serde::Deserialize)]
   → rmcp deserializes arguments into your struct
   → your function runs

If you know Java: serde = Jackson, schemars = bean-to-schema generation

serde::Deserialize — Rust's Jackson

In Java you use Jackson to deserialize JSON into a POJO: 

// Java — Jackson
public class SearchInput {
    @JsonProperty("query")
    public String query;
}
ObjectMapper mapper = new ObjectMapper();
SearchInput input = mapper.readValue(jsonString, SearchInput.class);

In Rust, #[derive(serde::Deserialize)] generates the equivalent of Jackson's reflective deserialization at compile time — no ObjectMapper instance needed: 

// Rust — serde
#[derive(serde::Deserialize)]
struct SearchInput {
    query: String,
}
// rmcp calls serde internally — you never write the deserialization call yourself

The derive feature flag you added in Lesson 3 enables this macro — it is opt-in, like adding the Jackson databind JAR to your Maven dependencies.

schemars::JsonSchema — generating JSON Schema from a struct

In Java you might use a library like jackson-module-jsonSchema or Swagger annotations to generate a JSON Schema from a class. schemars does the same thing in Rust: #[derive(schemars::JsonSchema)] generates the MCP inputSchema at compile time from the struct's field names and types.

#[derive(...)] — compile-time code generation, like Lombok

In Java, Lombok's @Data, @Builder, and similar annotations generate boilerplate code at compile time. #[derive(...)] is Rust's equivalent. The three derives on SearchInput each generate different code:

Rust deriveJava equivalent
DebugLombok @ToString — lets you print the struct with {:?}
serde::DeserializeJackson @JsonDeserialize / ObjectMapper.readValue()
schemars::JsonSchemaSwagger @Schema / jackson-module-jsonSchema

/// doc comments — Javadoc, but picked up by schemars

Javadoc comments (/** ... */) generate HTML documentation. Rust's /// doc comments do the same, but schemars also reads them and includes the text as the "description" field in the generated JSON Schema. That description is what Claude reads to understand how to fill in the argument — it is not just documentation, it is a runtime contract.

Parameters(input): Parameters<SearchInput> — destructuring, with no Java equivalent

This is the one pattern here that has no clean Java parallel. rmcp wraps the deserialized struct in a Parameters<T> container. Writing Parameters(input) in the function signature destructures that container: it unwraps the outer Parameters and binds the inner value directly to the name input.

The closest Java 16+ equivalent is pattern matching in an instanceof check — but in Rust, destructuring happens directly in function signatures and is used everywhere. Think of it as: instead of writing params.getInner() inside the function body, the unwrapping is declared in the parameter list.

format!() — String.format(), with {} instead of %s

JavaRust
String.format("Searching for '%s'", query) format!("Searching for '{}'", query)
"Hello " + name format!("Hello {}", name) — + exists but is rarely used

Step 1 — Add serde and schemars

Run these inside your dev container:

cargo add serde --features derive
cargo add schemars

The derive feature on serde enables the #[derive(Deserialize)] macro. Without it you'd have to implement the trait by hand — several hundred lines of boilerplate.

Step 2 — Define the input struct

Add this above the struct KukaServer definition in src/main.rs: 

#[derive(Debug, serde::Deserialize, schemars::JsonSchema)]
struct SearchInput {
    /// Search term to look up in the KUKA documentation
    query: String,
}

The three derives do separate jobs:

  • Debug — lets you print the struct with {:?} for quick debugging
  • serde::Deserialize — lets serde rebuild this struct from the JSON arguments
  • schemars::JsonSchema — lets schemars generate the MCP input schema at compile time

The /// doc comment on query is picked up by schemars and becomes the "description" field in the JSON Schema. That is the text Claude reads to understand how to fill in the argument — make it precise.

Step 3 — Add the search_docs tool

First, add Parameters to your use rmcp::{ ... } import block: 

handler::server::wrapper::Parameters,

Then add this method inside the #[tool_router] impl KukaServer block, after ping: 

#[tool(description = "Search KUKA robot documentation for a given query")]
fn search_docs(
    &self,
    Parameters(input): Parameters<SearchInput>,
) -> Result<CallToolResult, McpError> {
    Ok(CallToolResult::success(vec![Content::text(
        format!("Searching for '{}' — PDF ingestion not yet implemented.", input.query),
    )]))
}

Parameters<SearchInput> is rmcp's wrapper type that carries the deserialized tool arguments. The Parameters(input) in the function signature is Rust destructuring — it unwraps the Parameters wrapper and binds the inner SearchInput directly to input, so you can write input.query rather than input.0.query. This pattern is how rmcp 1.x passes all tool parameters.

format! is Rust's string interpolation macro. The {} placeholder calls the Display trait on input.query, which for a String just prints the string contents.

Step 4 — Build

cargo build

Expect a clean build. If you get an error about a missing import, check that your use rmcp::... block at the top of the file still includes tool, tool_router, and tool_handler.

If you get a compile error

Paste the full error output (everything starting from error[E...]) directly into chat with Claude. The most common issue at this stage is a mismatched import path after adding new dependencies.

Step 5 — Smoke-test the new tool

Run the server:

cargo run

Send the MCP initialize handshake:

{"jsonrpc":"2.0","method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"0.1"}},"id":1}

Confirm both tools appear with their schemas:

{"jsonrpc":"2.0","method":"tools/list","params":{},"id":2}

You should see search_docs listed alongside ping, with an inputSchema that marks query as a required string.

Now call the tool with a real query:

{"jsonrpc":"2.0","method":"tools/call","params":{"name":"search_docs","arguments":{"query":"AMR navigation"}},"id":3}

Expected response (trimmed):

{"result":{"content":[{"type":"text","text":"Searching for 'AMR navigation' — PDF ingestion not yet implemented."}],...}}

Press Ctrl+C to stop the server when done.

What you just proved

Your server correctly deserializes JSON arguments into a typed Rust struct and uses the value. The input.query field is exactly what you will pass to the PDF search index in Lesson 6 — the wiring is already in place.
