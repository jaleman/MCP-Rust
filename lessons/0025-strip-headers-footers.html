Lesson 25

Ignoring Headers & Footers at Search Time

~25 min Topic: HashMap → HashSet, tuple destructuring Prerequisite: Lesson 24

A PDF extracted to text carries its headers and footers on every page. "KUKA Robotics GmbH — Technical Reference" might appear 50 times in a single OKF file. Two things go wrong when search_docs sees that:

  1. Bad excerpt anchors — the excerpt builder collects every exact match position, so it can anchor a snippet on the 23rd occurrence of the page title instead of the paragraph that actually answers the query.
  2. Inflated ranking — the current score is the raw count of all term occurrences. A document where your query term appears in a 50-page running header outscores a short document that answers the question directly.
Why search-time, not extraction-time

The tempting fix is to strip repeated lines when writing the OKF file in extract.rs. Don't. Some repeated lines are real content: safety warnings, legal cautions, section labels like "DANGER: Deactivate before maintenance." Once stripped from the file, they're gone forever. Filtering at search time is non-lossy — the OKF files stay faithful to the source, and the filter only affects which positions the search engine uses.

The plan

Three small helper functions, two changes to search_docs. No new imports — HashMap and HashSet are already in scope.

  1. normalize_line — trim, collapse whitespace, lowercase
  2. repeated_lines — return a HashSet of every normalised line that appears 3 or more times in a text
  3. line_at_pos — given a byte position in a string, return the whole line it sits on
  4. In search_docs: filter positions through the boilerplate set before proximity scoring
  5. In search_docs: replace the raw occurrence count with the filtered position count

normalize_line

// Trims, collapses internal whitespace, and lowercases a line so that
// "  KUKA Robotics GmbH  " and "kuka robotics gmbh" count as the same line.
fn normalize_line(s: &str) -> String {
    s.trim()
        .split_whitespace()
        .collect::<Vec<_>>()
        .join(" ")
        .to_lowercase()
}

split_whitespace() splits on any run of whitespace and discards empty tokens, so multiple spaces, tabs, and non-breaking spaces all collapse to a single space. Without this, "KUKA Robotics" and "KUKA Robotics" would count as different lines and neither would reach the threshold.

repeated_lines

// Returns a HashSet of normalised lines that appear 3 or more times in text.
// These are the boilerplate candidates: running headers, footers, page titles.
fn repeated_lines(text: &str) -> HashSet<String> {
    let mut freq: HashMap<String, usize> = HashMap::new();
    for line in text.lines() {
        let norm = normalize_line(line);
        if !norm.is_empty() {
            *freq.entry(norm).or_insert(0) += 1;
        }
    }
    freq.into_iter()
        .filter(|(_, count)| *count >= 3)
        .map(|(line, _)| line)
        .collect()
}

The counting loop is the same entry / or_insert idiom you saw in lesson 25's original draft. The new part is what happens after: the HashMap is consumed and transformed into a HashSet in a single pipeline.

Breaking down the pipeline

freq.into_iter()              // consumes freq, yields (String, usize) tuples
    .filter(|(_, count)| *count >= 3)  // keep only frequent lines
    .map(|(line, _)| line)    // discard the count, keep the line
    .collect()                // Rust infers HashSet<String> from the return type

Two patterns here that are new:

  • Tuple destructuring in closure parameters — |(_, count)| names only the second field of the tuple and ignores the first with _. This is the Rust equivalent of ignoring a field in a Java record pattern.
  • *count — filter passes a reference to the tuple (&(String, usize)), so destructuring gives you &usize. The * dereferences it so you can compare it to the integer literal 3.

.collect() doesn't need a turbofish here because the return type HashSet<String> tells Rust exactly what to build.

line_at_pos

// Returns the trimmed line of text that contains the byte at position pos.
fn line_at_pos(text: &str, pos: usize) -> &str {
    let line_start = text[..pos].rfind('\n').map(|i| i + 1).unwrap_or(0);
    let line_end = text[pos..].find('\n').map(|i| pos + i).unwrap_or(text.len());
    text[line_start..line_end].trim()
}

rfind searches backwards from the end of the slice — the 'r' stands for reverse. The Java equivalent is String.lastIndexOf('\n', pos). Slicing text[..pos] restricts the backward search to before the match, and text[pos..] restricts the forward search to after it.

Slicing at \n boundaries is always safe — newlines are single ASCII bytes, so the byte after a \n is always a valid UTF-8 character boundary. The floor_char_boundary helper is not needed here.

Plug into search_docs — Step 1: filter positions

In search_docs, positions are currently collected and then immediately used for proximity scoring. Add the boilerplate filter in between, after positions.sort(); positions.dedup();: 

positions.sort();
positions.dedup();

// Build the boilerplate set from the body text and remove any position
// that lands on a line appearing 3+ times (running headers, footers, page titles).
let exact_positions_before_filter = positions.len();
let boilerplate = repeated_lines(&lower[body_start..]);
let positions: Vec<usize> = positions
    .into_iter()
    .filter(|&pos| !boilerplate.contains(&normalize_line(line_at_pos(&lower, pos))))
    .collect();

// Skip only when exact matches existed but ALL landed on boilerplate lines.
// If exact_positions_before_filter is 0, this was a fuzzy-only match — let
// it fall through to the body-start excerpt fallback below.
if exact_positions_before_filter > 0 && positions.is_empty() {
    continue;
}

// Proximity scoring: rank each position by how many distinct query terms …

lower[body_start..] restricts frequency counting to the document body — not the frontmatter. We build the boilerplate set once per document and reuse it for all position checks.

The let positions: Vec<usize> = positions.into_iter()… re-binds positions to the filtered result. This shadows the original binding — everything below this line sees only the clean positions.

The guard needs to distinguish two different reasons positions can be empty after filtering:

  • All exact matches were boilerplate — exact_positions_before_filter > 0 and positions.is_empty(). Skip the document: nothing useful to show.
  • Fuzzy-only match — exact_positions_before_filter == 0. The query matched via edit-distance (e.g. "reflecor" → "reflector") but match_indices found no exact substring. The body-start fallback excerpt below is the right response; continue would incorrectly discard it.

Plug into search_docs — Step 2: fix the ranking score

The current score at the bottom of the matching block counts raw occurrences including headers. Replace it with the filtered position count: 

// Before — counts every occurrence including repeated headers:
let score: usize = terms.iter().map(|term| lower.matches(*term).count()).sum();

// After — counts only non-boilerplate exact match positions:
let score = positions.len();

A document where the query term only appears in a 50-page running header will have all its positions filtered out. positions.is_empty() will be true, and the code skips that document entirely — it never appears in results at all. A short document with one precise body hit scores 1 and is returned.

Build and test

cargo build

Re-run a search that was previously returning bad excerpts. The excerpts should now anchor on body content rather than page titles.

Tests

Unit tests for the helpers — add to the mod tests block:

#[test]
fn repeated_lines_excludes_boilerplate() {
    let text = "KUKA Technical Reference\n\
                KUKA Technical Reference\n\
                KUKA Technical Reference\n\
                KUKA Technical Reference\n\n\
                Reflectors must be mounted at 150mm height.\n\n\
                KUKA Technical Reference\n\
                KUKA Technical Reference";
    let rep = repeated_lines(text);
    assert!(rep.contains("kuka technical reference"));
    assert!(!rep.contains("reflectors must be mounted at 150mm height."));
}

#[test]
fn line_at_pos_returns_correct_line() {
    let text = "first line\nsecond line\nthird line";
    assert_eq!(line_at_pos(text, 15), "second line");
}

Integration tests for search_docs behavior — add to the mod integration_tests block. These call search_docs_in (the extracted free function) directly with a temp directory path, so they never touch the process-global KUKA_KNOWLEDGE_DIR environment variable and are safe to run in parallel. 

#[test]
fn search_ignores_boilerplate_only_matches() {
    let temp_dir = tempfile::TempDir::new().unwrap();
    let doc = "---\ntype: technical-note\ntitle: Test\ndescription: .\n\
               resource: r.pdf\ntags: []\ntimestamp: 2026-01-01T00:00:00Z\n---\n\n\
               KUKA Manual Header\nKUKA Manual Header\nKUKA Manual Header\n\
               KUKA Manual Header\n\nOnly reflector content here.\n\
               KUKA Manual Header\nKUKA Manual Header";
    fs::write(temp_dir.path().join("doc.md"), doc).unwrap();

    let result = search_docs_in(temp_dir.path(), "kuka".to_string()).unwrap();
    assert!(format!("{:?}", result).contains("No results found"));
}
