Refactor Appendix 20

Word-Boundary Matching — Short Terms Without False Credit

Refactor step 14 KUKA MCP Server in Rust

Why a tiny matcher bug became a ranking bug

Step 12 stopped one-character and two-character terms from exploding into broad substring searches. That still left a three-character edge case: red could match vocabulary keys such as wired, powered, and reduced because the matcher used raw contains for every term of length three or more.

Step 11 then made that more visible. Search ranking now rewards documents for matching more distinct query terms, so a false red match is not just noisy text evidence. It gives the document false coverage credit and can change the order of results.

The old two-tier gate

The previous rule had one floor: exact matching below MIN_SUBSTRING_TERM_LEN, substring or fuzzy matching above it. Since that constant is 3, the word red crossed the floor and entered raw substring matching.

Rust before:

fn matching_keys(&self, term: &str) -> Vec<&str> {
    self.vocab
        .keys()
        .filter(|key| {
            if term.len() < MIN_SUBSTRING_TERM_LEN {
                key.as_str() == term
            } else {
                key.contains(term) || within_typo_tolerance(key, term)
            }
        })
        .map(String::as_str)
        .collect()
}

Java equivalent:

boolean matches(String key, String term) {
    if (term.length() < MIN_SUBSTRING_TERM_LEN) {
        return key.equals(term);
    }
    return key.contains(term) || withinTypoTolerance(key, term);
}

The new three-tier gate

The fix keeps the exact tier for one-character and two-character terms, keeps the old substring/fuzzy tier for terms long enough to be useful, and inserts a middle tier for three-character terms. The middle tier requires the vocabulary key to start with the term and allows only a short suffix.

Rust after:

pub(crate) const SHORT_TERM_MAX_SUFFIX: usize = 2;

fn matching_keys(&self, term: &str) -> Vec<&str> {
    self.vocab
        .keys()
        .filter(|key| {
            if term.len() < MIN_SUBSTRING_TERM_LEN {
                key.as_str() == term
            } else if term.len() < FUZZY_MIN_TERM_LEN {
                key.starts_with(term) && key.len() <= term.len() + SHORT_TERM_MAX_SUFFIX
            } else {
                key.contains(term) || within_typo_tolerance(key, term)
            }
        })
        .map(String::as_str)
        .collect()
}

Java equivalent:

static final int SHORT_TERM_MAX_SUFFIX = 2;

boolean matches(String key, String term) {
    if (term.length() < MIN_SUBSTRING_TERM_LEN) {
        return key.equals(term);
    }
    if (term.length() < FUZZY_MIN_TERM_LEN) {
        return key.startsWith(term)
            && key.length() <= term.length() + SHORT_TERM_MAX_SUFFIX;
    }
    return key.contains(term) || withinTypoTolerance(key, term);
}

Why prefix plus a length guard

A plain prefix rule fixes the original noise class: red no longer matches wired or powered. The length guard fixes the next loophole: red should not match reduce or redundant just because they start with the same three letters. With a two-character suffix allowance, plural and small inflection forms such as reds, AMRs, and boxes still work.

In Java, you might model this as named predicates. In Rust, the chain of if branches is clearer because the term length tiers are ordered and mutually exclusive. 

Predicate<String> exact = key -> key.equals(term);
Predicate<String> shortPrefix = key ->
    key.startsWith(term) && key.length() <= term.length() + SHORT_TERM_MAX_SUFFIX;
Predicate<String> longRecall = key ->
    key.contains(term) || withinTypoTolerance(key, term);

What the tests protect

The new regression fixture creates one document whose only related words are wired, powered, reduced, and redundant, plus a second document with genuine red and reds tokens. Searching for red must return only the genuine document.

The existing amr test now also includes AMRs. That protects the reason this is not exact-only: real manuals pluralize short technical words, and those matches should still count.
