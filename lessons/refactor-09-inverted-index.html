Turning the search inside out

Until now, every query read every document from disk, lowercased the entire corpus, and scanned every word. Fine for 9 documents; hopeless for hundreds. The inverted index flips the direction of the question — instead of "document, do you contain this term?" asked N times, it's "term, which documents contain you?" asked once:

Forward (old):  document → all its words    (scan everything per query)
Inverted (new): term → all its documents   (one map lookup per query)

Rust — the core structures in index.rs:

pub struct Index {
    docs: Vec<DocMeta>,                    // doc_id = position in this Vec
    vocab: HashMap<String, Vec<Posting>>,  // term → where it appears
}

struct Posting {
    doc_id: u32,
    freq: u32,           // every occurrence counts…
    positions: Vec<u32>, // …but only 16 byte-offsets are stored
}

Java equivalent — and a familiar name:

Map<String, List<Posting>> vocab = new HashMap<>();
record Posting(int docId, int freq, int[] positions) {}

This is, at miniature scale, exactly the data structure inside Lucene (and therefore Elasticsearch): vocabulary → posting lists. Building it by hand is the best way to understand what those systems do — and when this project outgrows the hand-rolled version, the swap target (tantivy, Rust's Lucene) will feel familiar rather than magical.

Three design decisions worth remembering

1. The position cap. Positions are the only part of the index that grows with corpus size rather than vocabulary size — uncapped, they cost roughly one u32 per word of text, approaching the size of the corpus itself. Capping at 16 per (term, document) bounds memory while keeping far more excerpt anchors than the 3 we ever show, and freq keeps counting past the cap so scoring stays honest. A test pins the behavior: 50 occurrences → freq == 50, positions.len() == 16.

2. Boilerplate is filtered at build time, not query time. The old engine computed repeated-line sets per document on every query and filtered match positions afterwards. Now the index build walks each document line by line and simply never indexes tokens on boilerplate lines. The whole query-time filtering dance was deleted — repeated headers don't exist as far as the index is concerned. (The test asserts vocab doesn't even contain "kuka" for a document where it appears only in headers.) Update (steps 6–7): cleanup moved all the way upstream to the extractor, which strips these lines (plus TOC dot-leaders and page markers) from the bundle files themselves — and this index-time filter was subsequently REMOVED: chunks carry no page structure, so it could not tell a running header from legitimately repeated content, and it silently un-indexed a real lookup table. The index now trusts the bundle. See Appendix 11 and Appendix 12.

3. Per-token lowercasing, original offsets. The very first code review of this project flagged a latent bug: match positions were computed on content.to_lowercase() but used to slice the original — and a handful of Unicode characters change byte length when lowercased. The index eliminates the bug by construction: each token is lowercased individually to form its vocabulary key, while its recorded position always points into the original bytes. There is no lowercased copy of the document anywhere — nothing whose offsets could disagree.

Fuzzy matching got orders of magnitude cheaper

The old engine ran levenshtein against every word of every document per query term. The new one scans the vocabulary — the set of unique words, which is tiny relative to the corpus and grows very slowly no matter how many documents you add (the real bundle: 12 documents but only 1,051 unique terms): 

fn matching_keys(&self, term: &str) -> Vec<&str> {
    self.vocab
        .keys()
        .filter(|key| key.contains(term) || within_typo_tolerance(key, term))
        .map(String::as_str)
        .collect()
}

Substring containment covers exact matches and "reflector" finding "reflectors"; the typo tolerance handles misspellings. A pleasant side effect: fuzzy matches now carry real positions (the positions of the matched vocabulary words), so a typo query gets genuine excerpts instead of the old "fall back to the start of the document."

Scoring: density beats bulk

score = (freq_total × 1000) / token_count     // per document

The old score was a raw match count, so a 100-page manual mentioning "reflector" forty times in passing outranked a focused 2-page note. Dividing by the document's token count (×1000 to stay an integer — SearchHit.score is still a usize) rewards density. This works especially well with step 5a's chunks, which are near-uniform in size: the chunk about mission status now beats the chunk that merely mentions it. This is the "TF" half of TF-IDF; the IDF half (rare terms count for more) is the natural next refinement if ranking ever feels off.

Excerpts without loading documents

DocMeta deliberately does not hold the document body — after the build, bodies are dropped and memory stays flat regardless of corpus size. Excerpts are read straight from disk around the recorded positions:

Rust:

let mut file = std::fs::File::open(path)?;
file.seek(SeekFrom::Start(start))?;
let mut buf = Vec::with_capacity(len as usize);
file.take(len).read_to_end(&mut buf)?;
String::from_utf8_lossy(&buf)   // partial chars at the cut edges → U+FFFD, trimmed

Java equivalent:

try (var raf = new RandomAccessFile(path, "r")) {
    raf.seek(start);
    byte[] buf = new byte[len];
    raf.readFully(buf);
    new String(buf, StandardCharsets.UTF_8);   // malformed edges → U+FFFD
}

A 450-byte read per excerpt, at most three per hit. No document is ever loaded whole at query time — the property that makes gigabyte corpora feasible.
