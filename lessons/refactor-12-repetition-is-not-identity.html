Repetition Is Not Identity — When Cleaning Eats the Data

Refactor step 7 — a data-loss bug in our own cleaner KUKA MCP Server in Rust

The incident

A user asked: "what are valid values for RobotType?" The MQTT Payload Definitions PDF contains a clean lookup table (0 = KMP 250P, 1 = KMP 600P, … 5 = KMF CB-1500P). Search found nothing. A Claude session investigated and concluded the table must be an image inside the PDF, needing OCR.

That diagnosis was plausible, confident — and wrong. One command disproved it: 

$ pdftotext -layout "MQTT Payload Definitions.pdf" - | grep -n "250P"
350:    0                     KMP 250P     ← the table IS text…
436: 0                           KMP 250P  ← …and appears 3 times
524: 0                           KMP 250P

$ grep -l "250P" knowledge/*.md
(nothing)                                 ← but never reached the bundle

The table was extracted perfectly and then deleted by our own cleaner. Debugging moral #1: before accepting a story about why data is missing ("it's an image", "it needs OCR"), check each pipeline stage for where it actually disappears. The evidence took two greps.

The flawed rule

The step-6 cleaner classified any line repeated ≥ 3 times as a running header/footer. But the RobotType table is printed under three payload sections — MissionCommand, MultiMissionCommand, MultiWorkflowCommand — so every one of its lines "repeated 3 times" and was stripped. So was the RobotType: (optional)… description sentence, and every other standard sentence that legitimately recurs under each payload definition. The user's report — "quite a bit of data missing for the MissionCommand payload" — was precisely this rule at work.

The false assumption: repetition identifies boilerplate. It doesn't. Real documents repeat content on purpose. What actually distinguishes a running header is where it repeats: at the edges of pages. In machine-learning terms, we were classifying on one feature (frequency) when the discriminating feature is position.

The fix: position-aware (zone-based) detection

const HEADER_ZONE_LINES: usize = 10;   // top-of-page candidate zone
const FOOTER_ZONE_LINES: usize = 5;    // bottom-of-page candidate zone

// Pass 1: count, per normalized line, total occurrences AND how many
//         fall inside a page-edge zone.
// Pass 2: boilerplate = repeats >= 3  AND  >= 80% of occurrences in zones.
// Pass 3: rebuild the text without boilerplate / page markers / TOC lines.

KUKA ROBOTICS CORP. (24 occurrences, all page-top) is still stripped. The RobotType table (3 occurrences, all mid-page) survives. The 80% zone-share tolerance means content that once drifts to a page top — a section that happens to start right after a page break — doesn't get misclassified from a single occurrence. And since "Page 8 of 24" is unique per page (repetition can never catch it), page markers are now matched by shape instead: a small match on the word pattern ["page", n, "of", m] — Rust slice patterns doing the work a regex would do in Java, without the dependency.

The index filter had to go too

Appendix 9 proudly filtered repeated lines at index-build time. Same flaw — and worse, it is unfixable there: by the time the index sees a chunk, page breaks are gone, so position-aware detection is impossible. Had we kept it, a chunk containing the table three times would silently un-index it again. It was removed entirely, which also simplified Index::build from a line-walking loop back to a single tokenize(doc.body()) pass.

The design principle that fell out: each data-quality rule lives at exactly one stage — the earliest stage that has the information the rule needs. Boilerplate detection needs page structure → it lives in extraction, and nowhere else. "Defense in depth" sounds prudent, but a duplicate filter with less information than the primary one isn't defense — it's a second chance to destroy data.

A test that failed correctly

The new test for "repeated mid-page content survives" failed on first run — and the failure was the algorithm being right: the synthetic page was so short that the table sat within 5 lines of the page bottom, i.e. inside the footer zone, where stripping is intended. The fix was to make the fixture honest (add closing text below the table, as the real document has), not to weaken the algorithm. When a test fails, "which one is wrong?" is a genuine question — here the test was, and understanding why sharpened the zone semantics.

The complete table, straight from the excerpt. And the query "RobotType valid values" now returns the step-6 retry hint instead of a bare "no results" — the two mechanisms compose: the hint prompts a reworded query, and the reworded query finds the restored data.
