Refactor step 3 — commit 793fa54 KUKA MCP Server in Rust

Stop words: from per-call rebuild to LazyLock

Every call to search used to build the stop-word HashSet from scratch — fifty-odd insertions, allocated and thrown away per query. The set never changes, so it belongs in a static built exactly once:

Rust:

use std::sync::LazyLock;

static STOP_WORDS: LazyLock<HashSet<&str>> = LazyLock::new(|| {
    HashSet::from(["a", "an", "the", "is", /* … */ "nor"])
});

// Used like a normal value — first access runs the closure, later
// accesses reuse the same set:
.filter(|word| !STOP_WORDS.contains(word))

Java equivalents — two familiar shapes rolled into one:

// Eager static initializer — Java's usual answer:
private static final Set<String> STOP_WORDS = Set.of("a", "an", "the", /* … */ "nor");

// Lazy, thread-safe, initialize-on-first-use — what LazyLock actually is
// (the "initialization-on-demand holder" idiom, without the ceremony):
private static class Holder {
    static final Set<String> STOP_WORDS = buildStopWords();
}

Why doesn't Rust just allow static STOP_WORDS: HashSet = HashSet::from(…) directly? Because statics must be computable at compile time, and building a HashSet allocates on the heap at runtime. LazyLock bridges the gap: a compile-time-constant wrapper around a runtime-initialized value, with thread-safe once-only initialization guaranteed by the type system rather than by class-loading rules as in Java.

Magic numbers become tuning knobs

The search code was sprinkled with bare numbers: 500, 150, 300, 3, 400, 7 — each meaningful, none labelled. They're now const declarations at the top of search.rs, so the tunable surface of the engine is one screenful: 

const PROXIMITY_WINDOW: usize = 500;      // ± window for co-occurrence scoring
const EXCERPT_BEFORE: usize = 150;        // excerpt context before a match
const EXCERPT_AFTER: usize = 300;         //   …and after
const MAX_EXCERPTS: usize = 3;            // excerpts shown per document
const FALLBACK_EXCERPT_LEN: usize = 400;  // body-start excerpt for fuzzy-only hits
const BOILERPLATE_MIN_REPEATS: usize = 3; // line repeated ≥ this = header/footer
const FUZZY_MIN_TERM_LEN: usize = 4;      // shorter terms must match exactly
const FUZZY_ONE_TYPO_MAX_LEN: usize = 7;  // ≤ this: 1 typo; longer: 2

A Rust const is closest to Java's static final int, with one difference worth knowing: consts are inlined at every use site — they have no address, no memory of their own. (A Rust static, by contrast, is one shared memory location, like a true Java static field.) Rule of thumb: plain numbers and short strings → const; anything heap-allocated or identity-sensitive → static, usually with LazyLock.

This step also cleared the last clippy warnings, mostly by asking a question Java developers never face: does this sort need to be stable? positions.sort_unstable() — positions are deduped integers, stability is meaningless, and the unstable sort is faster. But the final hit ranking kept a stable sort_by_key on purpose: equal-score documents should stay in bundle order rather than shuffle between runs. Java's Collections.sort is always stable, so the choice never comes up; Rust makes you pick, and the pick documents intent.

Test layers: the library tests the engine, the binary tests the wording

Moving presentation into main.rs raised a question: where do the tests for "missing bundle dir → isError: true" live now? That behavior belongs to the tool layer, so its tests moved into main.rs itself — binaries can have #[cfg(test)] modules too, and cargo test runs them as a separate suite (you can see it in the output: 23 lib tests + 4 bin tests).

One boundary surprised us on the way: the binary tests cannot use the library's test_util fixture. A binary is a separate crate, and when it links against the library, the library is compiled without cfg(test) — the test-only module simply doesn't exist in that build. The Java analogy: test fixtures in module A's src/test/java are invisible to module B's tests unless you publish a separate test-jar. Same principle, enforced by the compiler. So main.rs's tests build their own little fixture, and each layer asserts only what it owns:

LayerTests live inAssert on
engine (matching, scoring, excerpts) search.rs (lib) SearchHit fields
loading (frontmatter, missing dir) bundle.rs (lib) Document fields, Result
presentation (wording, isError, guards) main.rs (bin) CallToolResult text and flags
