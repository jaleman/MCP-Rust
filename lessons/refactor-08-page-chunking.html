Chunking Documents on Page Boundaries

Refactor step 5a — commit a79e531 KUKA MCP Server in Rust

Why chunk at all

Three consumers of this server all want document units of a few kilobytes, never a 300-page manual: search excerpts (anchored in a small file, always relevant), MCP resources (read_resource returns a whole file — a gigabyte resource would break any client), and the AI agent's context window (an agent can read several 8 KB hits; it cannot read a manual). Chunking at extract time solves all three at once — and improves the results: "MQTT Payload Definitions (pages 9–16)" tells the reader where to look in the real PDF.

The boundary comes free: pdftotext separates pages with a form-feed character (\x0c, the ASCII "new page" control code from the typewriter era). Split on it, and you have page-accurate sections with no parsing.

Update (step 6): the extraction pipeline now cleans the text before chunking — repeated headers/footers and table-of-contents lines are stripped so they never reach the bundle files. See Refactor Appendix 11.

The accumulator pattern

The core of chunk.rs is a pattern you've written many times in Java: iterate, accumulate into a buffer, flush when full.

Rust:

let mut chunks: Vec<Chunk> = Vec::new();
let mut cur_text = String::new();
let mut cur_first = 0;
let mut cur_last = 0;

for (i, raw_page) in text.split('\x0c').enumerate() {
    let page_no = i + 1;
    let page = raw_page.trim();
    if page.is_empty() { continue; }        // blank page: skip, numbering kept

    // Would this page overflow the open chunk? Flush it first.
    if !cur_text.is_empty() && cur_text.len() + 2 + page.len() > CHUNK_TARGET_BYTES {
        chunks.push(Chunk {
            text: std::mem::take(&mut cur_text),   // ← the interesting line
            first_page: cur_first,
            last_page: cur_last,
        });
    }
    if cur_text.is_empty() { cur_first = page_no; } else { cur_text.push_str("\n\n"); }
    cur_text.push_str(page);
    cur_last = page_no;
}

Java equivalent:

List<Chunk> chunks = new ArrayList<>();
StringBuilder curText = new StringBuilder();
int curFirst = 0, curLast = 0;

for (int i = 0; i < pages.length; i++) {
    String page = pages[i].trim();
    if (page.isEmpty()) continue;

    if (curText.length() > 0 && curText.length() + 2 + page.length() > TARGET) {
        chunks.add(new Chunk(curText.toString(), curFirst, curLast));
        curText.setLength(0);                    // reset the buffer
    }
    …
}

The line worth studying is std::mem::take(&mut cur_text). Java's curText.toString() copies the buffer and then clears it. Rust's take does neither: it moves the String into the Chunk (pointer handoff, zero copy) and leaves a fresh empty String behind. Without it, the borrow checker would rightly refuse — you can't move cur_text into the Vec while the loop still owns it for the next iteration. mem::take is the idiomatic "move it out and reset" tool, and there is no Java equivalent because Java never has to prove who owns a value.

Also note the loop destructures enumerate() into (i, raw_page) — pattern matching in the for-loop head, where Java would track an index variable.

Optional frontmatter fields

Chunk files carry two new fields linking them to their parent document; a stand-alone document's frontmatter must not change at all. That requirement maps exactly onto Option: 

pub struct OkfFrontmatter {
    …
    pub parent: Option<String>,   // slug of the parent document, if chunked
    pub pages:  Option<String>,   // "12-18", if chunked
    …
}

// In render():
if let Some(parent) = &self.parent {
    out.push_str(&format!("parent: {parent}\n"));
}

Refusing to write garbage

Chunking exposed an old bug's root cause. The bundle contained a 213-byte emergencyfirealarm.md — frontmatter with an empty body — which matched searches via its title and returned blank excerpts (see the finding in Appendix 5). The new pipeline makes that impossible: 

let chunks = chunk_pages(text.trim());
if chunks.is_empty() {
    bail!("no text could be extracted (image-only or empty PDF?)");
}

Re-extracting the real bundle proved the point: 9 of 10 PDFs extracted cleanly (two of them into 2 and 3 page-ranged chunks), and EmergencyFireAlarm.pdf — which turns out to be a scanned, image-only PDF — was refused with a clear error instead of silently producing an empty file. Fixing that document is an OCR problem, now visibly so.
