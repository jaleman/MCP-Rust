Serving Diagrams — Binary Resources over MCP

Refactor step 9b KUKA MCP Server in Rust

Why text alone wasn't enough

KUKA documentation is full of diagrams — building maps, clearance drawings, wiring figures — that carry information no text excerpt can. Modern AI models are multimodal: hand one an image and it can read it, describe it, and answer questions about it. The missing piece was plumbing: getting a PNG from inside a PDF, through the bundle, over MCP, and into the agent's context. Three stages, mirroring the text pipeline exactly.

Stage 1: extraction — deciding what counts as a diagram

pdfimages -png -p dumps every embedded image with its page number in the filename. "Every embedded image" includes the KUKA logo on all 24 page headers — so, like the text cleaner in Appendix 12, the interesting work is filtering: 

const MIN_IMAGE_BYTES: u64 = 10 * 1024;   // logos are ~2–5 KB; diagrams are bigger
const MAX_IMAGES_PER_DOC: usize = 20;     // bound the bundle's growth

// Identical bytes = the same graphic repeated on another page
let mut hasher = DefaultHasher::new();
bytes.hash(&mut hasher);
if !seen_hashes.insert(hasher.finish()) { continue; }

Three cheap heuristics — size floor, byte-hash dedupe, per-document cap — and the corpus yields 50 genuine diagrams instead of hundreds of header logos. The Java equivalent of the dedupe is a HashSet<String> of MessageDigest hashes; Rust's DefaultHasher is the std-library, no-dependency version (not cryptographic — fine for dedupe). Failure policy matches the OCR lesson: diagram extraction failing never blocks document ingestion — it logs a warning and the document ships without images.

Stage 2: association — pages are the join key

Each image knows its page (img-014-002.png → page 14); each chunk knows its page range. The join is a filter, recorded in frontmatter the same way as every other document property: 

images: [building-map-and-extension-map-p002-1.png, ...]   // only THIS chunk's pages

From there the field rides the existing rails: Document parses it, DocMeta carries it, SearchHit exposes it — the same field-threading pattern as every attribute added since Appendix 3.

Stage 3: serving — base64 blobs and in-band advertising

MCP resources carry text or binary. Binary travels as base64 inside JSON — the same trick as embedding images in data URLs or SOAP attachments:

Rust:

use base64::Engine as _;   // trait must be in scope for .encode()

let bytes = std::fs::read(&path)?;
let blob = base64::engine::general_purpose::STANDARD.encode(&bytes);
let contents = ResourceContents::blob(blob, uri.clone()).with_mime_type("image/png");

Java equivalent:

byte[] bytes = Files.readAllBytes(path);
String blob = Base64.getEncoder().encodeToString(bytes);   // static, always available

The Rust wrinkle: encode lives on the Engine trait, so the trait must be imported (as _ imports it anonymously — we need its methods, not its name). Java's Base64.getEncoder() is a static factory; Rust makes the engine choice (standard vs URL-safe alphabet) an explicit value.

Discovery follows the step-6 doctrine — tool output is the steering channel that reaches every harness. Hits now advertise their diagrams in-band, next to the text they illustrate: 

• EmergencyFireAlarm (pages 1-6)
  Resource: kuka://docs/emergencyfirealarm-p001-006
  Diagrams: kuka://images/emergencyfirealarm-p002-1.png, kuka://images/...

The traversal guard from Appendix 4 guards the new URI space too — same function, second caller.

The payoff, end to end

The scanned fire-alarm document — invisible until step 8's OCR — now returns text excerpts plus eight slide images. An agent asked "what does the fire alarm procedure look like?" searches, sees the Diagrams line, reads kuka://images/emergencyfirealarm-p002-1.png, and receives the actual slide ("blob":"iVBORw0KGgo…" — PNG magic bytes, base64'd) to look at. Text search finds the place; images show what words can't.
