Office and Text Ingestion

Refactor step 9a — more source formats, same extraction pipeline KUKA MCP Server in Rust

The design constraint

Step 9a adds .docx, .doc, .pptx, .ppt, and .txt input support, but it deliberately does not change search, indexing, frontmatter parsing, or chunking. That is the important part. A new source format should enter the same clean → chunk → OKF write path as PDFs, because every downstream guarantee already lives there: page ranges, OCR tags, small resources, and stable resource URIs.

The Java instinct might be to reach for Apache POI and parse Word or PowerPoint in process. That can work, but it asks the application to learn every Office layout rule. Here LibreOffice already knows how to render Office documents as PDFs, and our PDF path already knows how to extract page-aware text. So the small Rust program shells out to soffice and keeps the rest of the architecture unchanged.

PDF → pdftotext / OCR → clean → chunk → OKF markdown
Office → soffice temp PDF → pdftotext / OCR → clean → chunk → OKF markdown
TXT → read_to_string → clean → chunk → OKF markdown

Routing by extension

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum IngestKind {
    Pdf,
    Office,
    Text,
}

fn ingest_kind(path: &Path) -> Option<IngestKind> {
    let extension = path.extension()?.to_str()?;
    if extension.eq_ignore_ascii_case("pdf") {
        Some(IngestKind::Pdf)
    } else if ["docx", "doc", "pptx", "ppt"]
        .iter()
        .any(|candidate| extension.eq_ignore_ascii_case(candidate))
    {
        Some(IngestKind::Office)
    } else if extension.eq_ignore_ascii_case("txt") {
        Some(IngestKind::Text)
    } else {
        None
    }
}

The enum is tiny, but it makes the rest of the program more explicit. Batch mode silently skips None. Single-file mode turns None into an error. Once a path is classified, a match forces every supported kind to have a handling branch. In Java, this is the same clarity you get from a small enum and a switch expression instead of a growing chain of string comparisons scattered through the method.

Office conversion

fn convert_office_to_pdf(input: &Path, temp_dir: &tempfile::TempDir) -> Result<PathBuf> {
    let output = std::process::Command::new("soffice")
        .args(["--headless", "--convert-to", "pdf", "--outdir"])
        .arg(temp_dir.path())
        .arg(input)
        .output()
        .context("soffice not found — install libreoffice-writer libreoffice-impress")?;

    if !output.status.success() {
        let stderr = String::from_utf8_lossy(&output.stderr);
        bail!("soffice failed: {stderr}");
    }

    let pdf = temp_dir.path().join(format!(
        "{}.pdf",
        input.file_stem().unwrap().to_string_lossy()
    ));
    if !pdf.exists() {
        bail!("soffice produced no output for {}", input.display());
    }

    Ok(pdf)
}

Two details are worth noticing. First, the output PDF lives in a TempDir. When the directory value drops, the converted PDF goes away with it. Java can model the same lifecycle with Files.createTempDirectory plus a finally cleanup, but Rust makes the ownership visible in the function signature. Second, LibreOffice can exit successfully without producing a PDF, so the code checks the expected output file explicitly. Exit code success is not the same thing as usable data.

One write path

fn process_document(
    source_path: &Path,
    knowledge_dir: &Path,
    force_pdftotext: bool,
    written: &mut HashSet<PathBuf>,
) -> Result<()> {
    println!("Extracting: {}", source_path.display());

    let kind = ingest_kind(source_path)
        .with_context(|| format!("unsupported input type: {}", source_path.display()))?;

    let (text, used_ocr) = match kind {
        IngestKind::Pdf => extract_pdf_text(source_path, force_pdftotext)?,
        IngestKind::Office => {
            let temp_dir = tempfile::tempdir().context("cannot create temp dir for Office conversion")?;
            let pdf = convert_office_to_pdf(source_path, &temp_dir)?;
            extract_pdf_text(&pdf, true)?
        }
        IngestKind::Text => {
            let text = fs::read_to_string(source_path)
                .with_context(|| format!("cannot read text file {}", source_path.display()))?;
            (text, false)
        }
    };

    write_document(source_path, text, used_ocr, knowledge_dir, written)
}

source_path is the original document, even when the extractor temporarily reads a converted PDF. That keeps provenance honest: title, slug, and resource: kuka-docs/... name the user's source file, not an implementation detail in /tmp. Text files take the short path: read the string, skip OCR, and still use the same cleaner and chunk writer as everything else.

Java equivalent

enum IngestKind { PDF, OFFICE, TEXT }

Path tempDir = Files.createTempDirectory("office-pdf");
try {
    Process process = new ProcessBuilder(
            "soffice", "--headless", "--convert-to", "pdf", "--outdir",
            tempDir.toString(), input.toString())
        .start();

    int code = process.waitFor();
    if (code != 0) {
        throw new IOException("soffice failed");
    }

    Path pdf = tempDir.resolve(stripExtension(input.getFileName()) + ".pdf");
    if (!Files.exists(pdf)) {
        throw new IOException("soffice produced no output");
    }

    return extractPdfText(pdf);
} finally {
    deleteRecursively(tempDir);
}

This is not conceptually foreign to Java. The difference is where the language makes you pay attention. Java asks you to be disciplined with cleanup and checked exceptions. Rust asks you to encode fallible steps in the return type and lets owned temporary directories clean themselves up when they leave scope.
