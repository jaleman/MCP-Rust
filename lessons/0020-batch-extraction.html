Lesson 20

Batch Extraction

~30 minutes Hands-on — edit, build, cross-compile, run on Windows KUKA MCP Server in Rust

What you're building

extract.rs currently has the PDF path hardcoded and only processes one file. This lesson turns it into a proper Windows CLI tool that takes two explicit arguments — the input (a single PDF or a directory) and the output directory — and cross-compiles it to a .exe you can run directly on Windows without Cargo or Rust installed. 

# Batch migration — all 16 PDFs at once
extract.exe kuka-docs\ knowledge\

# Add one new document later
extract.exe kuka-docs\new-manual.pdf knowledge\
D:\projects\KUKA\
├── extract.exe          ← tool lives here
├── kuka-docs\           ← source PDFs
└── knowledge\           ← OKF output files

New concepts — translated from Java

Concept 1: Reading multiple CLI arguments with args().skip()

std::env::args() returns an iterator. .skip(1) discards the binary name at index 0, leaving only user-supplied arguments. Calling .next() twice on the iterator retrieves the first and second arguments in order. 

// Java: arguments arrive as a pre-built array
// public static void main(String[] args) {
//     String input  = args[0];
//     String output = args[1];
// }

// Rust: arguments come from an iterator; skip(1) drops the binary name
let mut args = std::env::args().skip(1);
let input_arg  = args.next(); // first user argument  — Option<String>
let output_arg = args.next(); // second user argument — Option<String>

Both calls return Option<String> — Some(value) if the argument was supplied, None if the user didn't provide it. Matching on both at once lets you check for missing arguments in one step: 

// Match on the tuple of both Options simultaneously.
// Only the (Some, Some) arm is a valid call — anything else prints usage and exits.
let (input_arg, output_arg) = match (args.next(), args.next()) {
    (Some(i), Some(o)) => (i, o),
    _ => {
        eprintln!("Usage: extract <pdf-file-or-directory> <output-dir>");
        std::process::exit(1);
    }
};

Concept 2: Cross-compiling to Windows from Linux

Cross-compilation means building a binary for a different OS than the one you're compiling on. Your dev container runs Linux; you want a Windows .exe. Rust supports this natively through targets — named combinations of OS, CPU, and toolchain.

The Windows GNU target (x86_64-pc-windows-gnu) uses the open-source MinGW toolchain, which is straightforward to install on Linux. The alternative (x86_64-pc-windows-msvc) requires the Windows SDK and is not practical to set up in a Linux container.

Step 1 — Replace extract.rs

Replace the entire contents of mcp-server/src/extract.rs with: 

use anyhow::Result;
use std::path::Path;

// Standalone binary that extracts KUKA PDFs to OKF markdown files.
//
// Usage:
//   extract <input> <output-dir>
//
//   <input>      — a single PDF file or a directory of PDFs
//   <output-dir> — the knowledge/ directory to write OKF files into
fn main() {
    let mut args = std::env::args().skip(1); // skip the binary name at index 0

    // Require exactly two arguments — input path and output directory
    let (input_arg, output_arg) = match (args.next(), args.next()) {
        (Some(i), Some(o)) => (i, o),
        _ => {
            eprintln!("Usage: extract <pdf-file-or-directory> <output-dir>");
            eprintln!("Example: extract kuka-docs\\ knowledge\\");
            std::process::exit(1);
        }
    };

    let input = Path::new(&input_arg);
    let knowledge_dir = Path::new(&output_arg);

    // Verify the output directory exists before processing any files
    if !knowledge_dir.is_dir() {
        eprintln!("Output directory does not exist: {}", knowledge_dir.display());
        std::process::exit(1);
    }

    if input.is_file() {
        // Single-file mode — exit non-zero on failure so callers can detect it
        if let Err(e) = process_pdf(input, knowledge_dir) {
            eprintln!("Error: {e}");
            std::process::exit(1);
        }
    } else if input.is_dir() {
        // Batch mode — process every PDF, logging failures and continuing
        let entries = std::fs::read_dir(input).unwrap_or_else(|e| {
            eprintln!("Cannot read directory: {e}");
            std::process::exit(1);
        });

        let mut ok: usize = 0;
        let mut failed: usize = 0;

        for entry in entries.flatten() {
            let path = entry.path();

            // Skip non-PDF files silently
            if path.extension().and_then(|e| e.to_str()) != Some("pdf") {
                continue;
            }

            // Match on the result so one bad PDF does not abort the batch
            match process_pdf(&path, knowledge_dir) {
                Ok(()) => ok += 1,
                Err(e) => {
                    eprintln!("Skipping {}: {e}", path.display());
                    failed += 1;
                }
            }
        }

        println!("\nDone: {ok} extracted, {failed} failed.");
    } else {
        eprintln!("Input path not found: {}", input.display());
        std::process::exit(1);
    }
}

// Extracts text from a single PDF and writes it as an OKF markdown file.
// Returns Ok(()) on success or an error the caller can log and continue from.
fn process_pdf(pdf_path: &Path, knowledge_dir: &Path) -> Result<()> {
    println!("Extracting: {}", pdf_path.display());

    // Extract all text — ? propagates any error back to the caller
    let text = pdf_extract::extract_text(pdf_path)?;

    // Build the output filename: lowercase with spaces replaced by dashes
    let stem = pdf_path.file_stem().unwrap().to_string_lossy();
    let filename = pdf_path.file_name().unwrap().to_string_lossy();
    let slug = stem
        .to_lowercase()
        .replace(' ', "-")
        .replace(['.', '(', ')'], ""); // strip punctuation that would break file paths

    // Write the OKF frontmatter block followed by the extracted text body
    let okf = format!(
"---
type: technical-note
title: {stem}
description: Extracted from KUKA documentation.
resource: kuka-docs/{filename}
tags: [extracted, technical-note]
timestamp: 2026-06-26T00:00:00Z
---

{body}",
        stem = stem,
        filename = filename,
        body = text.trim(),
    );

    let output = knowledge_dir.join(format!("{slug}.md"));
    std::fs::write(&output, &okf)?;
    println!("  → {}", output.display());

    Ok(())
}
Known limitation: headers and footers

pdf-extract extracts all text from a PDF including page headers and footers. For a KUKA technical note, every page typically has a header repeating the document title and revision number, and a footer with the page number and copyright. These repeated strings get extracted into the OKF body alongside the real content.

The practical effect: search terms like "mqtt" or "payload" appear hundreds of times throughout a large document — once in every page header — rather than only in the sections where they are meaningfully discussed. This can cause the search server to return excerpts anchored on header lines rather than on the actual answer content.

The search server compensates with proximity scoring (returning windows where multiple query terms co-occur), but the root fix is cleaner extraction. The lopdf crate exposes text position coordinates on the page, which would allow header and footer regions to be identified and stripped during extraction. This is planned as a future improvement once the current bundle is validated.

Step 3 — Cross-compile to Windows

mingw-w64 and the x86_64-pc-windows-gnu Rust target are both pre-installed by the devcontainer's postCreateCommand — nothing to set up manually. If you're ever on a container that doesn't have them: 

# Only needed outside the devcontainer — already handled by postCreateCommand
sudo apt-get install -y mingw-w64
rustup target add x86_64-pc-windows-gnu

Build the Windows release binary:

cargo build --release --bin extract --target x86_64-pc-windows-gnu

Your .exe is at:

target/x86_64-pc-windows-gnu/release/extract.exe

Copy it to your Windows machine alongside your kuka-docs\ and knowledge\ folders. It runs with no Rust or Cargo installation required.

Release vs debug builds

The --release flag enables compiler optimisations — the binary is smaller and faster. For a tool that processes large PDFs this matters. Debug builds (without --release) compile faster but run slower. Always use --release when building a binary you intend to distribute.
