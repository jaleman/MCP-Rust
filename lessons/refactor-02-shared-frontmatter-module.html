Refactor Appendix 2

One Frontmatter Module, Two Binaries

Refactor step 1 — commit 17967a8 KUKA MCP Server in Rust

The drift problem

The OKF frontmatter block is a contract between our two binaries: extract writes it at the top of every markdown file, and the server parses it back out for titles, types, and resource paths. Before this step, the two halves of that contract lived in different files with no connection to each other: 

// extract.rs — the WRITER: a hand-built format! string
let okf = format!(
    "---
type: technical-note
title: {stem}
description: Extracted from KUKA documentation.
resource: kuka-docs/{filename}
...
---

{body}", …);

// main.rs — the PARSER: a hand-rolled line scanner
fn extract_frontmatter_field(content: &str, field: &str) -> Option<String> { … }

Rename a field in the writer, and the parser silently starts returning None. No compiler error, no test failure — you'd discover it when search results lose their titles. In Java terms: it's a producer and a consumer exchanging JSON, where the producer builds the string with String.format() and the consumer parses it with substring() — instead of both sharing a Jackson-annotated DTO.

The fix: a shared type that both sides depend on

Rust — src/frontmatter.rs (new in this step):

/// The fields of an OKF frontmatter block, in the order they are written.
pub struct OkfFrontmatter {
    pub doc_type: String,
    pub title: String,
    pub description: String,
    pub resource: String,
    pub tags: String,       // rendered verbatim, e.g. "[extracted, technical-note]"
    pub timestamp: String,
}

impl OkfFrontmatter {
    /// Renders a complete OKF document: frontmatter block followed by the body.
    pub fn render(&self, body: &str) -> String {
        format!(
            "---\n\
             type: {}\n\
             title: {}\n\
             ...
             ---\n\n\
             {}",
            self.doc_type, self.title, /* … */ body,
        )
    }
}

// The parser moved into the SAME file:
pub fn extract_frontmatter_field(content: &str, field: &str) -> Option<String> { … }

The Java equivalent — a shared DTO in the core module:

// core module — the single definition both apps depend on
public record OkfFrontmatter(
        String docType,
        String title,
        String description,
        String resource,
        String tags,
        String timestamp) {

    /** Renders frontmatter + body — the writer half of the contract. */
    public String render(String body) {
        return """
               ---
               type: %s
               title: %s
               ...
               ---

               %s""".formatted(docType, title, /* … */ body);
    }

    /** The parser half lives beside it, in the same class. */
    public static Optional<String> extractField(String content, String field) { … }
}

A Rust struct with an impl block is the moral equivalent of that record: named fields, methods attached, no inheritance. Note there are no getters — Rust fields marked pub are accessed directly (fm.title), and immutability comes from the binding (let vs let mut) rather than from the type.

The extractor after the change

process_pdf in src/bin/extract.rs now builds the document through the shared type — compare with the raw format! it replaced: 

use mcp_server::frontmatter::OkfFrontmatter;

let frontmatter = OkfFrontmatter {
    doc_type: "technical-note".to_string(),
    title: stem.to_string(),
    description: "Extracted from KUKA documentation.".to_string(),
    resource: format!("kuka-docs/{filename}"),
    tags: "[extracted, technical-note]".to_string(),
    timestamp: "2026-06-26T00:00:00Z".to_string(),  // real timestamp: planned fix
};
let okf = frontmatter.render(text.trim());

Java's closest syntax is a constructor call or builder; Rust's struct literal names every field at the call site (like a builder, but built into the language), and the compiler errors if any field is missing — there is no "forgot to set a property" bug class.

The round-trip test: locking the contract

The real safety net is a new test in frontmatter.rs: whatever the writer produces, the parser must read back unchanged. This is the same idea as a JSON serialization round-trip test in Java (assertEquals(dto, mapper.readValue(mapper.writeValueAsString(dto), …))): 

#[test]
fn render_and_parse_round_trip() {
    let fm = OkfFrontmatter {
        doc_type: "technical-note".to_string(),
        title: "Reflector Guide".to_string(),
        // … all six fields …
    };
    let doc = fm.render("Body text.");

    assert_eq!(extract_frontmatter_field(&doc, "title"),
               Some("Reflector Guide".to_string()));
    assert_eq!(extract_frontmatter_field(&doc, "type"),
               Some("technical-note".to_string()));
    assert!(doc.ends_with("---\n\nBody text."));
}

Now the failure mode changes from "silent drift discovered in production" to "red test the moment either side changes." If we ever rename a frontmatter field, this test fails immediately — in both binaries' shared code, before either binary ships.
