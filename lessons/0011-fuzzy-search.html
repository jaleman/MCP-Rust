Lesson 11

Fuzzy Search

~25 minutes Hands-on — edit, build, test in your dev container KUKA MCP Server in Rust

What you're fixing

Right now search_docs does exact substring matching. Type "reflecor" instead of "reflector" and you get nothing. Ask about "reflector height" and it only matches if those two words appear consecutively — "reflector mounting height" would slip through.

This lesson adds two connected improvements:

  1. Multi-term search — the query is split into individual words and the document must contain all of them (anywhere in the text, not just side-by-side).
  2. Fuzzy matching — for each word in the query, if an exact match fails, the server checks whether any word in the document is close enough (within 1–2 typos) to count.
Before: query "reflecor height" → no results (exact substring not found)

After:  query "reflecor height"
          "reflecor" ≈ "reflector" (edit distance 1) ✓
          "height" found exactly ✓
          → Reflector Deployment Guide returned

New concepts — translated from Java

Concept 1: Adding a crate dependency (≈ adding a Maven dependency)

Rust's package registry is crates.io — the equivalent of Maven Central. You declare a dependency in Cargo.toml and Cargo downloads and builds it automatically. Compared to Maven, there is no separate "resolve" step — cargo build handles everything.

Maven (pom.xml)Cargo (Cargo.toml)
<dependency>
  <groupId>org.apache.commons</groupId>
  <artifactId>commons-text</artifactId>
  <version>1.12.0</version>
</dependency>		 strsim = "0.11"

The crate we need is strsim — string similarity metrics. It provides levenshtein(a, b), which counts the minimum number of single-character insertions, deletions, or substitutions needed to turn string a into string b. That count is the edit distance. 

// "reflecor" → "reflector": insert one "t" → distance = 1
strsim::levenshtein("reflecor", "reflector")  // returns 1

// "distanse" → "distance": substitute "e"→"c", "a"→"e" → distance = 2
strsim::levenshtein("distanse", "distance")   // returns 2

// identical strings
strsim::levenshtein("reflector", "reflector") // returns 0

Concept 2: Splitting a string into words — .split_whitespace()

To handle multi-word queries we first need to break the query string into its constituent words. Java's equivalent is query.trim().split("\\s+"); Rust has a dedicated method that does the same thing and handles all whitespace types including tabs and multiple spaces:

JavaRust
String[] terms = query.trim().split("\\s+"); let terms: Vec<&str> = query.split_whitespace().collect();
Arrays.stream(terms).allMatch(t -> matches(doc, t)) terms.iter().all(|t| fuzzy_word_match(&lower, t))

.split_whitespace() returns a lazy iterator over the words. .collect() materialises it into a Vec<&str> — same as Java's toList().

.all(|item| condition) short-circuits to false the moment one element fails the test — just like Java's Stream.allMatch(). A document only proceeds to the excerpt stage if every query word matches.

Concept 3: Checking if any word in a document is close enough — .any()

For each query term we loop over every word in the document and ask: "is this word within the edit-distance threshold?" If any document word passes, the term counts as matched.

JavaRust
Arrays.stream(docWords).anyMatch(w -> distance(w, term) <= threshold) doc_lower.split_whitespace().any(|word| levenshtein(word, term) <= threshold)

.any(|item| condition) is the complement of .all(): it short-circuits to true the moment one element passes — equivalent to Java's Stream.anyMatch().

Choosing the edit-distance threshold

A threshold of 1 allows one typo ("reflecor" → "reflector"). A threshold of 2 allows two. But applying the same threshold to a 3-letter word like "the" or "and" would cause massive false positives — almost everything is within 2 edits of "the". The helper function below skips fuzzy matching entirely for terms of 3 characters or fewer, and uses a threshold of 1 for medium-length words and 2 for long words.

Step 1 — Add the strsim dependency

Open mcp-server/Cargo.toml. Add one line to [dependencies]:

Cargo downloads crates alphabetically by convention — keeping the list sorted makes it easier to scan. The version "0.11" means "any 0.11.x release" — Cargo's semver rules won't auto-upgrade to a breaking 1.0 without you changing this line.

Step 2 — Add the use declaration and helper function

Open mcp-server/src/main.rs. At the very top, with the other use declarations, add: 

use strsim::levenshtein;

Then, anywhere outside the impl block (for example, just above the struct KukaServer line), add this helper function: 

fn fuzzy_word_match(doc_lower: &str, term: &str) -> bool {
    // Fast path: exact substring match anywhere in the document
    if doc_lower.contains(term) {
        return true;
    }
    // Short terms produce too many false positives when fuzzy-matched
    if term.len() <= 3 {
        return false;
    }
    let threshold = if term.len() <= 7 { 1 } else { 2 };
    doc_lower
        .split_whitespace()
        .any(|word| levenshtein(word, term) <= threshold)
}

Reading the helper function

  • fn fuzzy_word_match(doc_lower: &str, term: &str) -> bool
    A free function (not a method on a struct) that takes two string references and returns a boolean. In Java: private static boolean fuzzyWordMatch(String docLower, String term).
  • if ... { return true; }
    An early return — exactly the same as Java's if (...) return true;. Rust lets you write this without a semicolon too (the last expression in a block is the return value), but explicit return is clearer here.
  • let threshold = if term.len() <= 7 { 1 } else { 2 };
    In Rust, if/else is an expression — it produces a value. This is the equivalent of Java's ternary: int threshold = term.length() <= 7 ? 1 : 2;.
  • doc_lower.split_whitespace().any(|word| levenshtein(word, term) <= threshold)
    The last expression in the function with no semicolon — this is its return value. Rust functions implicitly return the last expression if there is no ;.

Step 3 — Update search_docs

Replace the body of search_docs. The changes are: split the query into terms, use .all() with the helper instead of a single lower.find(), and compute the excerpt position from whichever term has an exact match.

Before (the current inner block):

if let Ok(content) = std::fs::read_to_string(&path) {
    let lower = content.to_lowercase();
    if let Some(pos) = lower.find(&query) {
        let title = extract_frontmatter_field(&content, "title").unwrap_or_else(|| {
            path.file_stem().unwrap().to_string_lossy().to_string()
        });
        let resource = extract_frontmatter_field(&content, "resource").unwrap_or_default();

        let start = pos.saturating_sub(200);
        let end = (pos + query.len() + 400).min(content.len());
        let excerpt = content[start..end].trim();

        results.push(format!("• {title}\n  Source: {resource}\n\n  ...{excerpt}..."));
    }
}

After:

if let Ok(content) = std::fs::read_to_string(&path) {
    let lower = content.to_lowercase();

    if terms.iter().all(|term| fuzzy_word_match(&lower, term)) {
        let title = extract_frontmatter_field(&content, "title").unwrap_or_else(|| {
            path.file_stem().unwrap().to_string_lossy().to_string()
        });
        let resource = extract_frontmatter_field(&content, "resource").unwrap_or_default();

        // Anchor the excerpt at the first term that has an exact substring match.
        // Falls back to the document start if all matches were fuzzy-only.
        let pos = terms.iter().find_map(|term| lower.find(*term)).unwrap_or(0);
        let start = pos.saturating_sub(200);
        let end = (pos + 400).min(content.len());
        let excerpt = content[start..end].trim();

        results.push(format!("• {title}\n  Source: {resource}\n\n  ...{excerpt}..."));
    }
}

You also need to move the let terms line to just after the let query line, so that terms is in scope for the inner loop. Update the top of search_docs: 

let knowledge_dir = std::path::Path::new("knowledge");
let query = input.query.to_lowercase();
let terms: Vec<&str> = query.split_whitespace().collect();
let mut results: Vec<String> = Vec::new();

What changed and why

  • if let Some(pos) = lower.find(&query) is replaced by if terms.iter().all(|term| fuzzy_word_match(&lower, term)). The old version required the entire query string to appear verbatim. The new version requires each word to appear, either exactly or within the edit-distance threshold.
  • terms.iter().find_map(|term| lower.find(*term)) finds the position of the first query word that has an exact substring match — used to anchor the excerpt. The *term is needed because .iter() silently adds an extra layer of borrowing as it loops — so term inside the closure is slightly more indirect than you would expect, and lower.find() won't accept it directly. Writing *term peels that layer back. This is a recurring Rust pattern worth understanding; see Appendix 2: Iterators and References for the full explanation. If all matches were fuzzy-only, unwrap_or(0) starts the excerpt at the document beginning.
  • The excerpt window is simplified to a flat +400 lookahead. With multi-term queries the matched region may span a large area anyway, so a fixed window is clearer.

Step 4 — Build and test

cargo build

The first build after adding strsim will download and compile the crate. Expect a few extra seconds; subsequent builds are cached.

Restart the MCP server and try a deliberately misspelled query: 

What is the minimum reflecor mounting height?

You should get a result even though "reflecor" is not a real word. Also try a multi-word query where the words appear separately in the document: 

What is the warning zone distance for AMR robots?

Both "warning" and "zone" appear in the safety document — the search should find and return the relevant passage.

Search can only find what is in the bundle

The query "What are the minimum distances for KMP robots?" fails — not because the search is broken, but because the OKF document (knowledge/safe-minimum-distances.md) describes the categories of distances (protective stop, warning zone) but does not include the actual numeric values from the PDF. Search cannot return information that was never written into the bundle.

Additionally, "kmp" is only 3 characters, so the helper skips fuzzy matching and requires an exact match — but the safety document uses "AMR", not "KMP". The fix for both issues is to enrich the OKF document: add the specific numeric values extracted from the source PDF, and include model names like "KMP 1500" and "KMP 3000" so they become searchable terms.

Performance note

For each query term that fails an exact match, the helper function iterates over every word in the document and calls levenshtein() — an O(m×n) operation per word pair. For KUKA's small knowledge bundle this is instant. If the bundle ever grows to thousands of large documents, the right fix is to pre-index words at load time, not to skip fuzzy matching entirely.

Quick check

Primary source

strsim crate — docs.rs
Documents every metric the crate provides: Levenshtein, Jaro, Jaro-Winkler, and others. normalized_levenshtein() is worth knowing — it returns a 0.0–1.0 similarity score rather than a raw edit count, which makes threshold-setting easier when query words vary widely in length.
