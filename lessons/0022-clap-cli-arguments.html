Lesson 22

clap: Proper CLI Arguments

~20 min Crate: clap 4 (derive) Prerequisite: Lesson 20

Your extract binary currently parses arguments by hand — grabbing args.next() twice and crashing with a generic usage message if anything is missing. That works, but it gives you nothing for free: no --help, no version flag, no useful error when someone passes the wrong number of arguments.

This lesson replaces all of that with clap, Rust's standard CLI-argument library. The payoff is an auto-generated --help page, friendly error messages, and a cleaner foundation for the next lesson — where you'll add a --force-pdftotext flag in about two lines.

Java analogy: picocli

If you've used picocli in Java, clap's derive API will feel immediately familiar. You declare a class annotated with @Command and fields annotated with @Parameters or @Option, then call CommandLine.execute(args). picocli generates the usage page from those annotations.

clap does exactly the same thing — but in Rust, annotations are called attributes, and the "annotation processor" is a derive macro that runs at compile time.

Java (picocli)Rust (clap)
@Command(name="extract")
class Args { … }		 #[derive(Parser)]
#[command(name="extract")]
struct Args { … }
@Parameters(index="0", description="…")
String input;		 /// description here
input: String,
new CommandLine(new Args()).execute(argv) Args::parse()

Feature flags in Cargo.toml

clap ships in a modular form. Its derive macro support — the #[derive(Parser)] ability — is an optional component you must explicitly opt into. You do this with a feature flag in Cargo.toml.

Think of feature flags like Maven optional dependencies: the artifact is published in one bundle, but consumers decide which sub-features they actually compile in. 

# Before — clap without derive (can't use #[derive(Parser)])
clap = "4"

# After — clap with derive enabled
clap = { version = "4", features = ["derive"] }
Pattern to remember
Any time a Rust crate's README says "add the derive feature" (or macros, serde, etc.), this is what it means: change the simple crate = "x" entry to the table form and list the feature in features = […]. You already did this for serde and rmcp in this project.

The three moving parts

Upgrading extract.rs to clap touches three things:

  1. Cargo.toml — add the clap dependency with derive feature
  2. A new Args struct — describe your arguments declaratively
  3. main() — replace the manual match args.next() block with Args::parse()

Step 1 — Add clap to Cargo.toml

Instead of hand-editing Cargo.toml, use cargo add — it's the Cargo equivalent of mvn dependency:add or pasting a coordinate into pom.xml and letting your IDE resolve the version. Run this from mcp-server/: 

cargo add clap --features derive

Cargo resolves the latest compatible version, writes the line into [dependencies], and updates Cargo.lock — the same end state as editing the file by hand: 

[dependencies]
anyhow = "1.0.103"
clap = { version = "4", features = ["derive"] }   # ← written by cargo add
pdf-extract = "0.12.0"
# … rest unchanged
Java analogy
cargo add <crate> --features <flag> is like adding a Maven dependency with a classifier or optional module enabled — Cargo writes the entry and locks the resolved version, the same way your build tool would update pom.xml.

Step 2 — Define the Args struct

At the top of src/extract.rs, after the existing use lines, add this struct. Replace the entire top-of-file use section and the block comment with the following: 

use anyhow::Result;
use clap::Parser;         // ← new import
use std::path::Path;

// CLI argument definition — clap reads these field types and doc comments
// to generate --help output and argument validation automatically.
#[derive(Parser)]
#[command(
    name    = "extract",
    about   = "Extracts KUKA PDFs to OKF markdown files",
    version                      // reads version from Cargo.toml automatically
)]
struct Args {
    /// A single PDF file or a directory of PDFs
    input: String,

    /// The knowledge/ directory to write OKF files into
    output_dir: String,
}

Three things to notice:

  • #[derive(Parser)] — this is the compile-time macro that generates the parsing logic. Nothing else needs to change for clap to know about your fields.
  • #[command(…)] — configures the overall command: its name, description, and whether to expose --version.
  • /// doc comment — that triple-slash comment above each field becomes the help text for that argument. No separate annotation needed.

Step 3 — Replace main()

Find the current fn main() and replace only the manual argument-parsing block at the top. Here is the before and after:

Before

fn main() {
    let mut args = std::env::args().skip(1); // skip the binary name at index 0

    // Require exactly two arguments — input path and output directory
    let (input_arg, output_arg) = match (args.next(), args.next()) {
        (Some(i), Some(o)) => (i, o),
        _ => {
            eprintln!("Usage: extract <pdf-file-or-directory> <output-dir>");
            eprintln!("Example: extract kuka-docs\\ knowledge\\");
            std::process::exit(1);
        }
    };

    let input = Path::new(&input_arg);
    let knowledge_dir = Path::new(&output_arg);
    // … rest of main unchanged

After

fn main() {
    // clap parses, validates, and exits with a friendly error if args are wrong.
    // This single call replaces the entire manual match block.
    let args = Args::parse();

    let input = Path::new(&args.input);
    let knowledge_dir = Path::new(&args.output_dir);
    // … rest of main unchanged

Everything below those lines — the is_dir() check, the batch loop, the process_pdf() function — stays exactly as it was.

Step 4 — Build and try it

Inside your dev container terminal:

# Compile — clap will be downloaded and compiled on first run
cargo build

# Run with no arguments — clap catches the missing args and prints an error
./target/debug/extract

# Run with --help — clap generates this from your struct and doc comments
./target/debug/extract --help

# Run with --version — clap reads this from Cargo.toml [package] version
./target/debug/extract --version

The --help output will look like this:

Extracts KUKA PDFs to OKF markdown files

Usage: extract <INPUT> <OUTPUT_DIR>

Arguments:
  <INPUT>       A single PDF file or a directory of PDFs
  <OUTPUT_DIR>  The knowledge/ directory to write OKF files into

Options:
  -h, --help     Print help
  -V, --version  Print version
What you got for free
--help (generated from doc comments), --version (read from Cargo.toml), a descriptive error when args are missing, and automatic type coercion if you ever change a field from String to u32 or PathBuf. All of this would be dozens of lines of hand-written code otherwise.

Preview: next lesson

In the next lesson you'll add a --force-pdftotext flag so the extractor can fall back to the system pdftotext tool (from poppler-utils) for PDFs that pdf-extract returns empty text for. With clap already wired up, adding that flag is exactly two lines: 

struct Args {
    /// A single PDF file or a directory of PDFs
    input: String,

    /// The knowledge/ directory to write OKF files into
    output_dir: String,

    /// Always use pdftotext instead of pdf-extract (for scanned/PowerPoint PDFs)
    #[arg(long)]
    force_pdftotext: bool,      // ← two lines, that's it
}

This flag becomes --force-pdftotext on the command line automatically. clap maps the snake_case field name to kebab-case flags. You'll implement the actual fallback logic in Lesson 23.
