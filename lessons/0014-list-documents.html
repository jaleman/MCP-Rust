This also introduces HashMap — Rust's equivalent of Java's HashMap<K, V> — and the idiomatic pattern for grouping values under a key.

New concept: HashMap and the group-by pattern

You already know HashSet from Lesson 12. HashMap is its companion: instead of storing unique values it stores key-value pairs, with O(1) lookup by key.

JavaRust
Map<String, List<String>> grouped = new HashMap<>(); let mut grouped: HashMap<String, Vec<String>> = HashMap::new();
grouped.get("safety") → List<String> or null grouped.get("safety") → Option<&Vec<String>>

The most common operation on a HashMap<String, Vec<String>> is "add this value to the list under this key, creating the list if the key doesn't exist yet." In Java you'd write this as: 

// Java
grouped.computeIfAbsent(docType, k -> new ArrayList<>()).add(title);

Rust has a direct equivalent via the .entry() API: 

// Rust
grouped.entry(doc_type).or_insert_with(Vec::new).push(title);

Reading left to right:

  1. .entry(doc_type) — look up (or prepare to insert) this key
  2. .or_insert_with(Vec::new) — if the key is absent, insert a new empty Vec; either way, return a mutable reference to the Vec
  3. .push(title) — append the title to that Vec

One line, no null check, no separate get-then-put. This is the idiomatic Rust group-by pattern and you will see it constantly.

Step 1 — Add HashMap to the use declaration

Open mcp-server/src/main.rs. Update the collections import to include HashMap:

Before:

use std::collections::HashSet;

After:

use std::collections::{HashMap, HashSet};

The curly-brace syntax imports multiple items from the same module in one line — the same as Java's import java.util.{HashMap, HashSet} if Java supported that syntax (it doesn't; Java requires one import per line).

Step 2 — Add the list_docs tool

Inside the #[tool_router] impl KukaServer block, add this method after ping and before search_docs: 

#[tool(description = "List all documents in the KUKA knowledge bundle, grouped by type")]
fn list_docs(&self) -> Result<CallToolResult, McpError> {
    let knowledge_dir = std::path::Path::new("knowledge");
    let mut grouped: HashMap<String, Vec<String>> = HashMap::new();
    let mut total: usize = 0;

    if let Ok(entries) = std::fs::read_dir(knowledge_dir) {
        for entry in entries.flatten() {
            let path = entry.path();
            if path.extension().and_then(|e| e.to_str()) != Some("md") {
                continue;
            }
            if let Ok(content) = std::fs::read_to_string(&path) {
                let title = extract_frontmatter_field(&content, "title")
                    .unwrap_or_else(|| path.file_stem().unwrap().to_string_lossy().to_string());
                let doc_type = extract_frontmatter_field(&content, "type")
                    .unwrap_or_else(|| "uncategorised".to_string());
                grouped.entry(doc_type).or_insert_with(Vec::new).push(title);
                total += 1;
            }
        }
    }

    if grouped.is_empty() {
        return Ok(CallToolResult::success(vec![Content::text(
            "No documents found in the knowledge bundle.".to_string(),
        )]));
    }

    let mut type_keys: Vec<String> = grouped.keys().cloned().collect();
    type_keys.sort();

    let sections: Vec<String> = type_keys.into_iter().map(|doc_type| {
        let mut titles = grouped.remove(&doc_type).unwrap_or_default();
        titles.sort();
        let items: Vec<String> = titles.into_iter().map(|t| format!("  • {t}")).collect();
        format!("{doc_type}:\n{}", items.join("\n"))
    }).collect();

    Ok(CallToolResult::success(vec![Content::text(format!(
        "Knowledge bundle — {total} document(s):\n\n{}",
        sections.join("\n\n")
    ))]))
}

Reading the new code

  • grouped.entry(doc_type).or_insert_with(Vec::new).push(title) — the group-by pattern described above. doc_type is a String, so .entry() takes ownership of it (no &). The key is either used as the new map key or dropped if the key already existed. See Appendix 4: Ownership vs Reference for the general rule on when to pass owned values vs references.
  • grouped.keys().cloned().collect() — .keys() yields references (&String). .cloned() converts each &String into an owned String by cloning it. Without this step, the Vec would borrow from grouped, which would then prevent you from calling grouped.remove() later.
  • grouped.remove(&doc_type) — removes the key from the map and returns the value wrapped in Some(Vec<String>). This avoids cloning the Vec — instead of copying the list out, we move it. .unwrap_or_default() handles the None case (which cannot actually happen here, but the type system doesn't know that).
  • titles.sort() — sorts the title strings alphabetically in place. String implements Ord, so the default sort is lexicographic — same as Java's Collections.sort(list).
Why remove() instead of get()?

grouped.get(&doc_type) would return Option<&Vec<String>> — a reference. A reference cannot be sorted in place or moved into the closure's return value. grouped.remove() returns Option<Vec<String>> — the owned Vec — which can be sorted and consumed freely. Since we only visit each key once, removing it is safe.
