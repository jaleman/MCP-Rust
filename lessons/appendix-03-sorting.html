Java refresh: how List.sort() works

In Java, List.sort(Comparator) takes a comparator — a function that compares two elements and returns an integer. The sign of that integer tells the sort which element should come first:

Return valueMeaning
negative (e.g. -1)first argument goes before the second
zerothe two elements are considered equal
positive (e.g. +1)second argument goes before the first

The classic Java trick for sorting integers in descending order is to subtract in reverse: 

// Ascending (low to high)
list.sort((a, b) -> a.score - b.score);

// Descending (high to low) — swap a and b
list.sort((a, b) -> b.score - a.score);

When b.score is larger than a.score, b - a is positive, which tells the sort that b should go first. That puts the highest score at the front of the list.

Java also provides Comparator.comparingInt() and .reversed() as cleaner alternatives to manual subtraction: 

// Java idiomatic descending sort
list.sort(Comparator.comparingInt((Entry e) -> e.score).reversed());

Rust equivalent: sort_by()

Rust's Vec::sort_by() works on the same principle, but instead of returning an integer it returns an Ordering value — an enum with three variants:

Java return valueRust Ordering variantMeaning
negativeOrdering::Lessfirst argument goes before second
zeroOrdering::Equalelements are equal
positiveOrdering::Greatersecond argument goes before first

You rarely write Ordering::Less by hand. Instead you call .cmp() on a value, which computes and returns the right Ordering automatically: 

// Ascending (low to high)
results.sort_by(|a, b| a.0.cmp(&b.0));

// Descending (high to low) — swap a and b, same logic as Java
results.sort_by(|a, b| b.0.cmp(&a.0));

Comparing b.0.cmp(&a.0) when b.0 > a.0 returns Ordering::Greater — which tells the sort that b should come first. This is the exact same logic as Java's b.score - a.score, just expressed differently.

Why & in .cmp(&a.0)?

.cmp() takes its argument by reference. You pass &a.0 rather than a.0 so that Rust borrows the value for comparison without moving it out of the tuple — moving would consume it and make it unavailable afterwards. In Java this is invisible because all objects are already references.

Worked example — sorting search results by score

Here is a complete before-and-after using the same data in both languages. Three documents have been matched; we want the highest score first.

Java:

import java.util.*;

List<Map.Entry<Integer, String>> results = new ArrayList<>();
results.add(Map.entry(3,  "Fleet Operations Manual"));
results.add(Map.entry(18, "Reflector Deployment Guide"));
results.add(Map.entry(7,  "Safe Minimum Distances"));

// Before sort: [3, 18, 7]
results.sort((a, b) -> b.getKey() - a.getKey());
// After sort:  [18, 7, 3]

for (var entry : results) {
    System.out.println(entry.getKey() + " — " + entry.getValue());
}
// 18 — Reflector Deployment Guide
// 7  — Safe Minimum Distances
// 3  — Fleet Operations Manual

Rust:

let mut results: Vec<(usize, String)> = vec![
    (3,  "Fleet Operations Manual".to_string()),
    (18, "Reflector Deployment Guide".to_string()),
    (7,  "Safe Minimum Distances".to_string()),
];

// Before sort: [(3, ...), (18, ...), (7, ...)]
results.sort_by(|a, b| b.0.cmp(&a.0));
// After sort:  [(18, ...), (7, ...), (3, ...)]

for (score, title) in &results {
    println!("{score} — {title}");
}
// 18 — Reflector Deployment Guide
// 7  — Safe Minimum Distances
// 3  — Fleet Operations Manual

The pattern is identical: swap the arguments to flip from ascending to descending.

The cleaner alternative: sort_by_key()

When you are sorting by a single extracted value — as you are here (just the score) — sort_by_key() is cleaner than sort_by(). You provide a function that extracts the key from each element, and Rust sorts by that key: 

// Java equivalent
results.sort(Comparator.comparingInt(e -> e.score).reversed());

// Rust with sort_by_key — ascending
results.sort_by_key(|(score, _)| *score);

// Rust with sort_by_key — descending, using std::cmp::Reverse
use std::cmp::Reverse;
results.sort_by_key(|(score, _)| Reverse(*score));

Reverse is a wrapper type that flips the sort order of whatever it wraps. Wrapping the score in Reverse(*score) makes higher scores sort first without needing to write b.cmp(&a) manually. The *score dereferences the score (which comes out as &usize from the destructure) back to usize so Reverse can wrap it.

Chaining comparators

Java's Comparator.thenComparing() sorts by a second key when the first is equal. Rust's equivalent is .then() on an Ordering: 

// Java — sort by score descending, then by title ascending
results.sort(Comparator.comparingInt((Entry e) -> e.score)
    .reversed()
    .thenComparing(e -> e.title));

// Rust — same thing
results.sort_by(|a, b| {
    b.0.cmp(&a.0)          // compare scores descending
      .then(a.1.cmp(&b.1)) // if equal, compare titles ascending
});

Ordering::then() returns the first Ordering if it is not Equal, otherwise falls through to the second. This chains comparisons cleanly without nested if statements.
