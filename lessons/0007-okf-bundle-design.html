Lesson 7

OKF Bundle Design

~25 minutes Design + hands-on file authoring KUKA MCP Server in Rust

What you're building

The search_docs tool currently returns a stub. Before writing the Rust code to make it real, you need to decide what it will search over. This lesson designs the knowledge bundle — the directory of structured markdown files your server will query. By the end you will have a knowledge/ directory with three hand-authored OKF files that prove the schema works before any automation is built.

Why hand-author first?

Writing a few files manually before building the extractor forces you to decide what good knowledge looks like. A bad schema discovered after extracting 500 pages is expensive to fix. Three files now is cheap.

The OKF format

An OKF bundle is a directory of markdown files. Each file is one concept — a unit of knowledge about one thing. Every file has a small YAML frontmatter block followed by a markdown body. [OKF v0.1 Spec]

Required field

Only one field is required in every concept:

type: technical-note

type is a free string you define for your domain. Consumers must accept unknown types gracefully — so you can add new types at any time without breaking anything.

Recommended fields

Field Purpose
title Human-readable name — used in search results
description One-sentence summary — shown in previews
resource URI or path to the source document — enables citations
tags YAML list for filtering and grouping
timestamp ISO 8601 date of last change

The body

Plain markdown. The OKF spec recommends structured elements over prose — headings, bullet lists, tables, code blocks. Structured content is easier for both humans and AI agents to parse than paragraphs of flowing text.

The KUKA type vocabulary

Based on the document library, these are the types for this bundle:

Type Documents
operating-manual BA_KMF_1500P-CB, BA_KMP_1500P, BA_KMP_3000P
fleet-manual KUKA_AMR_Fleet_2.14, KUKA_AMR_Fleet_2.15
technical-note All KUKA Technical Note PDFs (MQTT, reflectors, ports, etc.)
safety Safe Minimum Distances, Emergency Fire Alarm
work-instruction KMP Calibration WI
reference MQTT Payload Definitions, Fleet Ports, Reflective Sticker Specs, FocusCenter screenshot
site-document Building map DOCX

The resource field convention

For PDF sources, use a relative path from the project root plus an optional page anchor: 

resource: kuka-docs/KUKA Technical Note -Reflector Deployment- V1.0.1.pdf#page=4

When your server returns a result, the resource field gives Claude the exact citation — which document, which page. This is what makes the answers trustworthy.

Step 1 — Create the knowledge directory

Inside your dev container terminal, create the bundle directory at the project root: 

mkdir -p /workspaces/MCP-Rust/knowledge

Step 2 — Write three hand-authored concepts

Create each file below in /workspaces/MCP-Rust/knowledge/. These are real entries drawn from your actual document library.

File 1: reflector-deployment.md

---
type: technical-note
title: Reflector Deployment Guide
description: Placement rules and minimum quantities for retroreflectors used in KUKA AMR laser navigation.
resource: kuka-docs/KUKA Technical Note -Reflector Deployment- V1.0.1.pdf
tags: [navigation, reflectors, installation, amr]
timestamp: 2026-06-26T00:00:00Z
---

# Overview

Retroreflectors provide fixed landmarks that KUKA AMR robots use for laser-based localisation. Correct placement is critical — too few or poorly positioned reflectors cause navigation failures.

# Key Requirements

- Minimum reflector height from floor: 150 mm
- Reflectors must be visible from robot scan height at all times
- Minimum of 3 reflectors visible from any robot position

# Installation Notes

- Mount on rigid surfaces only — vibrating surfaces cause false detections
- Avoid placement near glass or highly reflective backgrounds
- Document reflector positions in the building map after installation

File 2: safe-minimum-distances.md

---
type: safety
title: AMR Safe Minimum Distances
description: Required safety distances between KUKA AMR robots and people, obstacles, and other machines.
resource: kuka-docs/KUKA.AMR - Safe Minimum Distances v1.1.pdf
tags: [safety, amr, distances, compliance, risk-assessment]
timestamp: 2026-06-26T00:00:00Z
---

# Overview

KUKA AMR robots must maintain defined minimum distances to ensure safe operation around people and infrastructure. These values feed into the site risk assessment.

# Distance Categories

- **Protective stop distance**: distance at which the robot performs a protective stop
- **Warning zone distance**: distance at which the robot slows and emits a warning
- **Collaboration zone**: area where simultaneous human-robot work is permitted under defined conditions

# Important

Always refer to the source document for exact values — distances vary by robot model, speed setting, and payload.

File 3: focuscenter-ui.md

---
type: reference
title: FocusCenter Fleet Management UI
description: Reference for the KUKA FocusCenter fleet management dashboard, showing robot status, job queue, and map view.
resource: kuka-docs/FocusCenter.png
tags: [focuscenter, fleet-management, ui, dashboard, software]
timestamp: 2026-06-26T00:00:00Z
---

# Overview

FocusCenter is KUKA's browser-based fleet management software for coordinating multiple AMR robots across a facility.

# Main Dashboard Elements

- **Map view**: real-time robot positions overlaid on the facility map
- **Job queue**: pending, active, and completed transport orders
- **Robot status panel**: battery level, current task, error state per robot
- **Alert feed**: system warnings and robot fault notifications

# Notes

The source file is a screenshot of the FocusCenter interface. For interactive documentation, refer to the KUKA AMR Fleet manuals.

Step 3 — Verify the bundle looks right

List the files you just created:

ls /workspaces/MCP-Rust/knowledge/

You should see three .md files. Open one and check that the YAML frontmatter parses correctly — the dashes (---) must be on their own lines with no leading spaces.
