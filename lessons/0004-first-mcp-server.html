Lesson 4

Your First MCP Server

~25 minutes Hands-on — run inside your dev container KUKA MCP Server in Rust

What you're building

You have a Cargo project with rmcp in it. Now you'll write the actual Rust code — a skeleton MCP server for your KUKA knowledge base. It will expose one tool called ping that confirms the server is alive. By the end of this lesson:

  • You'll have a src/main.rs that compiles cleanly
  • You'll understand what every line does and why it's there
  • You'll smoke-test it by sending a real MCP message and seeing a real response

This is the foundation. Once this skeleton works, Lesson 5 will replace the stub ping tool with a real tool that queries KUKA PDFs.

If you know Java: the concepts in this lesson translated

async/await — Java's CompletableFuture, with cleaner syntax

In Java, non-blocking code looks like this: 

// Java
CompletableFuture<String> future = fetchDataAsync();
future.thenAccept(result -> System.out.println(result));

Rust's async/await is the same idea but reads like ordinary sequential code: 

// Rust
let result = fetch_data_async().await;
println!("{}", result);

async fn marks a function as non-blocking. .await suspends execution at that point until the result is ready — equivalent to Java's .get() on a future, but without blocking the thread. [Why async? — Async Rust Book]

#[tokio::main] starts the async runtime — equivalent to configuring a Java ForkJoinPool or Spring Boot's embedded server lifecycle. Without it, there is no scheduler to run async code. [Hello Tokio — Tokio Tutorial]

Don't go deep yet

For this lesson, recognise the pattern and move on. You'll see #[tokio::main], async fn, and .await — accept that they wire up the async machinery. We'll go deeper into Tokio when concurrency is actually needed.

struct + impl — Java's class, split in two

Java combines data and behaviour in one class block. Rust separates them: struct holds data fields; a separate impl block holds the methods.

JavaRust
class KukaServer { ... } struct KukaServer { ... } (data only)
Methods inside the class body impl KukaServer { fn new() ... fn ping() ... }
this (implicit) &self (explicit first parameter on every method)
class KukaServer implements ServerHandler impl ServerHandler for KukaServer { ... }

Traits — Java interfaces

A trait is Rust's interface. ServerHandler is a trait defined by rmcp; implementing it on KukaServer is exactly like a Java class implementing an interface. Default method implementations work the same way — rmcp provides defaults for every protocol method except get_info, which you must supply. [Traits — The Book, Ch. 10.2]

Result<T, E> — checked return values instead of exceptions

Java uses exceptions for error handling. Rust has no exceptions — errors are return values wrapped in a Result type:

JavaRust
String ping() throws IOException fn ping() -> Result<String, McpError>
return "ok"; Ok("ok") — success wraps the value
throw new IOException("bad") Err(McpError::new("bad")) — failure wraps the error

The caller must handle both cases — the compiler enforces this, like a checked exception that cannot be silently ignored.

#[derive(Clone)] — Cloneable without writing clone()

#[derive(Clone)] is like writing implements Cloneable and providing a clone() method — but the compiler generates it for you. rmcp requires Clone so it can hand copies of your server to concurrent requests.

Macros (#[tool_router]) — annotations that generate code

Rust macros look like Java annotations (@Override, @JsonProperty) but they are more powerful: they run at compile time and generate new code. #[tool_router] reads your impl block and generates a routing table. #[tool_handler] generates the MCP protocol methods. The closest Java equivalent is Lombok — annotations that generate boilerplate at compile time.

Two new concepts you need first

1. Async/await

Rust functions can be marked async. An async function doesn't run immediately when you call it — it returns a future, a description of work to be done later. [Why async? — Async Rust Book]

To actually run a future, you write .await after it. Think of it like placing a coffee order (calling the async fn) and then waiting at the counter for it (the .await).

But something has to run the counter — that's Tokio, an async runtime. The attribute #[tokio::main] on your main function starts Tokio and lets main itself be async. Without it, await wouldn't work anywhere in your program. [Hello Tokio — Tokio Tutorial]

You don't need to go deep yet

For this lesson, you only need to recognise the pattern. You'll see #[tokio::main], async fn, and .await — just accept that they wire up the async machinery. We'll go deeper into Tokio when we need to run things concurrently.

2. Traits

A trait in Rust is a contract: a list of methods a type must provide. If you implement a trait, you're promising those methods exist. Traits are Rust's version of interfaces in other languages. [Traits — The Book, Ch. 10.2]

rmcp defines a trait called ServerHandler. By implementing it on your struct, you're telling rmcp: "here is my server — it knows how to respond to MCP requests." The only method you must provide is get_info, which returns the server's name and capabilities. Every other protocol method has a default implementation inside rmcp.

How rmcp structures a server

rmcp uses two Rust macros to wire up tools. Here's the pattern:

#[tool_router] on your impl block
   ↳ marks which methods are MCP tools
   ↳ generates a router object that maps tool names → methods

#[tool_handler] on your ServerHandler impl
   ↳ wires the router into the MCP protocol
   ↳ generates list_tools and call_tool responses

You write the tools. The macros handle all the JSON-RPC plumbing.

Step 1 — Add the missing dependencies

Your Cargo.toml has rmcp, but you also need Tokio (the async runtime), anyhow (ergonomic error handling), and two tracing crates for logging. Run these inside your dev container: 

cargo add tokio --features rt-multi-thread,macros
cargo add anyhow
cargo add tracing
cargo add tracing-subscriber --features env-filter

rt-multi-thread is the Tokio scheduler; macros enables #[tokio::main]. Log output goes to stderr so it doesn't interfere with the MCP JSON-RPC messages on stdout — that's why we'll configure tracing to write there.

Step 2 — Write src/main.rs

Open mcp-server/src/main.rs and replace everything in it with the code below. We'll walk through each section right after. 

use anyhow::Result;
use rmcp::{
    ErrorData as McpError, ServerHandler, ServiceExt,
    handler::server::router::tool::ToolRouter,
    model::*,
    tool, tool_handler, tool_router,
    transport::stdio,
};

#[derive(Clone)]
struct KukaServer {
    #[allow(dead_code)]
    tool_router: ToolRouter<KukaServer>,
}

#[tool_router]
impl KukaServer {
    fn new() -> Self {
        Self {
            tool_router: Self::tool_router(),
        }
    }

    #[tool(description = "Ping the KUKA knowledge server to confirm it is running")]
    fn ping(&self) -> Result<CallToolResult, McpError> {
        Ok(CallToolResult::success(vec![Content::text(
            "KUKA Knowledge server is online and ready.",
        )]))
    }
}

#[tool_handler]
impl ServerHandler for KukaServer {
    fn get_info(&self) -> ServerInfo {
        ServerInfo::new(
            ServerCapabilities::builder()
                .enable_tools()
                .build(),
        )
        .with_instructions(
            "KUKA AMR robot knowledge server. \
             Use the ping tool to confirm the server is alive.".to_string(),
        )
    }
}

#[tokio::main]
async fn main() -> Result<()> {
    tracing_subscriber::fmt()
        .with_env_filter(tracing_subscriber::EnvFilter::from_default_env())
        .with_writer(std::io::stderr)
        .with_ansi(false)
        .init();

    tracing::info!("Starting KUKA MCP server");

    let service = KukaServer::new().serve(stdio()).await?;
    service.waiting().await?;
    Ok(())
}

Walking through the code

The struct

KukaServer is your server type. It holds one field: tool_router, which is generated by the #[tool_router] macro — it's a lookup table mapping the string "ping" to the ping method. The #[derive(Clone)] is required by rmcp so it can hand copies of your server to concurrent requests.

The tools block

The #[tool_router] attribute on the impl KukaServer block scans every method marked #[tool(...)] and wires them into the router. fn new() calls Self::tool_router() (a method the macro generates) to build the initial router and store it.

ping takes &self (a reference to the server, not ownership) and returns a Result: either a CallToolResult on success or an McpError on failure. CallToolResult::success wraps a list of content blocks — here, a single text block.

The ServerHandler block

impl ServerHandler for KukaServer implements the trait. #[tool_handler] auto-generates the list_tools and call_tool protocol methods by reading the tool router. You only write get_info — everything else is handled for you.

main

tracing_subscriber::fmt() sets up structured logging to stderr. The key line is KukaServer::new().serve(stdio()).await?: it creates your server, hands it to rmcp's stdio transport (which reads newline-delimited JSON from stdin and writes responses to stdout), and returns a service handle. service.waiting().await? blocks until the client closes the connection.

Step 3 — Build

cargo build

You'll see Tokio, anyhow, and tracing compile for the first time alongside rmcp. Expect a longer wait than usual. A successful build ends with: Finished dev [unoptimized + debuginfo].

Paste the full error (everything after error[E...]) into your chat with Claude. The most common issue at this stage is a typo in an import path — rmcp's module layout changed between versions and the error message will point directly at the problem line.

Step 4 — Smoke-test with real MCP messages

Run the server: 

cargo run

The terminal goes quiet — the server is waiting for input on stdin. This is MCP's stdio transport in action: it speaks JSON-RPC, one message per line.

The initialize handshake

MCP is a stateful protocol. Before any tool call can happen, the client and server must negotiate — agreeing on the protocol version and exchanging capability lists. This is the initialize handshake, and it must happen at the start of every session.

Paste this line and press Enter: 

{"jsonrpc":"2.0","method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"0.1"}},"id":1}

You should see a JSON response that includes "result", the server's capabilities, and your instructions string. That's the handshake complete.

Why you always need the handshake

The server will not respond to tool calls before initialization — send one and it will be ignored or rejected. In production, Claude Desktop sends this message automatically the moment it connects to your server, so you never think about it. When testing manually over raw stdio, you send it yourself. Think of it like a TCP handshake: it always happens, the software just normally does it for you.

List the tools

{"jsonrpc":"2.0","method":"tools/list","params":{},"id":2}

You'll see "ping" listed with its description.

Call the ping tool

Now actually invoke it: 

{"jsonrpc":"2.0","method":"tools/call","params":{"name":"ping","arguments":{}},"id":3}

Expected response: 

{"jsonrpc":"2.0","id":3,"result":{"content":[{"type":"text","text":"KUKA Knowledge server is online and ready."}],"isError":false}}

That "text" value is exactly what Claude receives when it calls your tool. Press Ctrl+C to stop the server when done.

What you just proved

A real MCP client sends exactly these same three messages — initialize, tools/list, tools/call — over stdio. Your server speaks the full protocol correctly, from handshake through to a tool response.
