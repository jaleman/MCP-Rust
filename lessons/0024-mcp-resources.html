Lesson 24

MCP Resources: Expose Docs by URI

~25 min Topic: ServerHandler resources Prerequisite: Lesson 23

Your server already has three tools — ping, list_docs, and search_docs. Tools are great for actions: Claude calls them with parameters and gets back results. But MCP has a second primitive called Resources.

A Resource is a piece of data with a URI. Claude can browse a catalogue of resources and fetch any one by its address — no parameters, no invocation logic. After this lesson, every OKF document in the knowledge bundle will be browseable by Claude at a URI like kuka://docs/reflector-guide.

Tools vs. Resources — the REST analogy

If you think in REST terms, the mapping is immediate:

MCP conceptREST analogyWhat Claude does
Tool POST endpoint Calls it with parameters to trigger work
Resource GET endpoint Browses a list, then fetches one by URI

When Claude receives a question about KUKA docs, it can now browse the resource list to see what documents exist, then read a specific document in full — without you having to write a "read a document" tool at all.

The URI scheme

MCP URIs look like URLs, but you invent the scheme. We'll use:

kuka://docs/stem
↑ custom scheme      ↑ filename without .md

So knowledge/reflector-guide.md maps to kuka://docs/reflector-guide. The stem is exactly what path.file_stem() already returns in the existing code.

What we're adding

Three small changes to src/main.rs:

  1. One line in get_info() to advertise resource support
  2. Two new imports to the use rmcp::{ … } block
  3. Two new methods in the #[tool_handler] impl ServerHandler for KukaServer block

Step 1 — Advertise resource support

Claude won't request a resource list unless the server declares it supports resources. Add .enable_resources() to the capabilities builder inside get_info(): 

fn get_info(&self) -> ServerInfo {
    ServerInfo::new(
        ServerCapabilities::builder()
            .enable_tools()
            .enable_resources()    // ← ADD
            .build(),
    )
    .with_instructions(
        "KUKA AMR robot knowledge server. \
         Use the ping tool to confirm the server is alive."
            .to_string(),
    )
}

Step 2 — Add imports

The resource handler methods use two types not yet in scope. Add them to the existing use rmcp::{ … } block: 

use rmcp::{

    model::*,

RequestContext<RoleServer> is the per-request context object — analogous to HttpServletRequest in Java EE. Every async handler receives one so it can cancel in-flight work or send progress notifications. We won't use it in these handlers, but the trait signature requires it.

Step 3 — Implement list_resources

Add this method inside the #[tool_handler] impl ServerHandler for KukaServer block, alongside get_info: 

async fn list_resources(
    &self,
    _request: Option<PaginatedRequestParams>,
    _context: RequestContext<RoleServer>,
) -> Result<ListResourcesResult, McpError> {
    let knowledge_dir = knowledge_dir();
    let mut resources: Vec<Resource> = Vec::new();

    if let Ok(entries) = std::fs::read_dir(&knowledge_dir) {
        for entry in entries.flatten() {
            let path = entry.path();
            if path.extension().and_then(|e| e.to_str()) != Some("md") {
                continue;
            }
            if let Ok(content) = std::fs::read_to_string(&path) {
                let stem = path.file_stem().unwrap().to_string_lossy().to_string();
                let title = extract_frontmatter_field(&content, "title")
                    .unwrap_or_else(|| stem.clone());
                let description = extract_frontmatter_field(&content, "description");
                let uri = format!("kuka://docs/{stem}");

                let mut raw = RawResource::new(uri, stem)
                    .with_title(title)
                    .with_mime_type("text/markdown".to_string());
                if let Some(desc) = description {
                    raw = raw.with_description(desc);
                }
                resources.push(Annotated::new(raw, None));
            }
        }
    }

    Ok(ListResourcesResult::with_all_items(resources))
}

What's happening here

Resource is a type alias for Annotated<RawResource>. The Annotated wrapper holds optional audience/priority metadata — we pass None because we don't need it. RawResource::new(uri, name) takes the URI and a slug (the file stem), then builder methods add the human-readable title and MIME type.

RawResource fieldWhat it holdsExample
uri The stable address Claude uses to fetch this resource kuka://docs/reflector-guide
name (the slug) Machine-readable identifier reflector-guide
title Human-readable display name (optional) Reflector Deployment Guide
description Short summary shown in the resource list (optional) From the OKF frontmatter

The _request parameter carries a pagination cursor — useful when you have thousands of resources. Our knowledge bundle is small, so we ignore it and return everything in one call.

Step 4 — Implement read_resource

Add this second method to the same impl block:

async fn read_resource(
    &self,
    request: ReadResourceRequestParams,
    _context: RequestContext<RoleServer>,
) -> Result<ReadResourceResult, McpError> {
    let uri = &request.uri;

    // Strip the kuka://docs/ prefix to recover the file stem
    let stem = uri.strip_prefix("kuka://docs/").ok_or_else(|| McpError {
        code: ErrorCode::RESOURCE_NOT_FOUND,
        message: format!("Unknown resource URI: {uri}").into(),
        data: None,
    })?;

    let path = knowledge_dir().join(format!("{stem}.md"));
    let content = std::fs::read_to_string(&path).map_err(|_| McpError {
        code: ErrorCode::RESOURCE_NOT_FOUND,
        message: format!("Resource not found: {uri}").into(),
        data: None,
    })?;

    Ok(ReadResourceResult::new(vec![
        ResourceContents::text(content, uri.clone())
    ]))
}
Note: argument order

ResourceContents::text(text, uri) — text content first, URI second. This is opposite to what you might expect. The compiler will catch it if you swap them (the types are both String, so it won't complain — but the emitted JSON will have the fields transposed, which breaks Claude). Double-check the order.

ErrorCode::RESOURCE_NOT_FOUND is MCP error code −32002 — the standard signal for "I don't have a resource at that URI." It lives in rmcp::model::*, so no extra import is needed.

The McpError { code, message, data } struct literal works here because ErrorData is a plain public struct — no constructor required. format!("…").into() converts the String into the Cow<'static, str> that the message field expects.

Step 5 — Build and test

cargo build

Restart the Claude Desktop MCP server, then try:

  • "What documents are available in the KUKA knowledge base?" — Claude will call resources/list and present the catalogue.
  • "Can you read the full reflector guide?" — Claude will call resources/read with the URI it got from the list.

You don't need to do anything else — the MCP protocol handles the round-trip automatically. Claude receives the list, picks the right URI, and fetches the content, all within a single conversation turn.

If Claude ignores the resource list

Some Claude Desktop builds default to tool-use and only fall back to resources when prompted. Try: "List available resources from the KUKA server" or "Use the KUKA resource list to find…" to trigger the resources/list call explicitly.
