Lesson 18

Integration Tests

~25 minutes Hands-on — edit, build, test in your dev container KUKA MCP Server in Rust

What you're building

Unit tests from Lesson 16 cover the pure helper functions. This lesson tests the MCP tools themselves — search_docs and list_docs — by creating a real temporary directory, writing a known OKF file into it, pointing the server at it via KUKA_KNOWLEDGE_DIR, and verifying the output.

These are integration tests: they exercise the full path through the code including filesystem I/O, just with controlled test data instead of your real knowledge bundle.

New concepts — translated from Java

Concept 1: Filesystem helpers in std::fs

Java (NIO)Rust (std::fs)
Files.createDirectories(path) std::fs::create_dir_all(&path)
Files.writeString(path, content) std::fs::write(&path, content)
FileUtils.deleteDirectory(dir) std::fs::remove_dir_all(&dir)
System.getProperty("java.io.tmpdir") std::env::temp_dir()

create_dir_all is the equivalent of mkdir -p — it creates all intermediate directories and does not fail if the directory already exists. remove_dir_all is rm -rf — it deletes the directory and everything inside it.

Concept 2: std::env::set_var() — global state warning

std::env::set_var("KEY", "value") sets an environment variable for the current process — the equivalent of Java's System.setProperty(). The important caveat: environment variables are global process state. If two tests run in parallel and both call set_var with different values, they will interfere with each other.

In Rust 2024 edition (which this project uses), set_var is also marked unsafe because modifying global state in a multi-threaded context can cause data races. You must wrap every call in an unsafe { } block to acknowledge this explicitly. Running with --test-threads=1 is still required to prevent the races at runtime — the unsafe block is just the compiler requiring you to confirm you are aware of the risk.

The fix is to tell cargo test to run tests one at a time: 

# The -- separator passes flags to the test binary rather than to cargo itself.
# --test-threads=1 prevents parallel execution, avoiding env var interference.
cargo test integration -- --test-threads=1

Concept 3: Debug formatting for assertions

The MCP CallToolResult type is not a plain String, so you cannot compare it directly to a string. The simplest way to inspect it in a test is to format it using the Debug trait — the same {:?} format you use in println!: 

// Convert the entire result struct to a debug string for assertion.
// The string will contain all field names and values, including the text content.
let output = format!("{:?}", result);
assert!(output.contains("Reflector Guide"));

This is pragmatic rather than elegant — it works reliably because the text content is embedded in the debug output. A more precise alternative would be to match on the Content enum variant directly, but that requires knowing the exact rmcp type structure.

Step 1 — Add the integration test module

Open mcp-server/src/main.rs. Add this module after the existing #[cfg(test)] mod tests { ... } unit test block: 

#[cfg(test)]
mod integration_tests {
    use super::*;
    use std::fs;
    use std::path::PathBuf;

    // Creates a temporary directory and writes one known OKF document into it.
    // Returns the path to the directory so tests can point KUKA_KNOWLEDGE_DIR at it.
    fn setup_test_bundle() -> PathBuf {
        let dir = std::env::temp_dir().join("kuka-integration-tests");
        // create_dir_all is like mkdir -p — safe to call if the dir already exists
        fs::create_dir_all(&dir).unwrap();

        // A minimal but valid OKF document containing known searchable text
        let doc = "\
---
type: technical-note
title: Reflector Guide
description: Test document for integration tests.
resource: kuka-docs/test.pdf
tags: [test]
timestamp: 2026-01-01T00:00:00Z
---

Reflectors must be mounted at a height of 150 to 2000 mm above floor level.
The maximum spacing between reflectors is 8 metres.";

        fs::write(dir.join("reflector-guide.md"), doc).unwrap();
        dir
    }

    // Removes the temp directory after a test completes.
    // Note: if a test panics before this is called, the directory is left on disk.
    // The OS will eventually clean it up, or you can delete it manually from /tmp.
    fn teardown(dir: &PathBuf) {
        // Ignore the result — if removal fails we don't want to mask the test failure
        let _ = fs::remove_dir_all(dir);
    }

    #[test]
    fn search_finds_matching_document() {
        let dir = setup_test_bundle();
        // Point the server at the test bundle instead of the real knowledge/ directory.
        // unsafe required in Rust 2024 edition — set_var modifies global process state.
        unsafe { std::env::set_var("KUKA_KNOWLEDGE_DIR", &dir); }

        let server = KukaServer::new();
        let input = SearchInput { query: "reflector height".to_string() };
        let result = server.search_docs(Parameters(input)).unwrap();

        // Format the result as a debug string — the text content is embedded in it
        let output = format!("{:?}", result);
        assert!(output.contains("Reflector Guide"), "Should find the document by title");
        assert!(output.contains("150"), "Should include excerpt containing the height value");

        teardown(&dir);
    }

    #[test]
    fn search_returns_no_results_for_unknown_query() {
        let dir = setup_test_bundle();
        std::env::set_var("KUKA_KNOWLEDGE_DIR", &dir);

        let server = KukaServer::new();
        let input = SearchInput { query: "hydraulic pump".to_string() };
        let result = server.search_docs(Parameters(input)).unwrap();

        let output = format!("{:?}", result);
        assert!(output.contains("No results found"), "Unrelated query should return no results");

        teardown(&dir);
    }

    #[test]
    fn list_docs_shows_document_grouped_by_type() {
        let dir = setup_test_bundle();
        std::env::set_var("KUKA_KNOWLEDGE_DIR", &dir);

        let server = KukaServer::new();
        let result = server.list_docs().unwrap();

        let output = format!("{:?}", result);
        assert!(output.contains("Reflector Guide"), "Should list the document title");
        assert!(output.contains("technical-note"), "Should group by type from frontmatter");

        teardown(&dir);
    }
}

Step 2 — Run the integration tests

Run only the integration tests, single-threaded to prevent env var interference: 

cargo test integration -- --test-threads=1

To run everything together (unit tests run in parallel, integration tests single-threaded after them is not directly expressible in one command — the simplest approach is to run all tests single-threaded when you need both): 

# Run all tests single-threaded
cargo test -- --test-threads=1
What if teardown() is not called?

If a test panics, teardown() is skipped — Rust has no finally block. The temp directory is left at /tmp/kuka-integration-tests. This is harmless (the OS cleans up /tmp on reboot) but means a subsequent test run starts with leftover files. The idiomatic fix is the tempfile crate, which provides a TempDir type that deletes itself automatically via Rust's Drop trait when it goes out of scope — the Rust equivalent of a try-with-resources block. That is a future improvement.
