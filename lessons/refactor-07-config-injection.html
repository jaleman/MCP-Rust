Refactor Appendix 7

Read the Environment Once — Constructor Injection

Refactor step 4 — commit 8b029e8 KUKA MCP Server in Rust

The smell: hidden dependencies

Every tool call used to invoke a free function that consulted the process environment on the spot: 

// bundle.rs — called by list_docs, search_docs, list_resources, read_resource…
pub fn knowledge_dir() -> PathBuf {
    PathBuf::from(std::env::var("KUKA_KNOWLEDGE_DIR")
        .unwrap_or_else(|_| "knowledge".to_string()))
}

That's the Rust spelling of a Java anti-pattern you'll recognise instantly: System.getenv() scattered through business logic, or a static Config.getKnowledgeDir() consulted from everywhere. The dependency is real but invisible — nothing in KukaServer's type says "I need a directory", so you can't tell from a signature what the server depends on, and you can't substitute the value in a test without mutating global process state.

The fix: what Spring calls constructor injection

Rust:

#[derive(Clone)]
struct KukaServer {
    /// Where the knowledge bundle lives. Resolved once at startup.
    knowledge_dir: PathBuf,
    tool_router: ToolRouter<KukaServer>,
}

impl KukaServer {
    fn new(knowledge_dir: PathBuf) -> Self { … }
}

// main() is now the ONLY place that reads the environment:
let knowledge_dir = PathBuf::from(
    std::env::var("KUKA_KNOWLEDGE_DIR").unwrap_or_else(|_| "knowledge".to_string()),
);
let service = KukaServer::new(knowledge_dir).serve(stdio()).await?;

Java equivalent — the same move Spring pushed the whole ecosystem toward:

// Before: hidden static lookup                 // After: declared dependency
class KukaServer {                               class KukaServer {
    void listDocs() {                                private final Path knowledgeDir;
        var dir = Config.knowledgeDir(); // ??       KukaServer(Path knowledgeDir) {
        …                                                this.knowledgeDir = knowledgeDir;
    }                                                }
}                                                }

Rust has no Spring container — and doesn't need one here. "Dependency injection" without a framework is just: the caller passes the dependency to the constructor. main() plays the role of the application context: it resolves configuration at the composition root and wires the object graph (currently one object; the step-5 index will be the second).

The payoff arrives immediately: the server itself is now testable

Before this step, no test ever constructed a KukaServer — the tests called helper functions with explicit dir parameters (the _in(dir, …) naming convention was invented purely to route around the hidden env dependency). Now the real, fully-wired server runs against a temp directory: 

#[test]
fn server_tools_use_injected_knowledge_dir() {
    let temp_dir = bundle_with_one_doc();
    let server = KukaServer::new(temp_dir.path().to_path_buf());

    let list = server.list_docs().unwrap();
    assert!(result_text(&list).contains("Reflector Guide"));

    let search = server
        .search_docs(Parameters(SearchInput { query: "reflector height".to_string() }))
        .unwrap();
    assert!(result_text(&search).contains("Found 1 result(s)"));
}

Note what is not in that test: no environment mutation, no std::env::set_var (which is unsafe in multi-threaded test runs — Rust's test harness runs tests in parallel by default, unlike JUnit's default single-thread-per-class), no working-directory tricks. The dependency goes in through the front door.

Also in this commit: the server finally introduces itself properly

MCP servers send an instructions string during the handshake — it's documentation for the AI agent deciding when to call your tools. Ours said only "Use the ping tool to confirm the server is alive", which undersells a search engine rather badly. Agents read this text the way developers read Javadoc; now it says what the server knows and which tool to reach for: 

"KUKA AMR robot knowledge server, grounded in official KUKA documentation
 (AMR fleet manuals, technical notes, safety and deployment guides).
 Use search_docs to find relevant passages for a query, list_docs to see
 every document grouped by type, and the kuka://docs/{name} resources to
 read full documents. ping confirms the server is alive."

Side-by-side summary

JavaRust
System.getenv() in business logic std::env::var() outside main() — same smell
Spring constructor injection plain fn new(dep: …) -> Self — no framework needed
application context / composition root main() resolves config and wires the graph
private final Path knowledgeDir knowledge_dir: PathBuf (immutable unless &mut)
JUnit runs test classes single-threaded by default cargo runs tests in parallel — env mutation is a footgun
