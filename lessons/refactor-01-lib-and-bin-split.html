Refactor Appendix 1

Splitting a Crate into a Library and Binaries

Refactor step 1 — commit 17967a8 KUKA MCP Server in Rust

Why we did this

Before this step, src/main.rs was ~740 lines holding four different jobs: the MCP server wiring, the entire search engine, the frontmatter parser, and every test. Meanwhile src/extract.rs — a completely separate binary — contained its own private knowledge of the OKF frontmatter format. Nothing stopped the two files from drifting apart, and nothing in the file layout told a reader where "search" ended and "MCP protocol" began.

The fix is the standard Rust crate shape: a library holding all the domain logic, and thin binaries that only wire that logic to the outside world (stdio for the server, the command line for the extractor).

Before: After:
src/ src/
  main.rs    (740 lines, everything)    lib.rs         ← declares the modules
  extract.rs (own OKF format)       frontmatter.rs ← OKF read + write (shared!)
                                 bundle.rs      ← knowledge dir + list_docs
                                 search.rs      ← the search engine + tests
                                 main.rs        ← MCP wiring only (~190 lines)
                                 bin/extract.rs ← CLI only, imports the lib

The Java equivalent: a multi-module Maven build

In Java you would reach for a multi-module project: a core module with the shared logic, and two application modules depending on it.

Java (three pom.xml files, three source trees):

kuka-parent/
  pom.xml                 <modules>core, server-app, extract-app</modules>
  core/
    src/main/java/com/kuka/core/Search.java
    src/main/java/com/kuka/core/Frontmatter.java
  server-app/
    pom.xml               <dependency>core</dependency>
    src/main/java/com/kuka/server/Main.java
  extract-app/
    pom.xml               <dependency>core</dependency>
    src/main/java/com/kuka/extract/Main.java

Rust (one Cargo.toml, one crate — Cargo's conventions do the wiring):

mcp-server/
  Cargo.toml              no [[bin]] config needed at all
  src/
    lib.rs                ← the "core module": pub mod frontmatter; pub mod bundle; pub mod search;
    frontmatter.rs
    bundle.rs
    search.rs
    main.rs               ← auto-discovered binary "mcp-server" (named after the package)
    bin/
      extract.rs          ← auto-discovered binary "extract" (named after the file)

Where Maven needs explicit <dependency> declarations, a Cargo binary in the same crate gets the library automatically. It imports the library by the crate name — with dashes converted to underscores: 

// src/bin/extract.rs — like `import com.kuka.core.Frontmatter;` in Java
use mcp_server::frontmatter::OkfFrontmatter;

// src/main.rs
use mcp_server::bundle::{knowledge_dir, list_docs_in};
use mcp_server::search::search_docs_in;

What changed in Cargo.toml

Before — every binary declared by hand:

[[bin]]
name = "mcp-server"
path = "src/main.rs"

[[bin]]
name = "extract"
path = "src/extract.rs"

After — both blocks deleted. Cargo auto-discovers src/main.rs as the default binary, src/lib.rs as the library, and every file in src/bin/ as a further binary. This is the same "convention over configuration" philosophy Maven applies to src/main/java — put files in the standard place and the build tool needs no telling.

We also moved tempfile from [dependencies] to a new [dev-dependencies] section. It is only used by tests, and dev-dependencies are only compiled for cargo test — exactly Maven's <scope>test</scope>.

Modules and visibility: pub, pub(crate), private

lib.rs is the module root — the closest Java analogy is a package-info plus the module's public API surface in one file: 

pub mod bundle;       // like a public package the outside can use
pub mod frontmatter;
pub mod search;

#[cfg(test)]                    // compiled ONLY for `cargo test`
pub(crate) mod test_util { … }  // visible inside this crate, invisible outside
JavaRust
public class Search pub fn search_docs_in(…)
package-private (no modifier) private (the default) — visible in the same module only
no direct equivalent (JPMS exports comes close) pub(crate) — public inside the crate, hidden outside
test fixtures in src/test/java #[cfg(test)] mod test_util — stripped from release builds
Maven <scope>test</scope> [dev-dependencies]

One visibility detail worth noticing: helper functions like fuzzy_word_match and repeated_lines stayed private in search.rs. Their tests still work because a #[cfg(test)] mod tests block is a child module of the file it lives in — and children can see their parent's private items. In Java terms: the test class effectively lives inside the class under test, so nothing needs to be made public just for testing.
