Lesson 16

Unit Tests

~20 minutes Hands-on — edit, build, test in your dev container KUKA MCP Server in Rust

What you're building

Every time you change the search logic you currently test it by rebuilding, restarting the server, and asking a question. That is slow and easy to forget. This lesson adds automated unit tests for the two pure functions in main.rs — fuzzy_word_match and extract_frontmatter_field — so you can verify their behaviour in seconds with a single command.

New concepts — translated from Java

Concept 1: The test module and #[cfg(test)]

In Java, tests live in a separate directory (src/test/java) and are compiled and run by the build tool. In Rust, tests live inside the same file as the code they test, in a dedicated module marked with #[cfg(test)].

Java (JUnit)Rust
// src/test/java/FuzzyMatchTest.java
class FuzzyMatchTest { ... }		 // bottom of src/main.rs
#[cfg(test)]
mod tests { ... }
@Test
void exactMatch() { ... }		 #[test]
fn exact_match() { ... }
assertEquals(expected, actual) assert_eq!(expected, actual)
assertTrue(condition) assert!(condition)

#[cfg(test)] is a conditional compilation attribute — the entire module is only compiled when you run cargo test. In a normal cargo build the test code does not exist in the binary at all. Java achieves the same separation by putting test code in a different source directory that the production build ignores.

Concept 2: use super::*

The test module is a child of the main file's module. To call fuzzy_word_match from inside mod tests, you need to bring it into scope. use super::* imports everything from the parent module — super means "one level up", and * means "everything". 

#[cfg(test)]
mod tests {
    // Import fuzzy_word_match, extract_frontmatter_field, and everything
    // else defined in the parent module (main.rs) into this test module
    use super::*;

    #[test]
    fn my_test() {
        // can now call fuzzy_word_match() directly
    }
}

The Java equivalent is that test classes in the same package automatically have access to package-private members. use super::* makes the access explicit.

Concept 3: assert! and assert_eq!

Both macros panic (fail the test) when the condition is not met. assert_eq! is preferred when comparing two values because it prints both values in the failure message, making it easier to diagnose what went wrong: 

// assert! — checks a boolean condition
assert!(fuzzy_word_match("reflector guide", "reflector"));

// assert_eq! — checks equality and prints both sides on failure
// left is conventionally "expected", right is "actual"
assert_eq!(
    Some("Reflector Guide".to_string()),  // expected
    extract_frontmatter_field(content, "title")  // actual
);

// Failure message from assert_eq!:
// left:  Some("Reflector Guide")
// right: None   ← you can see exactly what went wrong

Step 1 — Add the test module to main.rs

Open mcp-server/src/main.rs and add the following block at the very bottom of the file, after the closing brace of async fn main(): 

#[cfg(test)]
mod tests {
    // Bring all functions from main.rs into scope so tests can call them directly
    use super::*;

    // --- fuzzy_word_match tests ---

    #[test]
    fn fuzzy_exact_match() {
        // An exact substring match should always return true
        assert!(fuzzy_word_match("reflector deployment guide", "reflector"));
    }

    #[test]
    fn fuzzy_typo_within_threshold() {
        // "reflecor" is 1 edit away from "reflector" (missing 't') — within threshold
        assert!(fuzzy_word_match("reflector deployment guide", "reflecor"));
    }

    #[test]
    fn fuzzy_short_term_requires_exact() {
        // Terms of 3 chars or fewer skip fuzzy matching and need an exact substring match
        assert!(fuzzy_word_match("kuka amr robot", "amr"));
        assert!(!fuzzy_word_match("kuka robot", "amr"));
    }

    #[test]
    fn fuzzy_no_match() {
        // A completely unrelated word should not match
        assert!(!fuzzy_word_match("reflector deployment guide", "hydraulic"));
    }

    // --- extract_frontmatter_field tests ---

    #[test]
    fn frontmatter_field_present() {
        // A valid frontmatter block with the requested field should return Some(value)
        let content = "---\ntitle: Reflector Guide\ntype: technical-note\n---\n\nBody text.";
        assert_eq!(
            extract_frontmatter_field(content, "title"),
            Some("Reflector Guide".to_string())
        );
    }

    #[test]
    fn frontmatter_field_absent() {
        // A frontmatter block that does not contain the field should return None
        let content = "---\ntype: technical-note\n---\n\nBody text.";
        assert_eq!(extract_frontmatter_field(content, "title"), None);
    }

    #[test]
    fn frontmatter_no_frontmatter() {
        // A file with no frontmatter at all should return None
        let content = "Just plain text with no frontmatter.";
        assert_eq!(extract_frontmatter_field(content, "title"), None);
    }
}

Step 2 — Run the tests

cargo test

All 7 tests should pass. To run only a subset, pass a filter string — cargo runs every test whose name contains the string: 

// Run only the frontmatter tests
cargo test frontmatter

// Run only the fuzzy tests
cargo test fuzzy
Why only test the pure functions?

fuzzy_word_match and extract_frontmatter_field are pure functions — they take inputs and return outputs with no side effects and no I/O. Pure functions are trivial to test: no server to start, no files to read, no network involved. The MCP tools (search_docs, list_docs) interact with the filesystem, which requires integration tests. Those are a future lesson.
