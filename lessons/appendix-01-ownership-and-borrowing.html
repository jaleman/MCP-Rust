Appendix A

Ownership & Borrowing

~25 minutes Conceptual — for Java developers KUKA MCP Server in Rust

This appendix exists for one reason: & keeps appearing in the lesson code and the explanation "it's a reference" doesn't fully land until you understand why Rust needs references at all. That requires understanding ownership — the concept Java never needed because it has a garbage collector.

Why Java never made you think about this

In Java, when you pass an object to a method, a reference to it is copied onto the call stack. The object itself stays on the heap. When nobody holds a reference to it anymore, the garbage collector eventually frees it. You never think about when memory is freed — the GC handles it.

Rust has no garbage collector. Something else must decide when memory is freed. That something is ownership — a set of rules the compiler enforces at compile time, with zero runtime cost.

Rule 1: every value has exactly one owner

In Rust, every value on the heap has exactly one variable that owns it. When that variable goes out of scope, the memory is freed immediately — no GC needed. 

fn main() {
    let s = String::from("hello");  // s owns this String
}   // s goes out of scope — String is freed here, automatically

This is equivalent to Java's try-with-resources, but automatic and universal — every value, always, with no syntax required.

Rule 2: assigning or passing a value moves ownership

This is the rule with no Java equivalent. When you assign a heap value to another variable, or pass it to a function, ownership moves. The original variable becomes invalid. 

let s1 = String::from("hello");
let s2 = s1;                 // ownership moves from s1 to s2

println!("{}", s1);          // COMPILE ERROR: s1 was moved
Why this doesn't exist in Java

In Java, String s2 = s1 copies the reference — both variables point to the same object, and the GC tracks both. Rust has no GC, so it enforces that only one variable owns the data at a time. Freeing memory when the owner goes out of scope only works safely if there is exactly one owner.

Rule 3: you can borrow a value with &

Moving ownership into every function would make Rust unusable — you would lose access to a value the moment you passed it anywhere. Borrowing solves this: & creates a reference that lets a function read the value without taking ownership. When the function returns, the borrow ends and the original owner still has the value. 

fn print_length(s: &String) {     // borrows — does not take ownership
    println!("{}", s.len());
}

fn main() {
    let s = String::from("hello");
    print_length(&s);              // lend s to the function
    println!("{}", s);             // s is still valid — we never gave it away
}
JavaRust
Pass an object to a method — both caller and method can use it fn f(s: &String) — caller retains ownership; function reads only
Pass an object — method can modify it (Java has no way to prevent this) fn f(s: String) — ownership moves in; caller loses access

In Java, passing an object always works like Rust's & — you lend it, the caller keeps the reference. The difference is that Rust also has moves, and & is how you explicitly opt out of moving.

Two kinds of borrow

&T — shared (read-only) borrow

&T lets you read the value. You cannot modify it. Many parts of the program can hold a &T at the same time — unlimited simultaneous readers are safe. 

let s = String::from("hello");
let r1 = &s;
let r2 = &s;
println!("{} and {}", r1, r2);  // fine — two readers at once

&mut T — mutable (read-write) borrow

&mut T lets you modify the value. But there can only be one mutable borrow at a time, and no shared borrows while it exists. 

let mut s = String::from("hello");
let r = &mut s;
r.push_str(" world");           // fine — we have exclusive write access
println!("{}", s);              // fine after r is done being used
let mut s = String::from("hello");
let r1 = &s;
let r2 = &mut s;               // COMPILE ERROR: cannot borrow as mutable
                                // because it is also borrowed as immutable
The borrow checker rule, stated simply

At any given point in your code, you may have either:
— any number of read-only borrows (&T), or
— exactly one read-write borrow (&mut T)

Never both at the same time. This is what prevents data races at compile time — no runtime locking needed.

String vs &str — the most common source of confusion

Rust has two string types, and you will see both constantly:

TypeMeaningJava analogy
String Owned, heap-allocated string. Can grow. Has one owner. new StringBuilder(...).toString() — owned, mutable
&str Borrowed slice — a read-only window into a string. Cannot grow. No direct equivalent — closest is a read-only CharSequence view
&String A reference to an owned String. Functions that only need to read usually accept &str instead. Passing a String reference to a method

String literals like "hello" in Rust are &str — they are baked into the binary and borrowed for the entire program lifetime. When you need an owned, modifiable string (e.g., to build one dynamically), you use String::from("hello") or "hello".to_string(). 

let owned: String = String::from("hello");   // heap allocated, owned
let borrowed: &str = "hello";               // points into binary, borrowed
let slice: &str = &owned[0..3];            // points into owned, borrowed

What & means in the lessons

With this foundation, the uses of & you have seen in the lessons make sense:

Code from the lessonsWhat it means
fn ping(&self) Borrow the server — read it without taking ownership. Every method that reads but doesn't consume the struct uses &self.
extract_frontmatter_field(&content, "title") Lend content to the helper function. We keep ownership and can use content again on the next line to extract the excerpt.
&content[start..end] A borrowed slice — a read-only window into content with no allocation. No new String is created; we just point at a range of bytes in the original.
std::fs::write(&output, &okf)? Lend both the path and the content to the write function. Both remain usable after the call.
