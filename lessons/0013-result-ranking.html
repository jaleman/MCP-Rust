Lesson 13

Result Ranking

~20 minutes Hands-on — edit, build, test in your dev container KUKA MCP Server in Rust

What you're fixing

Right now, matching documents are returned in whatever order the filesystem hands them back — essentially random. If two documents both contain your query terms but one is far more relevant, there is no guarantee it appears first. As the knowledge bundle grows, this will increasingly matter.

This lesson adds a simple relevance score: the total number of times the query terms appear in the document. A document that mentions "reflector" twelve times is almost certainly more relevant than one that mentions it once. Results are sorted by this score, highest first, before being returned to the AI client.

New concepts — translated from Java

Concept 1: Tuples — anonymous pairs (and triples, and more)

Java has no built-in tuple type. When you need to pass a pair of values together you typically reach for Map.Entry, a custom record, or a library like Apache Commons Pair. Rust has tuples built into the language:

JavaRust
Map.Entry<Integer, String> (usize, String)
entry.getKey() tuple.0
entry.getValue() tuple.1

Fields are accessed by position — .0, .1, .2 — not by name. You can also destructure a tuple in a closure parameter, which lets you give the fields meaningful names on the spot: 

let pair: (usize, String) = (18, "Reflector Deployment Guide".to_string());

// Field access by position
println!("{}", pair.0);  // 18
println!("{}", pair.1);  // Reflector Deployment Guide

// Destructuring — name the fields as you unpack
let (score, title) = pair;
println!("{score}: {title}");

// Destructuring inside a closure parameter
results.iter().map(|(score, text)| format!("[{score}] {text}"));

// _ discards a field you don't need
results.iter().map(|(_, text)| text);

Concept 2: .map() — transform every element

.map() is the iterator method you reach for when you need to convert each element into something else. Java's equivalent is Stream.map().

JavaRust
stream.map(t -> lower.split(t, -1).length - 1) .map(|term| lower.matches(*term).count())
.mapToInt(...).sum() .sum()

str::matches(pattern) returns an iterator over every non-overlapping occurrence of the pattern in the string. Chaining .count() converts that into a number. So lower.matches(*term).count() is "how many times does this term appear?"

.sum() at the end adds up all the counts from each term into a single usize score. The full expression: 

let score: usize = terms.iter()
    .map(|term| lower.matches(*term).count())
    .sum();

reads as: "for each query term, count how many times it appears, then add the counts together."

Concept 3: sort_by() — sorting with a custom comparator

Java sorts a list with a Comparator: list.sort(Comparator.comparingInt(e -> -e.score)). Rust's equivalent is Vec::sort_by() with a closure:

JavaRust
list.sort((a, b) -> b.score - a.score) results.sort_by(|a, b| b.0.cmp(&a.0))

b.0.cmp(&a.0) compares b's score against a's. See Appendix 3: Sorting for a full Java-to-Rust comparison with worked examples. Because the arguments are in reverse order (b first), higher scores sort to the front. .cmp() returns an Ordering value — Less, Equal, or Greater — which is what sort_by expects. This is Rust's equivalent of a Java comparator returning a negative, zero, or positive integer.

Step 1 — Change the results type

Open mcp-server/src/main.rs. At the top of search_docs, change the results declaration from a Vec<String> to a Vec of tuples:

Before:

let mut results: Vec<String> = Vec::new();

After:

let mut results: Vec<(usize, String)> = Vec::new();

Step 2 — Compute a score and push a tuple

Inside the inner if let Ok(content) block, replace the final results.push(...) call:

Before:

results.push(format!(
    "• {title}\n  Source: {resource}\n\n  ...{excerpt}..."
));

After:

let score: usize = terms.iter()
    .map(|term| lower.matches(*term).count())
    .sum();

results.push((
    score,
    format!("• {title}\n  Source: {resource}\n\n  ...{excerpt}..."),
));

Step 3 — Sort and format

Replace the final let text = ... block:

Before:

let text = if results.is_empty() {
    format!("No results found for '{}'.", input.query)
} else {
    format!(
        "Found {} result(s) for '{}':\n\n{}",
        results.len(),
        input.query,
        results.join("\n\n")
    )
};

After:

results.sort_by(|a, b| b.0.cmp(&a.0));

let text = if results.is_empty() {
    format!("No results found for '{}'.", input.query)
} else {
    let ranked: Vec<String> = results.into_iter().map(|(_, text)| text).collect();
    format!(
        "Found {} result(s) for '{}':\n\n{}",
        ranked.len(),
        input.query,
        ranked.join("\n\n")
    )
};

What changed and why

  • results.sort_by(|a, b| b.0.cmp(&a.0)) — sorts the tuple Vec in-place, highest score first. The .0 accesses the score field of each tuple.
  • results.into_iter() — consumes the Vec, yielding owned tuples rather than references. This is needed here because the next step moves the String out of each tuple, which is not possible with a borrowed reference.
  • .map(|(_, text)| text) — destructures each tuple in the closure parameter, discards the score with _, and returns only the formatted string. The result is a plain Vec<String> that .join() can work with.

Step 4 — Build and test

cargo build

With only a few documents in the bundle the ranking effect is subtle, but you can verify scoring works by searching for a term that appears many times in one document and rarely in another. The document with more occurrences should appear first. 

What are the reflector deployment requirements?

If the Reflector Deployment Guide is in your bundle, it should rank above any other document that happens to mention "reflector" in passing.

Scoring fuzzy matches

The score counts only exact substring occurrences via str::matches(). A document matched by fuzzy logic (e.g. "reflecor" → "reflector") will score zero for that term even though it was included in the results. This is acceptable for now — fuzzy matches are already lower-confidence, so under-scoring them is a reasonable default. A future improvement would count near-matches at a fractional weight.
