OCR Fallback for Image-Based PDFs

Refactor step 8 — making scans searchable without changing search KUKA MCP Server in Rust

The missing document

EmergencyFireAlarm.pdf looked like a normal source document, but it never reached the knowledge bundle. Both text extractors returned an empty string, and extract correctly refused to write a frontmatter-only markdown file. The problem was not search, indexing, or chunking. The PDF pages were images, so there was no text layer to extract.

Step 7 taught the opposite lesson — do not assume missing data needs OCR until you check the pipeline. Here the pipeline check really did point to OCR: raw extraction was empty before cleaning ever ran. That placement matters. OCR is a fallback for missing raw text, not a repair pass for documents that already extracted text.

Why ocrmypdf instead of calling Tesseract directly?

Tesseract can OCR page images, but that would make Rust responsible for rendering PDF pages, naming temporary images, joining text back together, and reconstructing page boundaries. ocrmypdf does the PDF-shaped job: it creates a searchable PDF with the original pages preserved. Then our existing pdftotext -layout path extracts text with form-feed page separators, so page-accurate chunking and pages: provenance keep working unchanged.

normal PDF → pdf-extract / pdftotext → clean → chunk → OKF markdown
image PDF   → empty raw text → ocrmypdf → pdftotext → clean → chunk → OKF markdown

The Rust change

fn try_ocr(pdf_path: &Path) -> Result<String> {
    let temp = tempfile::Builder::new()
        .suffix(".pdf")
        .tempfile()
        .context("cannot create temp file for OCR output")?;

    let output = std::process::Command::new("ocrmypdf")
        .arg("--skip-text")
        .arg(pdf_path)
        .arg(temp.path())
        .output()
        .context("ocrmypdf not found — install ocrmypdf (apt-get install ocrmypdf)")?;

    if !output.status.success() {
        let stderr = String::from_utf8_lossy(&output.stderr);
        bail!("ocrmypdf failed: {stderr}");
    }

    try_pdftotext(temp.path())
}

The temporary file is ordinary Rust RAII: when temp drops, the OCR output PDF is deleted. There is no cleanup branch to remember and no stale file path for later runs to trip over. The function returns text, not a special OCR document type, because after OCR the rest of the extractor should not care how the text layer was created. 

let mut text = if force_pdftotext {
    try_pdftotext(pdf_path)?
} else {
    let extracted = pdf_extract::extract_text(pdf_path)?;
    if extracted.trim().is_empty() {
        println!("  pdf-extract returned empty text — trying pdftotext…");
        try_pdftotext(pdf_path)?
    } else {
        extracted
    }
};

let used_ocr = if text.trim().is_empty() {
    println!("  no text layer — running OCR…");
    text = try_ocr(pdf_path)?;
    true
} else {
    false
};

That one trim().is_empty() check is the design boundary. OCR runs only after the normal path has failed to produce raw text. It runs before clean_extracted_text, so a document is not OCR'd just because the cleaner strips navigation noise or because chunking later decides the body is empty.

OCR-derived body text is also prefixed with the source title before cleaning and chunking. That is not for display; the frontmatter already has a title. It makes filename-visible concepts searchable when the scanned pages never spell them out cleanly. In the real acceptance check, Fire appears in EmergencyFireAlarm.pdf, while the OCR text mostly says Emergency and Alarm.

Java equivalent

// Java sketch: same shell-out idea, more manual lifecycle management.
Path temp = Files.createTempFile("ocr-output", ".pdf");
try {
    Process process = new ProcessBuilder(
            "ocrmypdf", "--skip-text", pdfPath.toString(), temp.toString())
        .redirectErrorStream(false)
        .start();

    int code = process.waitFor();
    if (code != 0) {
        String stderr = new String(process.getErrorStream().readAllBytes(), UTF_8);
        throw new IOException("ocrmypdf failed: " + stderr);
    }

    return runPdfToText(temp);
} finally {
    Files.deleteIfExists(temp);
}

Rust's Command and Java's ProcessBuilder are close cousins: both build a child process command without invoking a shell, both capture stdout/stderr, and both force you to check the exit status. The difference you feel as a Java developer is ownership. NamedTempFile owns the file and deletes it on drop; Java usually uses a try/finally block or deleteOnExit, and the latter can accumulate files in a long-running process.

Keeping OCR auditable

fn extraction_tags(used_ocr: bool) -> &'static str {
    if used_ocr {
        "[extracted, ocr, technical-note]"
    } else {
        "[extracted, technical-note]"
    }
}

OCR text can contain recognition errors. The search engine's fuzzy matching helps with one- or two-character mistakes, but the bundle should still say where the text came from. The ocr tag is small, visible in frontmatter, and travels with both single-file and chunked outputs.
