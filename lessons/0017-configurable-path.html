Lesson 17

Configurable Knowledge Path

~15 minutes Hands-on — edit, build, test in your dev container KUKA MCP Server in Rust

What you're fixing

The knowledge directory path is hardcoded as "knowledge" in two places — list_docs and search_docs. This causes two problems:

  1. Tests can't use a temp directory. Integration tests need to create their own isolated files. With a hardcoded path there is no way to point the server at a different directory without editing the source code.
  2. Production deployments break if the working directory is wrong. The server resolves "knowledge" relative to wherever it was launched from. If that directory isn't the project root, the bundle is invisible.

This lesson extracts the path into a helper function that reads an environment variable — KUKA_KNOWLEDGE_DIR — and falls back to "knowledge" if it isn't set. Every existing deployment continues to work unchanged; new deployments and tests can override it.

New concepts — translated from Java

Concept 1: Reading environment variables

JavaRust
String dir = System.getenv("KUKA_KNOWLEDGE_DIR");
// returns null if not set		 std::env::var("KUKA_KNOWLEDGE_DIR")
// returns Result<String, VarError>

Java returns null when a variable is absent, which requires a null check. Rust returns Result<String, VarError> — either Ok(value) if the variable exists, or Err(VarError::NotPresent) if it doesn't. This forces you to handle the missing case explicitly, the same way Result forces you to handle errors.

Concept 2: PathBuf — the owned path type

You have used std::path::Path throughout the server. Path is a borrowed reference to a path — like &str is to String. PathBuf is the owned, heap-allocated version — like String. You need PathBuf here because the path is constructed at runtime from a String and must be returned from a function.

AnalogyStringsPaths
Owned (heap, can grow) String PathBuf
Borrowed (view, no allocation) &str &Path

Concept 3: unwrap_or_else — fallback on error

.unwrap_or_else(|_| default) returns the Ok value if present, or calls the closure to produce a fallback if the result is Err. The |_| discards the error value — we don't need to inspect why the variable was absent, just that it was.

JavaRust
String dir = System.getenv("KEY");
if (dir == null) dir = "knowledge";		 std::env::var("KEY").unwrap_or_else(|_| "knowledge".to_string())

Step 1 — Add the helper function

Open mcp-server/src/main.rs. Add this function above the struct KukaServer declaration, alongside the other helper functions: 

// Returns the knowledge bundle directory path.
// Reads KUKA_KNOWLEDGE_DIR from the environment if set; falls back to "knowledge".
// Using an env var lets tests and production deployments override the path
// without recompiling.
fn knowledge_dir() -> std::path::PathBuf {
    std::path::PathBuf::from(
        std::env::var("KUKA_KNOWLEDGE_DIR")
            .unwrap_or_else(|_| "knowledge".to_string())
    )
}

Step 2 — Use the helper in list_docs and search_docs

In both list_docs and search_docs, replace the hardcoded path line:

Before (appears twice — once in each tool):

// Hardcoded path — breaks if the server is not launched from the project root
let knowledge_dir = std::path::Path::new("knowledge");

After (same replacement in both tools):

// Read path from environment variable, falling back to "knowledge" if not set
let knowledge_dir = knowledge_dir();

The variable name knowledge_dir shadows the function name — that is fine in Rust. The local variable takes precedence within its scope, and the function can still be called by name anywhere else.

Step 3 — Build and verify

cargo build

The default behaviour is unchanged — if KUKA_KNOWLEDGE_DIR is not set, the server reads from knowledge/ as before. To verify the override works, set the variable before launching: 

# Point at a different directory temporarily
KUKA_KNOWLEDGE_DIR=/tmp/test-bundle cargo run --bin mcp-server

Also run the existing unit tests to confirm nothing regressed: 

cargo test
Why this unblocks integration tests

The next lesson will write integration tests that create a temporary directory, populate it with a known OKF file, set KUKA_KNOWLEDGE_DIR to that directory, and then call search_docs or list_docs to verify the output. Without a configurable path, the test would have no way to point the server at its own controlled data.
