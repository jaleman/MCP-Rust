Refactor Appendix 17

Bounding Search Output — Small Queries, Small Responses

Refactor step 12 KUKA MCP Server in Rust

Why this step exists

A user can reasonably search for a section number such as 2.2.9. The old tokenizer split that into the query terms 2, 2, and 9. Then matching_keys checked whether every vocabulary key contained each term. A single digit appears inside years, page numbers, part numbers, and section numbers across the bundle, so that tiny query expanded into a huge result set.

The second problem was presentation: run_search formatted every hit it received. The search engine could return a ranked list, but the tool response had no upper bound. Together those two choices produced an 80K-character answer from a small, normal-looking query.

Fix 1: exact matching for tiny terms

Substring matching is useful for real words: reflector should find reflectors, and amr should still find the tokenized KUKA.AMR content. The fix is a floor: terms shorter than three characters must match a vocabulary key exactly.

Rust before:

fn matching_keys(&self, term: &str) -> Vec<&str> {
    self.vocab
        .keys()
        .filter(|key| key.contains(term) || within_typo_tolerance(key, term))
        .map(String::as_str)
        .collect()
}

Rust after:

pub(crate) const MIN_SUBSTRING_TERM_LEN: usize = 3;

fn matching_keys(&self, term: &str) -> Vec<&str> {
    self.vocab
        .keys()
        .filter(|key| {
            if term.len() < MIN_SUBSTRING_TERM_LEN {
                key.as_str() == term
            } else {
                key.contains(term) || within_typo_tolerance(key, term)
            }
        })
        .map(String::as_str)
        .collect()
}

Java equivalent:

static final int MIN_SUBSTRING_TERM_LEN = 3;

boolean matches(String key, String term) {
    if (term.length() < MIN_SUBSTRING_TERM_LEN) {
        return key.equals(term);
    }
    return key.contains(term) || withinTypoTolerance(key, term);
}

This is not a special case for numbers. It is a general rule: a one-character or two-character token is too broad to use as a substring against the entire vocabulary. Exact matching keeps it meaningful without letting it explode.

Fix 2: cap the formatted response

Search can still legitimately find many documents. That is fine. The tool response must not serialize all of them into the agent's context. Because hits are already ranked, the presentation layer can show the top N and say exactly what it omitted.

Rust before:

let ranked: Vec<String> = hits.iter().map(format_hit).collect();
format!(
    "Found {} result(s) for '{query}':\n\n{}",
    hits.len(),
    ranked.join("\n\n")
)

Rust after:

const MAX_HITS_SHOWN: usize = 20;

let total = hits.len();
let shown = &hits[..total.min(MAX_HITS_SHOWN)];
let ranked: Vec<String> = shown.iter().map(format_hit).collect();
let mut text = format!(
    "Found {total} result(s) for '{query}'{}:\n\n{}",
    if total > shown.len() {
        format!(", showing top {}", shown.len())
    } else {
        String::new()
    },
    ranked.join("\n\n")
);
if total > shown.len() {
    text.push_str(&format!(
        "\n\n…{} more result(s) omitted. Add more specific terms to narrow the query.",
        total - shown.len()
    ));
}

Java equivalent:

static final int MAX_HITS_SHOWN = 20;

List<Hit> shown = hits.stream()
    .limit(MAX_HITS_SHOWN)
    .toList();

String body = shown.stream()
    .map(this::formatHit)
    .collect(Collectors.joining("\n\n"));

if (hits.size() > shown.size()) {
    body += "\n\n..." + (hits.size() - shown.size())
        + " more result(s) omitted. Add more specific terms to narrow the query.";
}

The Java lesson is the same as the Rust lesson: never trust an unbounded collect() into a response body. If the caller is an AI agent, the response body is context budget, not a log file.

What the tests protect

The index test builds a document containing 2026 and proves that searching for 2 no longer matches it by substring. It also confirms that searching for 2026 still works. The existing amr regression test protects the chosen threshold: three-character terms still substring-match.

The tool test builds more than 20 matching fixture documents and checks the response: exactly 20 bullet entries, the original total, and a clear omitted-count trailer. That is the contract agents need: ranked, bounded, and honest.
