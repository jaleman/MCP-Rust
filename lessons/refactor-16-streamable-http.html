Refactor Appendix 16

Streamable HTTP — One Server, Two Transports

Refactor step 10 KUKA MCP Server in Rust

Why add a second transport?

Until this step, the KUKA server spoke MCP only over stdio: a client starts the binary, writes JSON-RPC to stdin, and reads replies from stdout. That is perfect for local tools like Claude Code, Claude Desktop, and VS Code. It is awkward for browser-based clients, though, because browsers and remote MCP clients naturally connect to URLs. Streamable HTTP adds that URL shape while preserving the existing no-flag stdio behavior.

The important architectural rule is that transport is wiring, not domain logic. Search, resources, reloads, and the index stay exactly where they were. The new branch in main() decides only how requests reach the same KukaServer.

The CLI switch

Clap turns command-line arguments into a small typed struct:

Rust:

#[derive(Debug, Parser)]
struct Args {
    /// Listen address for streamable HTTP, e.g. 127.0.0.1:8382. Omit for stdio.
    #[arg(long)]
    http: Option<String>,
}

Java equivalent:

record Args(Optional<String> http) {}

Args args = parseArgs(argv);          // picocli, Spring Boot, or your own parser
if (args.http().isPresent()) {
    serveHttp(args.http().get());
} else {
    serveStdio();
}

Option<String> is doing the same job as Java's Optional<String>: the flag may be absent, and the absence is a value the compiler forces us to handle. The match in main() is the whole compatibility promise: 

let server = KukaServer::new(knowledge_dir, index);
match args.http {
    None => {
        let service = server.serve(stdio()).await?;
        service.waiting().await?;
    }
    Some(addr) => serve_http(addr, server).await?,
}

Axum as the embedded web server

In Java, this shape might be an embedded Jetty server, a Spring Boot controller, or a servlet filter chain. In Rust, axum is the small HTTP server layer, and rmcp provides a Tower service that already knows the MCP streamable-HTTP protocol: 

let service = StreamableHttpService::new(
    move || Ok(server.clone()),
    Arc::new(LocalSessionManager::default()),
    StreamableHttpServerConfig::default(),
);
let app = Router::new().route_service("/mcp", service);
let listener = tokio::net::TcpListener::bind(socket_addr).await?;
axum::serve(listener, app).await?;

The Java mental model is: StreamableHttpService is not a controller method we wrote; it is a reusable protocol component mounted at a route. Tower services are Rust's composable request/response middleware building blocks. Axum can route to them directly, much like a servlet container can mount a servlet or filter that owns a whole protocol.

One index, many sessions

HTTP introduces sessions: the initialize response returns an Mcp-Session-Id, and later requests send that header back. The session manager is per HTTP listener, but every session gets a clone of the same server handle: 

move || Ok(server.clone())

That clone is cheap because KukaServer holds the index as Arc<RwLock<Index>>, introduced in Appendix 10. In Java terms, each session receives another reference to the same service object, and the service owns a thread-safe shared index. When one session calls reload_docs, the write lock swaps the index and later searches from every session see the new data.

The security boundary is the bind address

This step deliberately does not add authentication. That keeps the learning step small and keeps stdio clients unchanged, but it means HTTP mode should bind to loopback by default: 

mcp-server --http 127.0.0.1:8382

Binding 0.0.0.0 exposes the server to other machines on the network, so the code logs a warning. Public Claude.ai-style deployment is a different project: public HTTPS, OAuth, and operational access controls. For experiments, a tunnel in front of the loopback listener is the interim path.
