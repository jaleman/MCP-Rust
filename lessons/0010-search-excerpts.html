New concepts — translated from Java

Concept 1: Option<T> — Rust's version of Optional<T>

In Java, String.indexOf() returns -1 when the substring isn't found — a sentinel value you have to remember to check. Miss the check and you pass -1 into substring() and get an exception at runtime.

Rust's str::find() returns Option<usize> — the same idea as Java's Optional<Integer>. It is either Some(42) meaning "found at position 42", or None meaning "not found". Rust forces you to unwrap it before you can use the value — you cannot accidentally pass None into a slice operation.

JavaRust
int pos = s.indexOf("reflector");
if (pos != -1) { ... }		 if let Some(pos) = s.find("reflector") { ... }

The if let Some(pos) syntax does two things at once: it checks whether the result is Some, and if so, binds the inner value to pos. Think of it as if (optional.isPresent()) { int pos = optional.get(); } — collapsed into one line.

Concept 2: usize — an integer that can never be negative

In Java, all integer types (int, long) are signed — they can hold negative values. Rust has both signed integers (i32, i64) and unsigned integers (u32, u64, usize).

usize is Rust's type for memory sizes and string positions. It is always non-negative — it literally cannot store a value below zero. This makes it safe for indexing, but it means subtraction can go wrong: 

// Java: fine — int can hold -190
int pos = 10;
int start = pos - 200; // start = -190, Java doesn't complain

// Rust: PANIC in debug mode — usize cannot hold -190
let pos: usize = 10;
let start = pos - 200; // integer overflow

The fix is saturating_sub(n), which clamps the result to 0 instead of panicking. The Java equivalent is Math.max(0, pos - 200): 

// Java equivalent:
int start = Math.max(0, pos - 200);

// Rust:
let start = pos.saturating_sub(200);

Concept 3: .min() — same as Java's Math.min(), but as a method

Java uses a static utility method: Math.min(a, b). Rust puts the same operation directly on the number as a method: a.min(b). They do exactly the same thing. 

// Java:
int end = Math.min(pos + 400, content.length());

// Rust:
let end = (pos + 400).min(content.len());

This caps end at the string length so you cannot accidentally slice past the end of the document — which would panic at runtime, just like Java's StringIndexOutOfBoundsException.

Concept 4: content[start..end] — same as Java's substring(), different syntax

Java: content.substring(start, end)
Rust: &content[start..end]

Both extract the characters from index start up to (but not including) end. The & at the front means "a reference to a slice of this string" — you're reading a window into the original string rather than allocating a new copy. Don't worry about the & deeply yet; think of it as Rust's way of saying "I'm reading, not copying." 

// Java:
String excerpt = content.substring(start, end).strip();

// Rust:
let excerpt = content[start..end].trim();
Why search in lower but slice from content?

lower is a lowercase copy of the document used for case-insensitive matching. find() returns the byte position of the match within lower. We then use that same position to slice from content (the original) so the excerpt preserves the original capitalisation. This works because lowercasing ASCII characters does not change their byte length — "A" and "a" both occupy exactly one byte — so the positions are identical in both strings.

Step 1 — Update the inner loop

Open mcp-server/src/main.rs. Inside search_docs, find the inner block that checks for a match and pushes a result. Replace it:

Before:

if let Ok(content) = std::fs::read_to_string(&path) {
    if content.to_lowercase().contains(&query) {
        let title =
            extract_frontmatter_field(&content, "title").unwrap_or_else(|| {
                path.file_stem().unwrap().to_string_lossy().to_string()
            });

        let resource =
            extract_frontmatter_field(&content, "resource").unwrap_or_default();
        results.push(format!("• {title}\n  Source: {resource}"))
    }
}

After:

if let Ok(content) = std::fs::read_to_string(&path) {
    let lower = content.to_lowercase();
    if let Some(pos) = lower.find(&query) {
        let title =
            extract_frontmatter_field(&content, "title").unwrap_or_else(|| {
                path.file_stem().unwrap().to_string_lossy().to_string()
            });

        let resource =
            extract_frontmatter_field(&content, "resource").unwrap_or_default();

        let start = pos.saturating_sub(200);
        let end = (pos + query.len() + 400).min(content.len());
        let excerpt = content[start..end].trim();

        results.push(format!(
            "• {title}\n  Source: {resource}\n\n  ...{excerpt}..."
        ));
    }
}

What changed and why

  • content.to_lowercase().contains(&query) has been split into two steps: first create lower, then call lower.find(&query). find gives you the position you need; contains does not.
  • The match arm is now if let Some(pos) instead of if .... The logic is the same — "only proceed on a match" — but you also bind pos.
  • Three new lines compute start, end, and excerpt using the patterns you just learned.
  • The format string adds \n\n ...{excerpt}... to each result entry.

Step 2 — Build

cargo build

This should compile cleanly. If you see a borrow error on excerpt, check that you wrote content[start..end].trim() (borrowed from content) and not something that tries to own the string slice.

Step 3 — Test

Restart the MCP server, then ask your AI client a specific factual question whose answer is in your knowledge bundle: 

What height should reflectors be mounted at for KUKA AMR navigation?

You should see the tool return a result that includes the actual specification text, not just the document name. Claude will then compose an answer from that text — grounded in your PDFs.

Tuning the window

The 200 (lookback) and 400 (lookahead) numbers are a starting point. If excerpts are too short to answer a question, increase them. If Claude's context window fills up on long results, decrease them or cap the number of results returned.
