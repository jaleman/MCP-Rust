Refactor Appendix 5

Return Data, Not Prose — the SearchHit API

Refactor step 3 — commit 793fa54 KUKA MCP Server in Rust

The smell: a search engine that writes sentences

Until this step, search_docs_in did everything — matching, scoring, excerpting — and then, in the same breath, composed the user-facing English: "Found 3 result(s) for '…'", bullet points, Source: lines. The function's return type was an MCP CallToolResult, so the search engine couldn't even be called without dragging the protocol library along.

In Java terms: a service class returning pre-rendered HTML instead of DTOs. The moment you want a second consumer (a different output format, the upcoming inverted index, a benchmark harness), you're stuck parsing your own prose. And our tests were doing exactly that: 

// The old tests — substring-fishing in Debug output
let result = search_docs_in(temp_dir.path(), "reflector height".to_string()).unwrap();
let output = format!("{:?}", result);
assert!(output.contains("Reflector Guide"));   // matches the title… or ANY
assert!(output.contains("150"));               // occurrence of "150" anywhere!

The fix: layered like a Spring app

The engine now returns plain data, and the binary owns every user-facing word — the same separation as a Spring service returning DTOs while the controller renders the response:

Rust — search.rs (the "service"):

/// One matching document, ready for whatever presentation layer wants it.
#[derive(Debug)]
pub struct SearchHit {
    pub title: String,
    pub resource: String,
    pub score: usize,
    pub excerpts: Vec<String>,
}

pub fn parse_query(query_lower: &str) -> Vec<&str> { … }   // stop-word filter

pub fn search(docs: &[Document], terms: &[&str]) -> Vec<SearchHit> {
    let mut hits: Vec<SearchHit> = docs
        .iter()
        .filter_map(|doc| match_document(doc, terms))   // Some(hit) kept, None dropped
        .collect();
    hits.sort_by_key(|hit| std::cmp::Reverse(hit.score));
    hits
}

Java equivalent:

public record SearchHit(String title, String resource, int score, List<String> excerpts) {}

public List<SearchHit> search(List<Document> docs, List<String> terms) {
    return docs.stream()
        .map(doc -> matchDocument(doc, terms))     // returns Optional<SearchHit>
        .flatMap(Optional::stream)                 // Rust's filter_map in two steps
        .sorted(Comparator.comparingInt(SearchHit::score).reversed())
        .toList();
}

Note filter_map: it maps each document to Option<SearchHit> and keeps only the Somes — Java needs map + flatMap(Optional::stream) to say the same thing (see Appendix 5). And Reverse(hit.score) is Rust's idiom for a descending sort key — the equivalent of Comparator.reversed().

The presentation layer moved to main.rs (the "controller"):

fn run_search(dir: &Path, query: &str) -> CallToolResult {
    let query_lower = query.to_lowercase();
    let terms = parse_query(&query_lower);
    if terms.is_empty() { /* "only common words" message */ }

    let docs = match load_bundle(dir) { /* error → CallToolResult::error */ };

    let hits = search(&docs, &terms);
    // …format hits into the exact same text output as before…
}

One design decision worth noticing: search() takes terms, not the raw query. The caller runs parse_query itself so it can detect the all-stop-words case and answer with guidance — the engine never needs to know that message exists. Splitting "parse" from "execute" is what lets each layer stay ignorant of the other's concerns.

What the tests look like now

let hits = run_search(temp_dir.path(), "reflector height");

assert_eq!(hits.len(), 1);
assert_eq!(hits[0].title, "Reflector Guide");     // the FIELD, not a substring
assert_eq!(hits[0].resource, "kuka-docs/test.pdf");
assert!(hits[0].score > 0);
assert!(hits[0].excerpts.iter().any(|e| e.contains("150")));

Three things became testable that simply weren't before: ranking order (a new test writes two documents and asserts hits[0] is the higher-scoring one), score semantics (fuzzy-only matches now provably return score == 0 with a body-start fallback excerpt), and which field contains a value (the old contains("150") would have passed if "150" appeared anywhere in the Debug dump, including the score).

The engine no longer knows MCP exists

search.rs now imports crate::bundle::Document, the standard library, and strsim — that's all. The rmcp imports are gone. This is the property that makes step 5 cheap: the inverted index will change how search() finds hits internally, and nothing above the Vec<SearchHit> boundary will notice.
