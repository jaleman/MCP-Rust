Appendix 2

Iterators and References

Reference — return to this when .iter() behaves unexpectedly KUKA MCP Server in Rust

The problem this explains

In Lesson 11 you encountered this line: 

terms.iter().find_map(|term| lower.find(*term))

The *term looks odd. You might expect to write lower.find(term) — after all, terms is a Vec<&str> and term should be &str. But without the * the compiler complains. This appendix explains why.

What .iter() actually gives you

In Java, when you iterate over a List<String>, you get each String directly: 

// Java
List<String> terms = List.of("reflector", "height");
for (String term : terms) {
    // term is a String
}

Rust's .iter() works differently. It does not give you the elements — it gives you references to the elements. This is because .iter() borrows the Vec without taking ownership of it, so it can only hand out references, not the values themselves. 

let terms: Vec<&str> = vec!["reflector", "height"];

for term in terms.iter() {
    // term is &&str — a reference to each &str element
}

The element type is &str. Iterating with .iter() adds one more &, giving you &&str. You never write &&str yourself — Rust creates it silently behind the scenes.

Vec<&str> contains: &str &str &str
.iter() yields: &&str &&str &&str
                             ↑ extra & added by .iter()

Why does this cause a problem?

str::find() expects a pattern — specifically something that implements the Pattern trait. &str implements Pattern. &&str does not. So: 

lower.find(term)   // term is &&str — compiler error: &&str is not a Pattern
lower.find(*term)  // *term is &str  — works fine

The * operator — peeling off one layer of reference

The * operator is called the dereference operator. It removes one layer of &: 

let s: &str = "reflector";
let r: &&str = &s;   // an extra reference added by .iter()

*r   // gives back &str — one layer removed

In Java, references are invisible — you never think about whether you have a reference to a reference. In Rust, every layer of indirection is explicit, and sometimes you need to peel one off manually.

Why doesn't .all() have this problem?

You may have noticed that .all() in the same lesson does not need *term: 

terms.iter().all(|term| fuzzy_word_match(&lower, term))

Here term is also &&str, but fuzzy_word_match accepts it without a *. The reason is that function arguments in Rust apply automatic coercions — &&str is automatically coerced to &str when passed to a function that expects &str. Think of it as Rust silently inserting the * for you at the call site.

lower.find(term) does not benefit from this coercion because find uses a generic Pattern trait bound rather than a concrete &str parameter — and trait matching is stricter about types than function argument coercion. This is an edge case in Rust's type system; the practical rule is:

Practical rule

When iterating over a Vec<&str> with .iter(), each element inside the closure is &&str. Passing it to a plain function parameter usually works without thinking about it. Passing it to something that uses a trait bound (like find()) may require *term to shed the extra &. If the compiler complains that a type doesn't implement a trait, try adding *.

.iter() vs .into_iter()

There are two common ways to iterate a Vec in Rust, and they differ in ownership:

MethodWhat each element isVec after the loop
.iter() A reference to the element (&T) Still usable — you only borrowed it
.into_iter() The element itself (T) Consumed — cannot be used again 
let terms = vec!["reflector", "height"];

// .iter() — borrows, Vec is still usable after
for term in terms.iter() {
    println!("{}", *term);  // term is &&str, need * or auto-deref in println
}
println!("{:?}", terms);  // fine — terms still exists

// .into_iter() — consumes, each element is &str directly (no extra &)
for term in terms.into_iter() {
    println!("{}", term);  // term is &str, no * needed
}
// println!("{:?}", terms);  // ERROR — terms was moved into the iterator

In the search code, .iter() is the right choice because you need terms to remain valid for the rest of the function. Using .into_iter() would consume it on first use.
