Lesson 6

Connecting to an AI Client

~20 minutes Configuration — dev container + AI clients KUKA MCP Server in Rust

What you're building

So far you have tested your server by typing raw JSON-RPC into a terminal. This lesson connects it to a real AI client — either GitHub Copilot or Claude Code — so you can ask natural-language questions and watch the tool be called for real.

The key concept: where does the client run?

The right configuration depends on where the AI client executes, because that determines whether it can reach the binary directly or needs to cross a container boundary.

Windows host
   → VS Code UI (renders the interface only)
   → Claude Code CLI ← runs here, on Windows
      ↕ remote extension protocol
   → Dev container (Linux)
      → VS Code Server
      → GitHub Copilot extension ← runs here, inside container
      → your MCP binary ← lives here

Copilot is co-located with the binary — it can call it directly. Claude Code is on Windows — it must cross the container boundary via docker exec.

You never run the server yourself

With either client, there is no cargo run step. The AI client launches the binary as a subprocess on demand, piping stdin and stdout as the MCP transport. When the conversation ends, the process exits. Your two responsibilities:

  1. Binary compiled — run cargo build after any code change
  2. Dev container running — always true while developing in VS Code

Option A — GitHub Copilot

Copilot extensions run inside the dev container. The binary is at /workspaces/MCP-Rust/mcp-server/target/debug/mcp-server — Copilot can call it directly.

Configuration: .vscode/mcp.json

This file is already in place at the workspace root:

{
  "servers": {
    "kuka": {
      "command": "/workspaces/MCP-Rust/mcp-server/target/debug/mcp-server"
    }
  }
}
VS Code key vs Claude Desktop key

VS Code uses "servers" as the root key. Claude Desktop uses "mcpServers". They are different formats — mixing them up silently breaks the connection.

Steps

  1. Reload VS Code: Ctrl+Shift+P → Developer: Reload Window
  2. Approve the server when prompted, or check with MCP: List Servers
  3. Open Copilot Chat, switch to Agent mode
  4. Ask: Please ping the KUKA knowledge server.

Option B — Claude Code

Claude Code runs on the Windows host. It cannot execute a Linux binary directly, so it uses docker exec to reach inside the named container.

The right way to register the server is with the claude mcp add command. This writes the server to Claude Code's user-level registry, making it available in every session automatically — no folder setup required each time.

Why not .mcp.json?

.mcp.json is project-scoped and only loads when its containing folder is part of the active session. In the Claude Code desktop app, you would need to add the folder to every new session manually. claude mcp add registers the server globally and removes that friction entirely.

Step 1 — Confirm the container name

The devcontainer.json includes "runArgs": ["--name=kuka-mcp-server"]. Verify the container is running with that name: 

docker ps --format "{{.Names}}"

You should see kuka-mcp-server in the list.

Step 2 — Register the server

Run this once in PowerShell:

claude mcp add kuka -- docker exec -i kuka-mcp-server /workspaces/MCP-Rust/mcp-server/target/debug/mcp-server

Verify it was registered:

claude mcp list

Step 3 — Use it

Start any new session in the Claude Code app. The kuka server will be available without adding any folder. Ask: 

Please ping the KUKA knowledge server.

How to verify a tool was actually called

AI clients can read your source code and simulate a tool response without ever invoking the binary. This looks convincing but is not a real MCP call. Always look for an explicit tool-use step: 

Ran ping — kuka (MCP Server)
Input:  {}
Output: KUKA Knowledge server is online and ready.

If you see that block, your Rust binary ran. If you only see prose with no tool-use step, the client read the code and guessed. The tool-use block is the only trustworthy confirmation.

What you just proved

The AI client launched your Rust binary as a subprocess, sent an MCP message over stdin, and received the response your code returned over stdout. The full pipeline — agent → MCP protocol → Rust binary → response — is working end to end.
