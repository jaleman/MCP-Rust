What is MCP?

~15 minutes Conceptual — no code yet KUKA MCP Server in Rust

The problem MCP solves

When you ask Claude a question, it can only answer from what it learned during training. It has no way to open a file on your computer, query a database, or read a PDF you wrote yesterday. Its knowledge is frozen at training time.

This is exactly the problem for your KUKA robot project. You have authoritative documentation in PDF form. Claude doesn't. So we need a way to give Claude live access to those documents at query time, not bake the information into a model.

MCP — the Model Context Protocol — is Anthropic's open standard for solving this. It defines a structured way for an AI agent to talk to an external server that knows things the AI doesn't. [MCP Docs]

The architecture: client and server

MCP uses a classic client–server model:

┌─────────────────────┐ ┌──────────────────────────────┐ │ MCP Client │ │ MCP Server (your Rust app) │ │ │ │ │ │ Claude / Codex / │◄───────►│ Reads KUKA PDFs │ │ any AI agent │ stdio │ Answers queries │ │ │ │ Exposes Tools & Resources │ └─────────────────────┘ └──────────────────────────────┘
  • Client: the AI agent (Claude Desktop, Codex, etc.). It sends requests.
  • Server: your Rust program. It responds with data from the KUKA PDFs.
  • Transport: how they talk. For local use, this is stdio — the server runs as a subprocess, reading from stdin and writing to stdout.
Key insight
You are building the server. Claude is the client. Your job is to give Claude a well-defined interface into the KUKA documentation so it can answer questions accurately, citing the PDFs rather than guessing.

The three primitives

MCP servers expose three types of things to clients. Think of these as the vocabulary of the protocol. [MCP Build Guide]

1. Resources — data the client can read

Resources are like files or records that the AI can fetch. They have a URI and content. In your case, each KUKA PDF (or sections of it) could be a Resource: kuka://docs/KMR_iiwa_manual.

2. Tools — functions the AI can call

Tools are the most powerful primitive. They are functions your server exposes that the AI can invoke with arguments and get results back. Think of them like API endpoints. Your server will have a tool like search_kuka_docs(query: String) — Claude calls it, your Rust code searches the PDFs, and returns matching passages.

3. Prompts — reusable prompt templates

Prompts let the server expose structured prompt templates the AI can use. Useful but optional for our project — we'll focus on Tools and Resources first.

For your KUKA server
Resources: the PDF documents themselves (metadata, full text sections)
Tools: search_kuka_docs, get_robot_specs, list_documents
Prompts: optional — skip for now

The transport: stdio

MCP supports multiple transports, but for a local server the simplest is stdio:

  • Claude starts your Rust binary as a child process
  • It sends JSON-RPC messages to your server's stdin
  • Your server replies to stdout
  • Logs go to stderr (never to stdout — that would corrupt the messages)

This is important: never use println!() for debugging in your MCP server. Always use eprintln!() or a logging library that writes to stderr. [Weather Server Example]

The protocol under the hood: JSON-RPC 2.0

MCP is built on JSON-RPC 2.0 — a simple, language-agnostic protocol for remote procedure calls over JSON. You don't need to implement this yourself; the rmcp crate (the official Rust MCP SDK) handles it for you. [rmcp docs]

What a raw message looks like (so you can recognise it in logs): 

// Claude asks: "call the search_kuka_docs tool"
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "search_kuka_docs",
    "arguments": { "query": "battery charging procedure KMR 200" }
  }
}
// Your server replies with the result
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [{ "type": "text", "text": "Section 4.3: Connect charger to..." }]
  }
}
