Refactor Appendix 10

Arc, RwLock, and Live Reload

Refactor step 5b — commit 2072866 KUKA MCP Server in Rust

The problem: shared state that occasionally changes

The index is built once at startup and read by every tool call. But documents get re-extracted — so we added a reload_docs tool that rebuilds the index while the server is running. That means one value, shared by many readers, replaced occasionally by a writer. In Java you would reach for ReentrantReadWriteLock, or an AtomicReference swap. Rust's spelling: 

#[derive(Clone)]
struct KukaServer {
    knowledge_dir: PathBuf,
    index: Arc<RwLock<Index>>,   // ← the interesting type
    tool_router: ToolRouter<KukaServer>,
}

Read it inside out — each layer answers one question:

LayerQuestion it answersJava equivalent
Index the value the object
RwLock<Index> who may touch it right now? (many readers OR one writer) ReentrantReadWriteLock guarding a field
Arc<RwLock<Index>> who keeps it alive? (all clones, reference-counted) every object reference is implicitly this; the GC does the counting

Java gives you Arc's behavior invisibly (shared references + GC) and makes the locking discipline your responsibility — forget the lock and you get a data race at 3 a.m. Rust inverts it: sharing must be spelled out, and the compiler refuses to compile access to the Index that doesn't go through the lock. The class of bug where someone reads a field while another thread writes it does not exist here.

Why the lock is needed at all: Clone + &self

Two constraints force the design. The MCP framework clones the server (#[derive(Clone)]) — so all clones must share one index, which is exactly what Arc provides (cloning an Arc copies a pointer and bumps a counter, not the index). And tool methods take &self — an immutable reference — yet reload_docs must replace the index. Mutating through a shared reference is called interior mutability, and RwLock is the thread-safe way to get it. This is the same pattern as Appendix 6's LazyLock, one step further: LazyLock is "write once, then read"; RwLock is "read mostly, write occasionally." 

// Readers: shared lock — many tool calls can search concurrently
fn search_docs(&self, …) -> … {
    Ok(run_search(&self.index.read().unwrap(), &input.query))
}

// Writer: exclusive lock — the swap is atomic from any reader's view
fn reload_docs(&self) -> … {
    match Index::build(&self.knowledge_dir) {
        Ok(new_index) => {
            *self.index.write().unwrap() = new_index;   // old index dropped here
            …
        }
        // Rebuild failed (directory vanished?): report it and KEEP the old
        // index — the server stays functional on stale data, loudly.
        Err(e) => Ok(CallToolResult::error(…)),
    }
}

Note the failure policy: build the new index before taking the write lock, and if the build fails, the old index survives. A reload can never leave the server worse off than before it. The test deletes the bundle directory out from under a live server and asserts both halves: the reload errors, and the old index still answers queries.

What's with the .unwrap() on locks?

read() returns Result because of lock poisoning: if a thread panics while holding the lock, the data might be half-updated, so Rust marks the lock poisoned and makes every later access return Err. Our writer replaces the whole value in one assignment — there's no half-updated state to observe — so .unwrap() (crash loudly on poison) is the honest choice, and the comment in the code says so. Java has no equivalent concept: a thread that dies holding a ReentrantReadWriteLock... simply never unlocks it, and the next reader deadlocks silently. Poisoning is Rust being noisy where Java hangs.

Fail fast at the composition root

One behavior changed shape deliberately. A missing knowledge directory used to be reported per tool call; now the index build happens in main() before the server attaches to stdio — bad configuration stops the process with a clear error instead of starting a server that can't answer anything: 

let index = Index::build(&knowledge_dir)
    .with_context(|| format!("failed to build knowledge index from {}", knowledge_dir.display()))?;

tracing::info!(
    "Starting KUKA MCP server: indexed {} document(s), {} unique term(s) in {:.1?} …",
    index.doc_count(), index.term_count(), started.elapsed(),
);

The startup log doubles as the performance receipt: 12 documents, 1,051 unique terms, 23 ms. If that number ever creeps toward seconds, the plan's §5c escape hatch (persist the index, or swap internals for tantivy) is the designed next move — and thanks to the Vec<SearchHit> API boundary from Appendix 5, nothing above the index would notice.
