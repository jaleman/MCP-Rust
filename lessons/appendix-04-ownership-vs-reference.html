Appendix 4

Ownership vs Reference

Reference — return to this when the compiler says "expected String, found &String" KUKA MCP Server in Rust

The question

When calling a function or method, how do you know whether to pass an owned value (my_string) or a reference (&my_string)?

The answer comes from two places: the function's signature tells you what it expects, and the intent of the function tells you why. Once you understand the intent, you rarely need to check the signature.

The signature is the contract

Every Rust function declares exactly what it expects in its parameter types. If the parameter is K it wants ownership. If it is &K it wants a reference. The compiler enforces this — you cannot accidentally pass the wrong thing. 

// Takes ownership — the function may store or drop the value
pub fn entry(&mut self, key: K) -> Entry<K, V>

// Takes a reference — the function only reads, never stores
pub fn get(&self, key: &K) -> Option<&V>

// Takes a mutable reference — the function modifies in place but gives it back
pub fn sort_by<F>(&mut self, compare: F)

The rule of thumb

What the function does with the value Parameter type You pass
Stores it permanently (e.g. as a map key) K — owned my_string
Reads or looks up, then forgets it &K — shared reference &my_string
Modifies in place, then returns control &mut K — mutable reference &mut my_string

Worked examples from this project

HashMap::entry() — takes ownership

grouped.entry(doc_type).or_insert_with(Vec::new).push(title);

entry() takes doc_type: String by value (ownership) because it might need to store the key inside the map. If the key is absent, the HashMap keeps it. If it is already present, the HashMap drops it. Either way, the function needs to own it to make that decision.

HashMap::get() — takes a reference

grouped.get(&doc_type)

get() only looks up — it never stores anything. A reference is enough. The map does not need to own the key to check whether it exists.

HashMap::remove() — takes a reference

grouped.remove(&doc_type)

remove() also takes a reference, even though it deletes the entry. The key you pass is just used to find the entry — the map then removes its own copy of the key. You keep your doc_type after the call.

fuzzy_word_match() — takes references

fn fuzzy_word_match(doc_lower: &str, term: &str) -> bool

Both parameters are &str because the function only reads the strings and returns a bool. It has no reason to own or store them.

What the compiler tells you

If you pass the wrong thing, the compiler error is usually clear: 

// You wrote:
grouped.entry(&doc_type)   // passed a reference

// Compiler says:
// expected `String`, found `&String`
// help: consider removing the `&`

Early in your Rust career, read these errors as the API teaching you its ownership requirement. Over time the patterns become familiar and you stop needing to check.

Java comparison

In Java, every object is already a reference — you never choose between passing by value and passing by reference for objects. Primitives (int, boolean) are passed by value; objects (String, List) are always passed as references. You never have to think about it.

Rust makes this distinction explicit for every type, including strings and collections. The upside is that you always know from the signature whether a function can store, modify, or only read what you give it — which is information Java's signatures do not convey.

When in doubt, try & first

References are the safer default — a function that takes &String cannot store or destroy your value. If the compiler tells you it needs ownership, remove the &. Starting with a reference and being corrected is lower-risk than accidentally giving ownership to a function you didn't intend to.
