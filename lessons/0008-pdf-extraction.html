Lesson 8

PDF Extraction

~25 minutes Hands-on — run inside your dev container KUKA MCP Server in Rust

What you're building

A second Rust binary — extract — that reads a KUKA PDF and writes an OKF markdown file into knowledge/. By the end you will have proved the full extraction pipeline: PDF on disk → Rust code → OKF file ready for search.

This lesson uses one PDF to keep scope tight. Lesson 9 will extend it to batch-process the whole library.

New concept: multiple binaries

So far your Cargo project has one binary: mcp-server, compiled from src/main.rs. Cargo supports multiple binaries in the same project — each one is a separate [[bin]] entry in Cargo.toml pointing to its own source file.

mcp-server/src/main.rs   →   cargo run --bin mcp-server
mcp-server/src/extract.rs →   cargo run --bin extract

They share the same dependencies but are compiled into separate executables. The extractor is a one-shot batch process; the server is long-running. Keeping them separate means you can run extraction without starting the MCP server.

If you know Java: the concepts in this lesson translated

Path vs PathBuf — like &str vs String, applied to file paths

Java's java.nio.file.Path is a single type that handles both cases. Rust splits it into two: Path is a borrowed reference (you're reading someone else's path); PathBuf is an owned value (you hold the path data).

Java (java.nio.file)Rust (std::path)
Path.of("../kuka-docs")Path::new("../kuka-docs")
path.resolve("file.pdf")path.join("file.pdf") → returns a new PathBuf
path.getFileName().toString()path.file_name().unwrap().to_string_lossy()
path.toString()path.display() — formats the path for printing

The extra step to_string_lossy() has no Java equivalent because Java's String is always valid Unicode. Rust filenames can theoretically contain non-Unicode bytes on Linux, so the OS returns an OsStr which you convert — to_string_lossy() replaces any invalid bytes with ? rather than panicking. For KUKA's ASCII filenames, this is always a no-op. [Path — std docs]

The ? operator — checked exception propagation as a return value

In Java you propagate errors with throws: 

// Java
String text = Files.readString(path);  // throws IOException
// caller must declare throws IOException or wrap in try-catch

In Rust, errors are Result values (not exceptions). The ? operator is shorthand for: "if this returned Err, return that error from the current function immediately; otherwise unwrap the Ok value." 

// Rust
let text = pdf_extract::extract_text(pdf_path)?;
// If extraction fails, the function returns Err(...) here and stops.
// If it succeeds, `text` holds the String.

This is why main returns Result<()> — it needs somewhere to propagate errors to, just like Java's public static void main(String[] args) throws Exception.

std::fs::write() — Files.write(), Rust edition

Java (java.nio.file)Rust (std::fs)
Files.writeString(path, content); std::fs::write(&path, &content)?;

Both overwrite the file if it exists and create it if it doesn't. There is no explicit close() — Rust automatically closes file handles when the value goes out of scope (the same idea as Java's try-with-resources, but always on, requiring no syntax).

New concept: Path in Rust

Rust has two path types in std::path:

  • Path — a borrowed reference to a path string, like &str
  • PathBuf — an owned, growable path, like String

Path::new("../kuka-docs") creates a path reference. path.join("file.pdf") appends a component and returns a new PathBuf. path.file_stem() returns the filename without its extension. Because filenames can contain non-UTF-8 bytes on some systems, these methods return OsStr, which you convert to a normal string with to_string_lossy(). [Path — std docs]

Step 1 — Add the dependency

Inside your dev container terminal:

cargo add pdf-extract

Step 2 — Declare both binaries in Cargo.toml

Open mcp-server/Cargo.toml and add these two sections after the [package] block: 

[[bin]]
name = "mcp-server"
path = "src/main.rs"

[[bin]]
name = "extract"
path = "src/extract.rs"

Once you add any explicit [[bin]] entry, Cargo stops auto-detecting src/main.rs, so you must declare both.

Step 3 — Write src/extract.rs

Create mcp-server/src/extract.rs with this content: 

use anyhow::Result;
use std::path::Path;

fn main() -> Result<()> {
    let pdf_path = Path::new("../kuka-docs/KUKA Technical Note -Reflector Deployment- V1.0.1.pdf");
    let knowledge_dir = Path::new("../knowledge");

    println!("Extracting: {}", pdf_path.display());
    let text = pdf_extract::extract_text(pdf_path)?;

    // Build the output filename: lowercase, spaces become dashes
    let stem = pdf_path.file_stem().unwrap().to_string_lossy();
    let filename = pdf_path.file_name().unwrap().to_string_lossy();
    let slug = stem.to_lowercase()
        .replace(' ', "-")
        .replace(['.', '(', ')'], "");

    // Write the OKF file
    let okf = format!(
"---
type: technical-note
title: {stem}
description: Extracted from KUKA documentation.
resource: kuka-docs/{filename}
tags: [extracted, technical-note]
timestamp: 2026-06-26T00:00:00Z
---

{body}",
        stem = stem,
        filename = filename,
        body = text.trim(),
    );

    let output = knowledge_dir.join(format!("{slug}.md"));
    std::fs::write(&output, &okf)?;
    println!("Wrote: {}", output.display());

    Ok(())
}

Walking through the code

pdf_extract::extract_text(pdf_path)? reads the entire PDF and returns the text content as a String. The ? propagates any error (file not found, corrupt PDF) up to main which returns it as an anyhow::Result.

to_string_lossy() converts the OS filename to a Rust String, replacing any non-UTF-8 bytes with ?. KUKA filenames are all ASCII so this is safe — but it's the correct Rust idiom regardless.

std::fs::write(&output, &okf)? creates or overwrites the file in one call. No file handle, no explicit close — Rust drops the file when the function returns.

Step 4 — Build and run

Run from inside the mcp-server/ directory so the relative paths (../kuka-docs/, ../knowledge/) resolve correctly: 

cargo run --bin extract

You should see: 

Extracting: ../kuka-docs/KUKA Technical Note -Reflector Deployment- V1.0.1.pdf
Wrote: ../knowledge/kuka-technical-note--reflector-deployment--v101.md

Step 5 — Inspect the output

cat ../knowledge/kuka-technical-note--reflector-deployment--v101.md | head -20

You should see the OKF frontmatter followed by extracted text. The text will be raw — no headings, no formatting, just the words pulled from the PDF. That is expected at this stage. Lesson 9 adds structure to the output.

If the text looks garbled

Some PDFs store text as glyph indices rather than Unicode characters. pdf-extract handles most cases but may struggle with embedded fonts. If you see garbage characters, paste the first few lines into chat — we will decide whether to switch to a more capable crate or pre-process that specific file.

What you just proved

A Rust binary read a KUKA PDF, extracted its text, wrapped it in OKF frontmatter, and wrote it to the knowledge bundle. The search_docs tool now has real content to search — the wiring comes in Lesson 9.
