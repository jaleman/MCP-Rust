Refactor Appendix 18

Soft-AND Ranking — Recall Without Noise

Refactor step 11 KUKA MCP Server in Rust

Why strict AND was brittle

The old search behavior treated every query term as mandatory. That sounds precise, but it made real agent searches fragile: one guessed word such as meaning or tower could erase an otherwise useful query. If three-color indicator light matched the right page but meaning appeared nowhere, the whole result became empty.

Step 12 bounded broad output. Step 11 improves recall: documents can now surface when they match some, but not all, query terms. The important guardrail is ranking by coverage first, so a document that matches more distinct terms beats a document that repeats only one term many times.

The old early return

The strict behavior came from two places: returning empty when any term matched nowhere, then intersecting all per-term document maps.

Rust before:

for term in terms {
    let mut docs_for_term: HashMap<u32, (u32, Vec<u32>)> = HashMap::new();
    // fill docs_for_term...
    if docs_for_term.is_empty() {
        return Vec::new();
    }
    per_term.push(docs_for_term);
}

let mut candidates: Vec<u32> = per_term[0]
    .keys()
    .copied()
    .filter(|id| per_term.iter().all(|m| m.contains_key(id)))
    .collect();

Java equivalent:

for (String term : terms) {
    Map<Integer, PostingSummary> docsForTerm = findDocs(term);
    if (docsForTerm.isEmpty()) {
        return List.of();
    }
    perTerm.add(docsForTerm);
}

Set<Integer> candidates = new HashSet<>(perTerm.get(0).keySet());
for (Map<Integer, PostingSummary> map : perTerm) {
    candidates.retainAll(map.keySet());
}

Union candidates, rank by coverage

The new version keeps each term's matches, even if that map is empty. Candidates are the union of every document that matched at least one term. Then each candidate receives a coverage count: how many distinct query terms matched that document.

Rust after:

for term in terms {
    let mut docs_for_term: HashMap<u32, (u32, Vec<u32>)> = HashMap::new();
    // fill docs_for_term...
    per_term.push(docs_for_term);
}

let mut candidates: Vec<u32> = per_term
    .iter()
    .flat_map(|m| m.keys().copied())
    .collect();
candidates.sort_unstable();
candidates.dedup();

if candidates.is_empty() {
    return Vec::new();
}

Java equivalent:

for (String term : terms) {
    perTerm.add(findDocs(term));
}

Set<Integer> candidates = new HashSet<>();
for (Map<Integer, PostingSummary> map : perTerm) {
    candidates.addAll(map.keySet());
}
if (candidates.isEmpty()) {
    return List.of();
}

A private ranking key

The public SearchHit type did not need a new field. Coverage is an internal ranking key carried beside the hit and discarded before returning: 

let coverage = per_term.iter().filter(|m| m.contains_key(&doc_id)).count();
let freq_total: u32 = per_term
    .iter()
    .filter_map(|m| m.get(&doc_id))
    .map(|(freq, _)| freq)
    .sum();

ranked.push((coverage, SearchHit { /* existing fields */ }));
ranked.sort_by(|a, b| b.0.cmp(&a.0).then_with(|| b.1.score.cmp(&a.1.score)));
ranked.into_iter().map(|(_, hit)| hit).collect()

In Java, this would often be a private DTO or record: 

record RankedHit(int coverage, SearchHit hit) {}

ranked.sort(
    Comparator.comparingInt(RankedHit::coverage).reversed()
        .thenComparing(r -> r.hit().score(), Comparator.reverseOrder())
);
return ranked.stream().map(RankedHit::hit).toList();

The middle of the search function — proximity scoring and excerpt reading — stayed the same. It already works from the positions that actually exist, so it naturally handles a candidate matching fewer terms than the original query contained.

What the tests prove

One test searches the existing reflector fixture for reflector hydraulic. The old strict-AND behavior returned empty because hydraulic was absent. The new behavior returns the Reflector Guide because reflector is useful evidence.

Another test creates two documents: one contains both indicator and light; the other repeats indicator many times and has a higher frequency score. The full-coverage document must rank first anyway. That test protects the core rule: coverage before score.
