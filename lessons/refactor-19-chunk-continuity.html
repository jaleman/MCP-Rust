Refactor Appendix 19

Chunk Continuity — Helping Agents Keep Reading

Refactor step 13 KUKA MCP Server in Rust

Why continuity belongs in the tool output

Page chunking made large manuals safe for agents to read: every kuka://docs/... resource is small. The tradeoff is that a table or section can cross a chunk boundary. In the real trace that drove this step, the indicator-light table started on one chunk and continued on the next, but the search hit did not say where to go.

The extractor already wrote the needed metadata: parent names the original source document and pages records the chunk's page range. Step 13 stops discarding those fields, computes the next chunk once during indexing, and surfaces that fact in both places an agent sees content: search results and resource reads.

Carrying metadata forward

The bundle loader now keeps parent and pages on each document. The index copies them into DocMeta and adds a derived next_stem field.

Rust:

pub struct Document {

Computing the next chunk once

Instead of guessing from filenames at presentation time, the index groups chunks by parent, sorts each group by the starting page parsed from pages, and writes each chunk's successor. Search can then copy this cheap string into SearchHit.

Rust:

fn populate_next_stems(docs: &mut [DocMeta]) {
    let mut by_parent: HashMap<String, Vec<(usize, u32)>> = HashMap::new();
    for (index, doc) in docs.iter().enumerate() {
        if let (Some(parent), Some(start_page)) = (&doc.parent, doc.start_page()) {
            by_parent.entry(parent.clone()).or_default().push((index, start_page));
        }
    }

    for chunks in by_parent.values_mut() {
        chunks.sort_by_key(|(_, start_page)| *start_page);
        for pair in chunks.windows(2) {
            let current = pair[0].0;
            let next = pair[1].0;
            docs[current].next_stem = Some(docs[next].stem.clone());
        }
    }
}

Java equivalent:

Map<String, List<ChunkRef>> byParent = new HashMap<>();
for (int i = 0; i < docs.size(); i++) {
    DocMeta doc = docs.get(i);
    doc.startPage().ifPresent(start ->
        doc.parent.ifPresent(parent ->
            byParent.computeIfAbsent(parent, p -> new ArrayList<>())
                    .add(new ChunkRef(i, start))));
}

for (List<ChunkRef> chunks : byParent.values()) {
    chunks.sort(Comparator.comparingInt(ChunkRef::startPage));
    for (int i = 0; i < chunks.size() - 1; i++) {
        docs.get(chunks.get(i).index()).nextStem =
            Optional.of(docs.get(chunks.get(i + 1).index()).stem);
    }
}

Surfacing the hint where agents look

Search hits now show a Continues: line only when one exists. Reading the full resource also appends a short trailer. That matters because an agent may skip search once it has opened a resource. 

if let Some(next) = &hit.continues {
    out.push_str(&format!("\n  Continues: kuka://docs/{next}"));
}

if let Some(next) = continues {
    content.push_str(&format!(
        "\n\n---\n[This section continues in kuka://docs/{next}]"
    ));
}

What the tests prove

The index tests prove the derived chain: a two-chunk fixture sets the first chunk's next_stem to the second chunk and leaves the final chunk empty. The search test proves SearchHit carries that value. The binary tests prove the MCP presentation actually says Continues: kuka://docs/... and appends the resource trailer.
