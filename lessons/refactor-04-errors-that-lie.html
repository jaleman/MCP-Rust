Refactor Appendix 4

Errors That Lie — Result, Context, and the Traversal Guard

Refactor step 2 — commit d0b9f07 KUKA MCP Server in Rust

The bug: a missing directory that looked like an empty one

Before this step, every consumer wrapped the directory walk in if let Ok(entries) = std::fs::read_dir(dir). That pattern reads as "handle the error", but it actually discards it — if the directory doesn't exist, the loop body simply never runs, and the tool responds "No documents found in the knowledge bundle."

Why does this matter so much here? Because MCP clients launch this server from an arbitrary working directory, and our default path is the relative "knowledge". The single most likely failure mode of the entire server is "directory not found" — and it presented as a calm, plausible, wrong answer. An error that lies is worse than a crash: it sends whoever is debugging (human or AI agent) straight past the actual cause.

The Java equivalent of the old code is an empty catch block: 

// Java — the same sin, in its native syntax
List<Document> docs = new ArrayList<>();
try (var stream = Files.list(dir)) {
    stream.forEach(p -> docs.add(Document.load(p)));
} catch (IOException e) {
    // swallowed — caller sees an empty list and thinks the bundle is empty
}

The fix: errors travel in the return type

load_bundle now returns Result<Vec<Document>>, and the missing-directory case is a real error with the path and the remedy in its message: 

pub fn load_bundle(dir: &Path) -> Result<Vec<Document>> {
    let entries = std::fs::read_dir(dir).with_context(|| {
        format!(
            "knowledge directory not found: {} — set KUKA_KNOWLEDGE_DIR or run from the project root",
            dir.display()
        )
    })?;
    …
}

Two pieces of machinery here, both from earlier lessons but now doing real work:

? is a typed throws. Where Java propagates exceptions invisibly (any call might throw), Rust makes propagation a visible, one-character operator on functions whose signature says Result. read_dir(dir)? is the moral equivalent of letting an IOException bubble up a throws IOException chain.

.with_context(…) wraps like exception chaining. anyhow's context is Java's throw new RuntimeException("knowledge directory not found: " + dir, cause) — the original OS error stays attached underneath. Printing with {e:#} shows the whole chain, which is how the live server now reports: 

Cannot read knowledge bundle: knowledge directory not found: knowledge —
set KUKA_KNOWLEDGE_DIR or run from the project root: No such file or directory (os error 2)

Choosing an error policy per failure, not per function

Not every failure deserves the same treatment. The refactor made three deliberate, different choices inside load_bundle and its callers:

FailurePolicyWhy
bundle directory missing hard error — MCP isError: true nothing can work; the message names the fix
one file unreadable tracing::warn! and skip it one bad file must not take down the other 400
bundle genuinely empty normal success message a valid state, not a fault

The tool layer converts the hard error into an MCP result the client can see — note this is CallToolResult::error(…), not success, so agents know the call failed rather than "found nothing": 

let docs = match load_bundle(dir) {
    Ok(docs) => docs,
    Err(e) => {
        return Ok(CallToolResult::error(vec![Content::text(format!(
            "Cannot read knowledge bundle: {e:#}"
        ))]));
    }
};

The traversal guard: never trust a URI

The same commit closed a real security hole. read_resource takes a URI like kuka://docs/reflector-guide, strips the prefix, and joins the rest onto the knowledge directory. Nothing stopped a client from sending kuka://docs/../../etc/passwd — and join would cheerfully escape the bundle and read any .md-suffixed path on the machine. (Java's Path.resolve() has exactly the same behavior — path traversal is the classic servlet vulnerability, not a Rust-specific one.) 

/// Only plain file stems are valid — no separators, no parent references.
pub fn resource_stem_is_safe(stem: &str) -> bool {
    !stem.is_empty() && !stem.contains(['/', '\\']) && !stem.contains("..")
}

The guard is a plain, pure function in bundle.rs rather than inline in the handler — deliberately. The async handler with its RequestContext is awkward to unit-test; a pure &str -> bool function is trivial to test exhaustively (and is: valid stems, ../, backslashes, empty). Verified against the live server: 

→ {"method":"resources/read","params":{"uri":"kuka://docs/../../etc/passwd"}}
← {"error":{"code":-32002,"message":"Invalid resource URI: kuka://docs/../../etc/passwd"}}

Side-by-side summary

JavaRust
throws IOException (invisible at call sites) -> Result<T> + visible ? at each call
new RuntimeException(msg, cause) .with_context(|| msg) (anyhow)
e.printStackTrace() / logger chain format!("{e:#}") prints the cause chain
empty catch {} block if let Ok(x) = … with no else — same smell!
servlet path-traversal check resource_stem_is_safe before Path::join

The habit to take away: in Rust, error-swallowing is syntactically visible — any if let Ok(…) without an else branch deserves the same suspicion an empty catch block gets in Java code review.
