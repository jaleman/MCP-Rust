Lesson 12

Stop Word Filtering

~15 minutes Hands-on — edit, build, test in your dev container KUKA MCP Server in Rust

What you're fixing

After Lesson 11, every word in the query must match somewhere in the document — including words like "what", "are", "the", and "for". These words are called stop words: they carry no meaning on their own and appear in almost every document. Requiring them to match adds noise and occasionally causes a query to fail.

More importantly, if a stop word is 3 characters or fewer — "are", "the", "for", "of" — the fuzzy helper skips it (too short to fuzz) and requires an exact substring match. "The" and "are" will appear in any document, so this rarely breaks things, but it wastes time checking words that contribute nothing to the result.

The fix: strip stop words from the query before searching, so only meaningful terms reach fuzzy_word_match.

Query: "what are the safe minimum distances for AMR robots"

Before: terms = ["what", "are", "the", "safe", "minimum", "distances", "for", "amr", "robots"]
           all 9 words must match in the document

After:  terms = ["safe", "minimum", "distances", "amr", "robots"]
           only 5 meaningful words must match

New concepts — translated from Java

Concept 1: HashSet — a set with O(1) lookup

A HashSet stores unique values and answers "is this value in the set?" in constant time — it does not scan through the list, it hashes the value and jumps directly to the answer. In Java you already know this as HashSet<String>. Rust's version works identically.

JavaRust
Set<String> stops = Set.of("a", "the", "for"); let stops: HashSet<&str> = HashSet::from(["a", "the", "for"]);
stops.contains("the") stops.contains("the")

To use it you need a use declaration — HashSet lives in Rust's standard library under std::collections, the same module that holds HashMap, BTreeMap, and other collection types: 

use std::collections::HashSet;

HashSet::from([...]) creates a set from an array literal. The square brackets are a fixed-size array (not a Vec), but from() converts it for you. This pattern is the closest Rust equivalent to Java's Set.of(...).

Why not a Vec?

You could store stop words in a Vec<&str> and call vec.contains(&word) — it would work. But Vec::contains() scans every element from the start, so it is O(n). With 30 stop words that is 30 comparisons in the worst case. HashSet::contains() is O(1) — one hash lookup regardless of how many words are in the set. For a fixed list of known words, HashSet is the right tool.

Concept 2: .filter() — keep only elements that pass a test

.filter() is a method on any iterator that passes each element through a predicate (a closure returning bool) and keeps only the elements where the predicate returns true. In Java you know this as Stream.filter(predicate).

JavaRust
Arrays.stream(words)
  .filter(w -> !stops.contains(w))
  .collect(toList())		 words.split_whitespace()
  .filter(|w| !stops.contains(w))
  .collect()

Like .all() and .any(), .filter() is lazy — it does not run until something consumes the iterator. The .collect() at the end is what triggers the work and builds the Vec.

Step 1 — Add the use declaration

Open mcp-server/src/main.rs. Add this line at the top with the other use declarations: 

use std::collections::HashSet;

Step 2 — Replace the terms line in search_docs

Find this block near the top of search_docs: 

let query = input.query.to_lowercase();
let terms: Vec<&str> = query.split_whitespace().collect();

Replace it with:

let query = input.query.to_lowercase();
let stop_words: HashSet<&str> = HashSet::from([
    "a", "an", "the", "is", "are", "was", "were", "be", "been",
    "have", "has", "had", "do", "does", "did", "will", "would",
    "can", "could", "should", "may", "might", "shall",
    "i", "you", "he", "she", "it", "we", "they",
    "what", "which", "who", "when", "where", "why", "how",
    "in", "on", "at", "to", "for", "of", "with", "by", "from",
    "and", "or", "but", "not", "no", "nor",
]);
let terms: Vec<&str> = query
    .split_whitespace()
    .filter(|word| !stop_words.contains(word))
    .collect();
if terms.is_empty() {
    return Ok(CallToolResult::success(vec![Content::text(
        "Query contains only common words. Please add specific search terms.".to_string(),
    )]));
}

What each part does

  • HashSet::from([...]) — builds the stop word set from an array literal. The type annotation HashSet<&str> tells Rust what type of values to store. Without it, Rust could not infer the type from the array alone.
  • .filter(|word| !stop_words.contains(word)) — keeps only words that are not in the stop word set. The ! is logical negation — same as Java.
  • if terms.is_empty() — guards against queries like "what is the" where every word is a stop word. Without this guard, terms.iter().all(...) would return true for every document (vacuous truth: all zero terms match), and the server would return every document as a result.
  • return Ok(...) — an early return from the function. Note the explicit return keyword here; without it, this would need to be the last expression in the function to be treated as the return value.
Vacuous truth — why the empty guard matters

terms.iter().all(|term| ...) with zero terms returns true automatically. Logically: "all zero terms match" is vacuously true, the same way "all students in an empty classroom passed the exam" is technically true. Without the guard, an all-stop-word query would match every document — the opposite of useful.

Step 3 — Build and test

cargo build

Then try a query heavy with stop words: 

What are the safe distances for AMR robots?

Before this lesson, 9 words were sent to the matcher. After, only "safe", "distances", and "amr" (or "robots") reach fuzzy_word_match — all of which appear in the safety document.

Also verify the empty-query guard works. Ask your AI client something like: 

What is it?

You should get the "Query contains only common words" response instead of every document being returned.
