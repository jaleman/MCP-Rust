Lesson 3

Cargo Fundamentals

~15 minutes Hands-on — run these commands inside your dev container KUKA MCP Server in Rust

What you're doing and why

Your dev container is running. Now you need a Rust project inside it — a directory structure with source files and a manifest that tells Rust how to build them. In Rust, the tool that creates, builds, and manages dependencies for a project is called Cargo. By the end of this lesson you will have:

  • A Cargo project called mcp-server
  • rmcp — the official Rust MCP SDK — added as a dependency
  • A successful cargo build that downloads and compiles everything

This is the foundation everything else is built on. Once this compiles cleanly, you're ready to start writing the actual server code.

Before you start
Open your dev container in VS Code (Command Palette → Dev Containers: Reopen in Container), then open a terminal inside it (Ctrl+`). All commands below run inside the container, not on Windows. Check you're in the right place with pwd — you should see /workspaces/MCP-Rust or similar.

What Cargo is

Cargo is Rust's build system and package manager — it does the job that npm, pip, or Maven do in other ecosystems. Every Rust project has a file called Cargo.toml at its root. That file is the single source of truth for:

  • Your project's name, version, and Rust edition
  • Which external libraries (called crates) your code depends on
  • Optional build configuration

When you run cargo build, Cargo reads Cargo.toml, downloads any missing crates from crates.io (Rust's package registry), and compiles everything. [Why Cargo exists — Cargo Book]

If you know Java: Cargo = Maven + Gradle in one tool

You already know the problem Cargo solves — managing dependencies and building code. In Java you likely use Maven or Gradle. Cargo is Rust's equivalent, but it is the only tool: there is no ecosystem split, no plugins to choose between. Every Rust project uses Cargo.

Java / MavenRust / Cargo
pom.xmlCargo.toml — project metadata + dependencies in one file
Maven Centralcrates.io — the single public package registry
JAR artifact / librarycrate — Rust's name for a compiled package

Maven optional dependencies / profilesfeatures — named optional chunks of a crate you opt into with --features

The one concept with no clean Java analogy is features. In Maven, you either depend on a JAR or you don't. In Rust, a crate can be split into optional pieces — you only compile (and pay the build-time cost for) what you actually use. rmcp's procedural macros are an optional feature; you activate them with --features macros so projects that don't need them don't pay for them.

Step 1: Create the project

Run this inside your dev container terminal:

cargo new mcp-server --bin

The --bin flag means "I want an executable program" (as opposed to a library). Cargo creates a directory called mcp-server/ with this layout:

mcp-server/
├── Cargo.toml ← project manifest
└── src/
    └── main.rs ← your program's entry point

Move into that directory: 

cd mcp-server

Step 2: Read Cargo.toml

Open Cargo.toml. It will look like this:

[package]
name = "mcp-server"
version = "0.1.0"
edition = "2024"

[dependencies]

There are exactly two sections right now:

  • [package] — project metadata. The name becomes the name of your compiled binary. edition = "2024" means you're using the 2024 version of the Rust language (the current standard, stable since Rust 1.85).
  • [dependencies] — empty for now. Every external crate you add will appear here as a name/version pair.

And src/main.rs contains the canonical "Hello, world!" to prove it compiles: 

fn main() {
    println!("Hello, world!");
}

Verify it works before touching anything:

cargo run

You should see Hello, world! printed. Cargo compiled the binary and ran it. First run takes a moment because Cargo builds the standard library toolchain — subsequent builds are faster.

Step 3: Add rmcp

rmcp is the official Rust SDK for the Model Context Protocol — the library that handles all the JSON-RPC plumbing of MCP so you don't have to. [rmcp on docs.rs] [rust-sdk on GitHub]

Add it with one command:

cargo add rmcp --features macros

cargo add looks up the latest version of rmcp on crates.io and writes it into your Cargo.toml automatically — you never edit the version number by hand. The --features macros part enables a set of optional helpers inside rmcp that make defining tools much less verbose. You'll see them in action in the next lesson.

Open Cargo.toml again — it now looks like:

[package]
name = "mcp-server"
version = "0.1.0"
edition = "2024"

[dependencies]
rmcp = { version = "x.y.z", features = ["macros"] }

The x.y.z will be the current version Cargo resolved. The features key tells Cargo which optional parts of rmcp to compile in.

What are features?
Many Rust crates are split into optional pieces called features. This keeps compile times short — you only pay for what you use. rmcp ships its procedural macros as an optional feature rather than always compiling them in. [Cargo features — Cargo Book]

Step 4: Build to verify

Run a full build. This will download rmcp and all of its dependencies (including Tokio, the async runtime) and compile everything: 

cargo build

The first run after adding a new dependency always takes longer — Cargo is compiling the dependency tree from scratch. You'll see a stream of Compiling ... lines. When it ends with Finished dev [unoptimized + debuginfo], you're done. No errors means your project correctly references rmcp and everything compiles.

cargo build vs cargo run
cargo build compiles but does not execute the binary. Use it to check that your code is correct without running anything. cargo run compiles and then runs the resulting binary. When writing a server you'll usually use cargo build to catch errors early, then cargo run to start the server.

One more file appeared: Cargo.lock. This is Cargo's record of the exact version of every dependency (and every dependency of those dependencies) that was resolved. You never edit it — Cargo manages it. Commit it to version control so that everyone (including future-you) builds against identical versions. [Cargo.toml vs Cargo.lock — Cargo Book]
